## はじめに

AIコーディングエージェントを使うと、実装の初速はかなり上がります。

ただ、チーム開発で使っていると、単に「速く書ける」だけでは足りないと感じます。

むしろ困るのは、速く書けるからこそ、次の問題が一気に増えることです。

- 何を実装するかが曖昧なまま進む
- AIが親切にスコープを広げる
- レビュー時に「今回の正解は何か」を確認し直す
- テスト観点が実装後に後付けになる
- PRの差分が大きくなって人間が読みづらい

そこで、自分は PlanGate というワークフローを作り、AIに実装させる前に計画を作らせ、人間が承認してから実装に進める形にしています。

考え方はシンプルです。

> 承認なし、コードなし。

この記事では、PlanGateを作った背景と、チーム開発でどのように効くのかを体験寄りにまとめます。

## 先に結論

PlanGateで一番効いたのは、AIの実装力を上げることではありません。

**AIがコードを書く前に、人間が判断する場所を作れたこと**です。

実装後に大きな差分を見ながら「これは今回必要だったのか」と確認するより、実装前に `plan.md` を見て「この範囲で進めてよい」と判断するほうが、チーム開発では扱いやすくなります。

PlanGateは、AIを止めるための仕組みではありません。
AIをチームで使える形にするための仕組みです。

この記事は、次のような人向けです。

- AIが作るPRの差分が大きくなり、レビューが重くなっている人
- AIに任せる範囲と、人間が判断する範囲を分けたい人
- AI駆動開発をチームで回すための運用ルールを探している人
- Claude Code や Codex CLI を使い始めて、次の段階としてガードレールを考えたい人

## 導入前後で変わったこと

PlanGateを入れて変わったのは、AIの出力そのものよりも、人間側の確認タイミングでした。

| 観点 | 以前 | PlanGate後 |
| --- | --- | --- |
| 実装開始 | Issueを読ませてすぐ実装 | まずplanを作り、人間が承認してから実装 |
| スコープ | AIの判断で広がりやすい | Non-goalsで「やらないこと」を先に固定 |
| テスト観点 | 実装後に確認しがち | `test-cases.md` を先に作る |
| レビュー | 差分を見ながら目的を確認 | plan / test-cases / 差分の対応を見る |
| 失敗時 | 差分を戻す・説明を聞く | 計画段階で修正できる |

この変化によって、AIの作業を細かく管理するというより、**チームが判断しやすい状態を先に作る**感覚に近くなりました。

## 以前の使い方で困ったこと

最初は、AIにかなり直接的に頼んでいました。

```markdown
このIssueを読んで実装してください。
関連ファイルを確認し、必要な変更を入れてください。
```

この依頼でも、かなりの確率で動くコードは出ます。

でも、チームでレビューする段階になると、困ることが増えました。

たとえば、小さなUI修正のつもりだったのに、AIが周辺コンポーネントの整理までしてくれることがあります。

単体で見ると、悪い変更ではありません。

しかし、レビューする側から見るとこうなります。

- 今回の目的に必要な変更なのか分からない
- ついでの改善を入れるかどうかをレビュー中に判断する必要がある
- 確認すべきテスト範囲が増える
- PRが大きくなり、レビューの焦点がぼやける

AIが悪いというより、実装前の境界が曖昧だったのだと思います。

人間同士でも、スコープが曖昧なタスクは膨らみます。AIの場合は、それが速く、大量に起きます。

## 先に計画を作らせる

そこで、AIにいきなりコードを書かせるのをやめました。

まず、次の成果物を作らせます。

```text
plan.md
todo.md
test-cases.md
review-self.md
```

そして、人間が `plan.md` を読んでから判断します。

- APPROVE: 進めてよい
- CONDITIONAL: 条件付きで進めてよい
- REJECT: 計画を直す

この承認を PlanGate では C-3 gate と呼んでいます。

ポイントは、AIを止めたいわけではないことです。

AIに自由に実装させる前に、人間が見るべきものを前倒ししたいだけです。

## planで見るもの

自分がplanで特に見ているのは、次の5つです。

```markdown
## Goal
何を達成するか

## Non-goals
今回はやらないこと

## Scope
どの範囲を変更するか

## Test plan
どう確認するか

## Risks
壊れやすいところ、前提が怪しいところ
```

この中で一番効いたのは `Non-goals` です。

AIは、関連しそうな改善をよく提案します。

たとえば、次のようなものです。

- 命名を整える
- 周辺コンポーネントも整理する
- 似た画面にも同じ変更を広げる
- APIレスポンスの形を少し直す
- テスト基盤もついでに更新する

どれも単体では良さそうに見えます。

でも、今回の目的と関係が薄いなら、レビュー可能性を下げます。

`Non-goals` に「今回はやらないこと」を書いておくと、レビュー時にこう言えます。

```markdown
この変更自体は良さそうですが、今回のNon-goalsに入っているので別PRにしたいです。
```

これはとても大事です。

レビューで揉めるのは、変更の良し悪しだけではありません。
「今回入れるべきか」が曖昧なときにも時間を使います。

## テスト観点を先に出す

もう一つ効いたのは、実装前に `test-cases.md` を作ることです。

これは、最初から完全なテストコードを書かせるという意味ではありません。

まずは、何を満たせばOKかを言語化します。

```markdown
# test-cases.md

## 正常系
- 必須項目を入力すると保存できる

## 異常系
- APIが500を返したら保存済み表示にしない

## 境界値
- 空白だけの入力は未入力として扱う

## 回帰確認
- 既存の成功時メッセージは変えない
```

これがあると、実装後のレビューがかなり楽になります。

レビューの入口が「コード全体を読む」ではなく、「事前に決めた観点と差分が対応しているか」になるからです。

AIにも、実装後にこう聞けます。

```markdown
test-cases.md の各項目について、どの変更で満たしたか説明してください。
満たせていない項目、未確認の項目があれば明記してください。
```

このとき、未確認が出るのは悪いことではありません。

人間がレビューで見るべき場所が分かるからです。

## Hookで守る

最初は、プロンプトで「承認前に実装しないで」と指示していました。

ただ、運用していると、プロンプトだけでは弱いと感じます。

そこで、PlanGate v8.5.0 では Hook enforcement を整えました。

たとえば、次のような条件を hook や CLI で検査できます。

- `plan.md` がない状態で production code を編集していないか
- C-3承認なしで実装に進んでいないか
- 承認後に `plan.md` が改変されていないか
- `test-cases.md` なしで検証に進んでいないか
- 検証 evidence なしでPRを作ろうとしていないか
- `forbidden_files` に触っていないか
- C-3 / C-4 承認なしでマージしようとしていないか

これをすべて最初からstrictにする必要はありません。

導入初期はwarningで観測し、運用が固まったものだけブロックに上げるのがよいと思います。

## 実際に変わったこと

PlanGateのような流れにして、一番変わったのは「実装の前に会話できる」ことでした。

以前は、AIが作った差分を見てから、こういう会話をしていました。

```text
この変更、今回の目的に必要？
このテスト観点は考慮していた？
このファイルまで触る必要あった？
```

今は、実装前に同じ会話ができます。

```text
この計画で進めてよい？
このNon-goalsで足りる？
このTest planで確認できる？
このRiskを見落としていない？
```

実装後の差分を戻すより、実装前に計画を直すほうが圧倒的に楽です。

AIを速く走らせるほど、止める場所の価値が上がります。

## 導入するときの注意点

PlanGateのような仕組みは、重くしようと思えばいくらでも重くできます。

ただ、最初から全部やる必要はありません。

最初は、1つのIssueだけで十分です。

1. AIにplanを作らせる
2. 人間がplanだけレビューする
3. 承認してから実装させる
4. test-casesと差分の対応を見る
5. PRで残リスクを書く

このくらいで、かなり効果が分かります。

逆に、最初から全Hook、全Schema、全Evalを入れると、仕組みの説明コストが勝ってしまうかもしれません。

小さく始めて、効いたところだけ強くするのがよいです。

## まとめ

AIコーディングエージェントを使っていて思うのは、これから重要になるのは「どのAIが一番賢いか」だけではないということです。

同じくらい重要なのは、**AIをどこで止めるか**です。

PlanGateは、AIに実装させる前に計画を作らせ、人間が承認してから進めるためのワークフローです。

実装を遅くするための仕組みではありません。

むしろ、後戻りを減らして、レビューできる形を保つための仕組みです。

AIが速くなるほど、チーム側には「承認」「スコープ」「テスト観点」「検証証拠」を残す力が必要になります。

PlanGateは、そのための小さなガードレールとして作っています。
