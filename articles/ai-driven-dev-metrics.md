:::message
この記事では、AI導入の成果を **速度ではなく手戻りの減少** で見る考え方を整理します。

- AI導入の効果を、実装速度だけで評価しない理由
- 最初に追うべき `手戻り率 / 差し戻し率 / 再オープン率` の考え方
- 監視ではなく改善につながるメトリクス設計の始め方
:::

## はじめに

AI駆動開発の効果は、「速くなった気がする」で語られがちです。
ただし、初速が上がっても、後工程で差し戻しや再作業が増えるなら、全体では改善していない可能性があります。

まずは、後工程で戻ってきた作業を小さく数えるところから始めます。

## 1. なぜ速度だけでは判断を誤るのか

### 初速は上がりやすい

AIは、ボイラープレート生成、テスト下書き、リファクタのたたき台作成で明確に効きます。
そのため、最初に観測しやすいのは「書き始めの速さ」です。

### 後工程のコストは見えにくい

一方で、次のようなコストは見えにくいまま増えます。

- 仕様差し戻し
- レビューでの再修正
- テストの組み直し
- いったん閉じた作業の再オープン

ここを見ないまま「速くなった」と判断すると、局所最適に陥ります。

## 2. 最初に見るべき指標

まず追うべき指標は3つです。
`手戻り率 / レビュー差し戻し率 / 再オープン率` を並べると、AIで速く閉じた作業が後工程で戻っていないかを見やすくなります。

### 手戻り率

一度「終わった」と見なした作業が、どれだけ再作業に戻ったかを見る指標です。
計測例: 再作業になったタスク数 ÷ 完了タスク総数

### レビュー差し戻し率

レビュー段階で、設計変更や要件の再確認が必要になった PR の割合です。
実装そのものより、前提の揃え方や設計の粗さが見えます。
計測例: 差し戻しが発生した PR 数 ÷ レビュー完了 PR 総数

### 再オープン率

close 済み Issue や完了扱いのタスクが、どれだけ再度開かれたかを見ます。
特に「AIが早く閉じたが、人間が後で戻した」ケースの検出に向いています。
計測例: 再オープンされた Issue 数 ÷ クローズ済み Issue 総数

## 3. どう解釈するか

### 速いが手戻りが多い状態

短期的な見た目は良いですが、実務では危険です。
前工程の確認不足やレビュー観点の不足が疑われます。

### 遅いが再現性が高い状態

導入初期には、こちらのほうが健全なことがあります。
計画、レビュー、検証が揃ってくると、ここから徐々に速度が上がります。

## 4. 小さく始める計測設計

### 最小セットで十分

最初から厳密な生産性ダッシュボードを作る必要はありません。
Issue、PR、レビュー履歴から拾える最小セットで足ります。

| 計測項目 | データソース | 対応指標 |
| -------- | ------------ | -------- |
| 差し戻しが発生した PR 数 | GitHub PR ラベル / レビューコメント | レビュー差し戻し率 |
| 再オープンした Issue 数 | GitHub Issue ログ | 再オープン率 |
| 完了後に追加修正が入った件数 | PR / Issue のクローズ後更新 | 手戻り率 |

### 粒度をそろえる

PBI、Issue、PR のどれを単位にするかを先に決めます。
単位が混ざると比較ができません。

## 5. 改善につなげる

### 前工程を見る

手戻り率が高いなら、Plan や要件確認の不足を疑います。

### 後工程を見る

差し戻し率が高いなら、レビュー観点や検証コマンドの不足を疑います。

### 人を評価しない

メトリクスを個人評価に使うと、正直な運用データが取れなくなります。
改善の入口として使うほうが、継続しやすいです。

## 6. やってはいけない測り方

- コミット数: 小さく刻んだだけでも増えるため、ユーザー価値や手戻り削減を示しません
- 生成コード行数: 行数が多いほど、レビューや削除のコストが増えることもあります
- 会話ターン数: 丁寧に確認した結果なのか、迷走した結果なのかを区別できません
- AIが提案した件数そのもの: 採用されない提案が増えても、成果にはつながりません

## まとめ

まずは直近1か月の PR を見て、差し戻しが発生した件数、クローズ後に追加修正が入った件数、再オープンされた Issue 数を数えます。
そのうえで、差し戻しが多い工程にだけ Plan、レビュー観点、検証コマンドの改善を当てると始めやすいです。
