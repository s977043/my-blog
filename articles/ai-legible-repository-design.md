Claude Code や Codex を使い始めると、最初にやりたくなるのは「AI向けの長い指示ファイル」を作ることです。

もちろん、入口ファイルは大事です。
ただ、そこに全部を詰め込むと、しばらくして次の問題が出ます。

- 重要なルールが長文の中に埋もれる
- 領域ごとの注意点が混ざる
- 守らせたい制約が自然言語のお願いで止まる
- どの情報が正本か分からず、AIが推測で進む

AIに読ませる文章を増やす前に、**リポジトリ側の置き場所を分ける**ほうが効くことがあります。

## TL;DR

- 入口ファイル: `AGENTS.md` や `CLAUDE.md` は短い地図にする
- 局所ルール: 危険領域ごとの注意点は近くに置く
- 強制する仕組み: 必ず守らせたい制約は hooks や CI に寄せる
- 正本と外部参照: 変わる情報は手書きのMarkdownに固定しない

## 1. 入口ファイル（AGENTS.md / CLAUDE.md）

`AGENTS.md` や `CLAUDE.md` に置きたいのは、全セッションで必要な情報です。

- このリポジトリは何をするものか
- 主要ディレクトリの責務は何か
- どのコマンドで検証するか
- どのファイルが正本か
- 触ってはいけない領域はどこか

逆に、たまにしか使わない手順や、特定領域だけの細かい注意点まで入れると、入口が重くなります。

入口ファイルは、百科事典ではなく地図です。

具体的には、入口ファイルはこのくらいの粒度で十分です。

```markdown
# AGENTS.md

## このリポジトリ

Zenn / Qiita の記事を管理するリポジトリ。

## 主な置き場所

- articles/: Zennの記事
- books/: Zennの本
- public/: Qiitaの記事

## 検証

- npm run check

## 作業ルール

- published: true の記事は、明示依頼がない限り大きく書き換えない
- 下書き記事は front matter と参考リンクを確認してから編集する
```

ここで大事なのは、AIが最初に読むべき「入口」だけを書くことです。
細かい文体ルール、公開フロー、記事ごとのレビュー観点まで全部入れると、入口ファイルが読みにくくなります。

長くなってきたら、次のように分割します。

| 情報 | 置き場所 |
| --- | --- |
| 全体の地図 | `AGENTS.md` |
| 記事の書き方 | `docs/editorial-guide.md` |
| 公開手順 | `docs/publish-runbook.md` |
| 危険な作業 | 対象ディレクトリの `AGENTS.md` |

AI向けのドキュメントは、長くするほど賢くなるわけではありません。
必要なときに必要なファイルへ誘導できるほうが、実務では強いです。

## 2. 局所ルール（危険領域ごとのAGENTS.md）

認証、課金、DBマイグレーション、公開フローのような領域は、全体ルールだけでは足りないことがあります。

たとえば、こう分けます。

```text
repo-root/
├── AGENTS.md
├── docs/
│   ├── architecture.md
│   └── runbook.md
├── src/
│   ├── auth/
│   │   └── AGENTS.md
│   └── billing/
│       └── AGENTS.md
└── spec/
    ├── requirements.md
    ├── design.md
    └── tasks.md
```

`src/auth/AGENTS.md` には、認証まわりの禁止事項や確認観点だけを書きます。
全体に効く話と、局所的に効く話を混ぜないのがポイントです。

局所ルールは、AIに「このディレクトリは空気が違う」と伝えるためのものです。
たとえば認証まわりなら、次のように書けます。

```markdown
# src/auth/AGENTS.md

## この領域で守ること

- セッション検証の挙動を変える場合は、必ず既存テストを確認する
- 認可チェックを UI 側だけに置かない
- エラーメッセージにユーザーの存在有無を出さない

## 変更時の確認

- npm run test:auth
- npm run lint
```

課金まわりなら、確認観点は変わります。

```markdown
# src/billing/AGENTS.md

## この領域で守ること

- 金額計算はサーバー側を正本にする
- webhook の冪等性を壊さない
- plan id と表示名を混同しない

## 変更時の確認

- npm run test:billing
- webhook の再送時に二重反映されないこと
```

このように、危険領域ごとに「何を怖がっているのか」を短く書くと、AIの判断が安定します。
全体の `AGENTS.md` に全部書くより、変更箇所の近くに置いたほうが読まれやすいです。

## 3. 強制する仕組み（hooks / CI / ブランチ保護）

自然言語のルールは、あくまで文脈です。
AIは読んで従おうとしますが、必ず守る保証にはなりません。

だから、必ず守らせたいことは hooks、CI、テスト、ブランチ保護などに寄せます。

- format を必ず走らせる
- 特定パス変更時にテストを走らせる
- secret を検出したら止める
- main への直接 push を防ぐ
- migration 変更時にレビューを必須にする

AIに「気をつけて」と書くより、通れないゲートを作るほうが運用として強いです。

ここは、AIコーディングで特に重要です。
AIは指示を読んでくれますが、指示を読んだことと、組織として保証したい品質が満たされることは別です。

たとえば、次のように責務を分けます。

| 守りたいこと | 自然言語で足りるか | 仕組みに寄せる例 |
| --- | --- | --- |
| 文体を揃える | ある程度足りる | editorial guide |
| formatを崩さない | 足りない | formatter / CI |
| secretを混ぜない | 足りない | secret scan |
| migrationを安全にする | 足りない | migration test / review rule |
| 公開記事を壊さない | 足りない | diff review / branch protection |

自然言語は「意図」を伝える場所です。
CIやhooksは「通してよいか」を判定する場所です。
この2つを混ぜると、AIへのお願いが増えるだけで、事故はあまり減りません。

最小なら、次の3つから始めるだけでも効果があります。

```text
1. npm run check をCIで必須にする
2. secret scan を入れる
3. 重要ディレクトリ変更時のCODEOWNERSを設定する
```

AIに任せる範囲が広がるほど、人間の注意力ではなく、通過条件で守るほうが安定します。

## 4. 正本と外部参照（変わる情報は固定しない）

PRの状態、Issueの担当者、外部サービスの現在値、APIのレスポンスなどは変わります。

こうした情報を手書きのMarkdownに固定すると、すぐ古くなります。

リポジトリに置くべきなのは、長く効く構造化された知識です。
変わる情報は、GitHub、Notion、MCP、APIなど、その時点の正本から取りに行くほうが安全です。

```text
長く効くもの: docs / spec / runbook / ADR
変わるもの: PR状態 / Issue状態 / 外部APIの現在値 / 本番データ
```

この境界があると、AIが古いメモを信じて進む事故を減らせます。

### 正本を明記する

AIが迷う原因の1つは、同じ情報が複数の場所にあることです。
たとえば、仕様がIssue、Notion、README、古い設計メモに散らばっていると、AIはどれを信じるべきか推測します。

入口ファイルやdocsには、正本を明記します。

```markdown
## Source of Truth

- 仕様: spec/requirements.md
- UI設計: docs/design.md
- API仕様: docs/api.md
- タスク状態: GitHub Issue
- 現在のPR状態: GitHubから取得する
```

ここで「現在のPR状態」のような変わる情報は、Markdownに書き写さないほうがいいです。
AIには「このファイルを信じろ」だけでなく、「これは都度取りに行け」も伝えます。

書き写す情報と取りに行く情報を分けると、古いドキュメントに引っ張られる事故が減ります。

## AIが読みやすいリポジトリの完成形

最小構成を少し広げるなら、私は次の形を目指します。

```text
repo-root/
├── AGENTS.md
├── README.md
├── docs/
│   ├── architecture.md
│   ├── editorial-guide.md
│   └── runbook.md
├── spec/
│   ├── requirements.md
│   ├── design.md
│   └── tasks.md
├── src/
│   ├── auth/
│   │   └── AGENTS.md
│   └── billing/
│       └── AGENTS.md
└── .github/
    ├── workflows/
    └── CODEOWNERS
```

ポイントは、AI向けファイルを増やすことではありません。
AIが次に読むべき場所を迷わない状態にすることです。

- `README.md` は人間向けの概要
- `AGENTS.md` はAI向けの入口
- `docs/` は長く効く説明
- `spec/` は作るものの正本
- 局所 `AGENTS.md` は危険領域の注意点
- CI / CODEOWNERS は守りたい制約

この分担があると、AIに「全部読んでいい感じにやって」と頼む必要が減ります。
読む順番と判断基準が、リポジトリ側に埋め込まれるからです。

## 最小チェックリスト

AI向けにリポジトリを整えるなら、まずはこの7点を見ます。

- 入口ファイルが長くなりすぎていないか
- 目的、地図、検証コマンド、正本が分かるか
- 危険領域の局所ルールがあるか
- 必ず守る制約がCIやhooksに寄っているか
- 仕様、設計、タスクの置き場所が分かれているか
- 変わる情報をMarkdownに固定していないか
- AIが迷ったとき、どこを読めばよいか分かるか

このチェックリストで引っかかった箇所を、いきなり全部直す必要はありません。
最初は、AIがよく間違える場所から整えるのが現実的です。

おすすめの順番は次です。

1. 入口ファイルを短くする
2. 検証コマンドを明記する
3. 正本を明記する
4. 危険領域に局所ルールを置く
5. 強制したい制約をCIへ移す

この順番なら、ドキュメント整理だけで終わらず、実際の事故を減らす方向へ進めます。

## まとめ

AIが迷わないリポジトリを作るうえで、長いプロンプトは最初の答えではありません。

入口ファイル、局所ルール、強制する仕組み、正本と外部参照という4つの置き場所を分けておくだけで、AIは推測で進みにくくなります。

AIに何を言うかより先に、AIがどこを見ればよいかを設計する。
ここから始めると、AIコーディングの安定性はかなり変わります。
