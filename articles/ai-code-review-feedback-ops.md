- 指摘を `Must Fix / Should Fix / Nice to Have` に分類して扱う方法

## はじめに

AIコードレビューを導入すると、最初は「思ったより細かく見てくれる」と感じます。

一方で、運用を続けると別の不満が出てきます。

- 指摘が細かすぎて、重要な論点が埋もれる
- チームのルールとズレた一般論が返ってくる
- 同じ変更でも、実行のたびに指摘が変わる

これは、AIレビューが使えないという話ではありません。
**レビュー基準と記録の仕組みがないまま使うと、不安定なレビュアーを毎回呼んでいる状態になる** という話です。

たとえば、React の変更に対して毎回「`useMemo` で最適化したほうがよい」という指摘が返ってくるケースがあります。
一般論としては自然ですが、チームでは React Compiler 前提で `useMemo` を基本的に使わない方針なら、その指摘は正解ではありません。
こうしたズレが毎回起きると、AIレビューは品質向上の仕組みではなく、訂正コストを増やす仕組みになってしまいます。

この記事では、AIレビューを「その場のコメント生成」で終わらせず、指摘の分類・記録・改善ループとして運用する方法を整理します。

## TL;DR

- AIレビューが不安定なのは、モデルの賢さより評価基準の不足が大きい
- まずは `Must Fix / Should Fix / Nice to Have` の3分類を決める
- 採否と理由を記録し、繰り返し出る指摘をルールに戻す
- 毎回完璧に整えず、同じズレが3回出たら仕組み化する

## 1. なぜAIレビューは安定しないのか

AIは、与えられた文脈の中で「それらしいレビュー」を返します。
そのため、レビュー観点が曖昧なままだと、次の3つが起きやすくなります。

### 細かすぎる

命名や表現のような軽い論点に反応しすぎて、本当に止めるべき指摘が埋もれます。
レビュー件数が多いほど良いわけではなく、**重要な指摘が見えなくなるならノイズ** です。

### チームルールとズレる

一般論として正しい提案でも、今のプロジェクトでは採らない設計や書き方があります。
このズレを放置すると、レビューは毎回「一般論 vs 現場判断」のやり取りになります。

### 実行ごとに結果がぶれる

プロンプト、周辺差分、モデルの揺らぎが重なると、同じ変更でも注目点が少しずつ変わります。
基準が固定されていない限り、再現性は上がりません。

## 2. 最初に決めるべきこと

AIレビューの精度を上げる前に、チーム側で最低限の判断基準を決めます。

### 何を指摘対象にするか

たとえば次のように、対象を明示します。

- バグになりうる差分
- 仕様違反
- セキュリティ・権限・データ破壊のリスク
- 保守性を下げる実装

逆に、今回は対象外にするものも先に決めます。

- 完全に好みのスタイル差分
- 現状のチームルールで許容している書き方
- リファクタ提案だけで成立する大規模な話

### どのレベルで止めるか

「指摘する」と「マージを止める」は分けたほうが運用しやすいです。
この境界が曖昧だと、細かい改善提案が全部 blocker のように見えてしまいます。

### 重要度をどう分けるか

最初は 3 段階で十分です。
分類が増えすぎると、記録コストばかり上がって定着しません。

## 3. 指摘は3分類で扱う

おすすめは次の3分類です。

### Must Fix（必ず直す）

放置すると不具合、仕様逸脱、事故につながるものです。

- 明確なバグ
- 権限・認可の穴
- 例外処理漏れ
- 仕様と異なる振る舞い

### Should Fix（できれば直す）

今すぐ壊れないが、保守性や拡張性を下げるものです。

- 責務分離の崩れ
- テストしづらい設計
- 将来の変更コストが高い実装

### Nice to Have（今回は見送れる）

改善余地はあるが、今止める理由は弱いものです。

- 書き換え候補
- 別実装の提案
- 命名や表現の軽微な改善

この3分類を入れるだけで、`細かすぎる` 問題はかなり抑えられます。
重要度の低いコメントをゼロにするのではなく、**止める指摘と流す指摘を分ける** のが目的です。

## 4. チームルールとの差分を埋める

AIに一般論だけを渡すと、レビューはどうしても一般論に寄ります。
プロジェクト固有の判断を反映したいなら、繰り返し出るズレを仕組みに戻す必要があります。

### 毎回出るズレは記録する

たとえば次のようなものです。

- このリポジトリでは `useMemo` を基本的に使わない
- API層ではこのバリデーション方針を採る
- 例外ログはこの粒度で出す

同じ訂正を人間が何度も返しているなら、それは個人の判断ではなくルール候補です。

### ルールの置き場所を決める

ルールは性質に応じて置き場所を分けます。

- 全体ルール: リポジトリ全体のルール文書
- 特定領域の判断: 局所ルールや設計ドキュメント
- 自動で守らせたいこと: lint / CI / テスト
- レビューで毎回見ること: レビューチェックリストや reviewer 用スキル

このリポジトリなら `AGENTS.md` や reviewer 用スキルが置き場所になりますが、重要なのはファイル名ではなく役割です。
自然言語のお願いだけで済ませず、必要に応じて **文書化・自動化・レビュー観点化** に分解します。

## 5. CIに組み込むと、レビュー観点を毎回そろえやすい

ここまでの話を実運用に落とすと、レビュー基準は手元のメモだけでなく、CIから呼ばれるAIレビューの prompt にも入れたほうが安定します。

自分たちの運用でも、GitHub Actions から AIレビューを走らせるときに、単に「このPRをレビューして」と投げるのではなく、レビュー手順と観点を prompt に固定するようにしました。

たとえば、次のような流れです。

### 先にPR状態とCI状態を確認する

レビュー本文の前に、まず PR のコンフリクト有無と CI の失敗状況を確認させます。

```bash
# コンフリクト確認（mergeable: CONFLICTING なら人間が対処してからレビュー）
gh pr view --json mergeable,mergeStateStatus

# CI 状態確認（failed があればコードレビュー前に対処）
gh pr checks
```

これを最初にやるだけで、「コード以前に止まっている問題」と「コードレビューで見る問題」が混ざりにくくなります。

### 変更内容に応じて review_focus（重点観点）を切り替える

変更コンテキストから `review_focus`（「この変更で特に見るべき観点」を prompt に渡すパラメータ）を切り替えます。

| 変更種別 | review_focus の重点観点 |
| -------- | ----------------------- |
| migration を含む | 後方互換性 |
| API変更を含む | スキーマ互換性 |
| shared-components 変更 | cross-site impact |
| 認証変更 | auth / security |
| フロント変更 | stale state / hydration / responsive |

毎回同じ総花的レビューをさせるのではなく、**変更の種類に応じて注視点を増やす** ほうがノイズが減ります。

### レビューを三相に分ける

自分たちの prompt では、レビューを `Upstream / Midstream / Downstream`（設計・実装・テストの上流から下流）の3フェーズに分けています。

- `Upstream`（上流）: 設計と依存方向
- `Midstream`（中流）: 実装、バグ、性能、セキュリティ
- `Downstream`（下流）: テストの有無と品質

こうしておくと、AIの指摘が命名や軽いスタイルに寄りすぎず、「設計」「実装」「テスト」のどこで問題が起きているかを整理しやすくなります。

### 自動検出できることは指摘させない

これはかなり効きました。
`phpstan`、`phpcs`、`eslint`、`typecheck` で拾える問題は、AIレビューでは指摘しないようにしました。

型エラーやフォーマット違反まで AI が繰り返し言い始めると、人間が読む価値のある指摘が埋もれます。
AIには、静的検査では拾いにくい設計判断や文脈依存の問題に集中させたほうがよいです。

### 出力形式も固定する

レビュー結果は自由記述にせず、次の形式に揃えています。

- PR状態
- スコア
- 判定
- 総評
- 指摘事項

指摘は `critical / major / minor / info` の severity 順に並べ、`minor` と `info` は合計件数も絞ります。
この制約がないと、AIは「たくさん言うほど良いレビュー」になりがちです。

スコア運用そのものが正解とは限りません。
ただ、`auto-approve / human-review-recommended / human-review-required` のように出口を固定すると、レビュー結果を人間が扱いやすくなります。

## 6. 指摘を記録する

コメントを読んで終わりにすると、次回また同じ問題が出ます。
最低限、次の項目だけは残したほうが改善につながります。

| 項目 | 意味 |
| --- | --- |
| 指摘カテゴリ | `Must Fix / Should Fix / Nice to Have` |
| 採否 | 採用 / 保留 / 却下 |
| 理由 | なぜその判断にしたか |
| 再発防止策 | ルール化、テンプレート化、lint化など |
| 反映先 | ルール文書 / CI / reviewer skill / 設計書 など |

重要なのは、「AIがこう言った」ではなく **なぜ採否を決めたか** を残すことです。
この理由がないと、あとで見返しても知識として再利用できません。

ただし、すべてのコメントを残す必要はありません。
たとえば、次の粒度から始めると運用しやすいです。

- `Must Fix` は基本的に残す
- `Should Fix` は繰り返し出るものだけ残す
- `Nice to Have` は原則として残さない

## 7. 改善ループは小さく回す

最初から完璧な分類表やダッシュボードを作る必要はありません。
むしろ、小さく回して定着させたほうがよいです。

### まずは週次で3件だけ見る

レビュー結果を全部見直すのではなく、繰り返し出た指摘だけを拾います。
たとえば、次の条件に絞ると運用しやすいです。

- 同じ系統の指摘が 3 回以上出た
- 毎回人間が同じ理由で却下している
- `Should Fix` が実質 `Must Fix` として扱われている

### 仕組み化の優先順位を決める

優先順位は次の順が現実的です。

1. 事故を防ぐものを CI やテストに寄せる
2. 判断のズレが多いものをルール文書に寄せる
3. 表現や観点の揺れを reviewer のチェックリストに寄せる

## 8. 失敗しやすい進め方

### 分類を増やしすぎる

5段階、8段階と増やすと、レビューより記録作業のほうが重くなります。

### 全部をルール化しようとする

プロジェクトには、毎回その場で決めたほうがよい判断もあります。
ルール化すべきなのは、繰り返し出て、再利用価値があるものです。

### AIの揺らぎだけを責める

出力の揺れはゼロになりません。
重要なのは、揺れても最終判断が安定する運用にすることです。

## まとめ

AIコードレビューが不安定に見えるのは、モデルが信用できないからではありません。
レビュー基準、重要度の分類、採否の記録が仕組みとして固定されていないからです。

まずは `Must Fix / Should Fix / Nice to Have` の3分類から始めて、同じズレが繰り返し出たらルールやCIに戻す。
さらに、CIから呼ぶAIレビュー prompt に観点、除外範囲、出力形式まで固定すると、毎回のレビュー品質をかなり揃えやすくなります。

この流れを作ると、AIレビューは単発のコメント生成ではなく、チームの知識を育てる工程に変わります。
