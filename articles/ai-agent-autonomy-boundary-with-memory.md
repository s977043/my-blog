:::message
**この記事で得られること**

- AI エージェントとの対話で起きる「進めて / 確認」のすれ違いの構造
- 自走して良い領域と人間操作が必要な領域の線引き
- Claude Code の memory 機能（セッションを跨いでエージェントが覚えていてくれる領域）で、学び（feedback）を永続化する書き方
:::

## はじめに

AI エージェント（Claude Code / Codex など）と協働していて、同じ確認ループに **5 回連続で入った** ことがあります。

```text
ユーザー: 進めて
エージェント: A / B / C どれから着手しますか？

ユーザー: 進めて
エージェント: A か B のどちらにしますか？

ユーザー: 進めて
エージェント: A を進める前に、対象ファイルを確認しますか？

ユーザー: 進めて
エージェント: 対象ファイルは X / Y / Z です。どれから編集しますか？

ユーザー: 進めて
エージェント: ……（ここで 5 回目。ようやく「自走合図だった」と気づく）
```

こちらは計画を承認して「走ってほしい」と伝えているつもり、向こうは丁寧に選択肢を列挙して待つ。**親切のつもりが、ユーザー体験を悪化させている** パターンでした。

5 回目でようやく「これは自走合図だったのか」と気づいて、次セッション以降は確認を減らすようになったのですが、それを手動で覚えているのは持続しません。この記事は、こうした **対話中の動的な判定境界** を、memory として永続化して扱う設計の話です。

## なぜ確認ループが起きるか

**確認ループは、エージェントのガード設計と、ユーザーの「進めて」が持つ 2 モードが噛み合っていないために起きます。**

エージェント側の立場で振り返ると、確認を挟むのは次のような理由です。

- 破壊的操作（force-push、外部 API 呼び出し）を避けたい
- 優先順位の選択肢が複数あって、勝手に決めたくない
- 「タスクを完了」と判断すべきかが不明

これ自体はガードとして正しいのですが、**ユーザーが既に計画を承認している場面** でも同じガードが発動するため、冗長になります。

一方、ユーザー側の「進めて」には 2 つのモードがあります。

1. **初回合図**: これから計画を立ててほしい、または選択肢を提示してほしい
2. **承認合図**: 直前に提示された計画を承認し、実行に移ってほしい

エージェントがこの 2 つを区別できないと、承認合図をもらっても「どれを選びますか」と返してしまう。

## 自走して良い領域と人間操作が必要な領域

経験的に、エージェントの「自走 OK」と「人間確認必須」は次のように分かれます。

### 自走 OK

- ファイル編集 / 作成 / 削除（ローカル、かつ reversible）
- `git commit`（feature branch へのみ）
- テスト実行 / lint / typecheck
- ドキュメント更新 / コメント追加

### 人間確認必須

- `git push`（feature branch でも、初回 push は確認）
- `git push --force` / `git reset --hard` 等の破壊操作
- `git push main`（保護ブランチへの直接 push）
- PR の作成（`gh pr create`）と マージ
- 外部サービス（SaaS、CMS、本番環境）への操作
- セキュリティ / 認証情報に関わる変更
- 新しい依存関係の導入

### グレーゾーン（状況次第）

- 大規模リファクタ → 影響範囲を見て判断
- ライブラリ更新 → 破壊的変更の有無で判断
- CI 設定変更 → 既存ジョブへの影響で判断

## 「進めて」を自走合図として扱う記法

**ルールはシンプルです。「直前に番号付きプランを提示していて、ユーザーが『進めて』と返したら、リストの上から確認なしで実行する」。** この 1 行を合意できれば、5 回ループは起きません。

私が今採用しているルールは「直前に番号付きプランを提示していて、ユーザーが『進めて』と返したら、リストの上から実行する」です。

具体例:

```
エージェント: 実行計画:
  1. ブランチ作成
  2. 記事リライト
  3. コミット + PR 作成
  どれから進めますか？

ユーザー: 進めて
→ エージェント: 1 から順に実行（確認なし）
```

一方、グレーな破壊操作が途中で出てきた場合だけ確認を挟みます。これで 5 回のループは発生しなくなりました。

## memory で永続化する

**合意したルールは、会話の中で伝えるのではなく memory ファイルに置きます。そうすればセッションを跨いで効きます。**

Claude Code には、conversation（セッション）を跨いで維持される **memory 機能**（エージェント側に保存される永続メモ）があります。上のルールをエージェントの記憶として残すことで、**毎セッション冒頭で説明しなくても** 自走合図として扱ってくれます。

```markdown
---
name: User prefers 進めて/続き as autonomous proceed signal
type: feedback
---

ユーザーは「進めて」「続き」を、直前に提示した計画をそのまま承認して
自走する合図として使う。

**Why:** 過去のセッションで同じ入力を 5 回繰り返し、その度に確認して
止まっていた。ユーザーは「選択肢を列挙して待つ」ではなく「自分で
優先順位を決めて手を動かす」ことを期待していた。

**How to apply:**
- 直前に番号付きプランを提示していて、ユーザーが「進めて」と
  返したら、リストの上から実行する。1 個ずつ確認しない
- ただし以下は依然ユーザー操作が必要: PR のマージ、外部サービスの
  手動操作、本番反映、破壊的変更
- 手持ちの自走タスクを使い切ったら、その旨を正直に報告する。
  無理やり仕事を作らない
```

ポイントは **Why 欄** です。ルールだけでなく、そのルールが生まれた背景（過去の失敗）を記録しておくと、別のエージェントや将来の自分がルールの edge case を判断しやすくなります。

## 自走の限界を認識する

**最後にもう一つ大事なルール。自走タスクを使い切ったら、正直に報告する** こと。

ユーザーから「進めて」と言われ続けると、エージェント側に「何かやらねば」という圧力が生まれます。そこで無理やりスコープ外の改修を生やすと、かえってレビューコストが増えて迷惑になります。

```
エージェント: 手持ちの自走タスクを使い切りました。
以下はユーザー操作が必要です:
- PR #67 のマージ
- note 管理画面での再インポート
- books/ 統合の publish 先確認

どれかご指示いただければ続行できますが、それまで待機します。
```

こう報告できるエージェントのほうが、無理に作業を生やすエージェントより信頼できます。

## 補足: Codex 併用時の共有規約

`AGENTS.md` を併用している読者向けの補足です。

私のリポジトリでは Claude Code と Codex を併用しているため、上記のような判定ルールは `AGENTS.md` にも反映しています。`AGENTS.md` はエージェント共通の規約、`CLAUDE.md` は Claude Code 固有のツール索引、という 2 層構成です。

```markdown
# AGENTS.md

## 自走と確認の境界

- 「進めて」「続き」= 直前のプラン承認 + 自走
- 破壊的操作は都度確認
- 手持ちタスクが尽きたら正直に報告
```

これだけで、エージェント横断で同じ判断が共有できます。

## まとめ

AI エージェントとの「進めて」問題は、**対話中の動的な判定境界をどう共有するか** の問題です。

- 自走 OK / 人間確認必須の領域をマッピングする
- 「進めて」= 直前プランの承認 + 自走、として扱うルールを合意する
- memory に永続化して、セッションを跨いでも同じ判定が効くようにする
- 手持ちタスクが尽きたら、正直に報告する

エージェントが「待つ」か「走る」かの判断を賢くできると、人間側の認知負荷が大きく下がります。丁寧さと自走は、両立させる設計の問題です。
