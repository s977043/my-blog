## はじめに

AI エージェント（Claude Code / Codex）と協働するリポジトリで、学びを追記していくドキュメント `AGENT_LEARNINGS.md` を運用しています。

このファイルには、過去の失敗・成功パターンを記録して、次のセッションで同じ罠を踏まないための経験則を溜めます。運用していくうちに、**AI エージェント固有の落とし穴** に気づきました。

それは、**推測で書いた学びが次セッションの罠になる** というパターンです。この記事は、その体験と、そこから作ったセルフチェックルールを共有します。

## 前段: 推測ベースで学びを断定した

あるセッションで、WXR（WordPress エクスポート形式）の著者フィールドについて、WordPress 一般慣例から以下のように推測しました。

> `<dc:creator>` には author_login（システム ID）を入れる。表示名はここに入れない。

エントリとしては明快で、一見「学び」に見えます。しかし **実測していなかった**。

## 次セッション: 同じスクリプトが失敗

同じスクリプトを別の記事で実行したところ、note 側で「インポート失敗」エラー。公式エクスポートを再解析したら、note の実装では以下のようになっていました。

| 場所 | WordPress 一般慣例（推測・未検証） | note 公式（実測） |
|---|---|---|
| `<dc:creator>` | login ID | **表示名** |
| `<wp:author_display_name>` | 表示名 | **login ID** |

**完全に逆** でした。前のエントリを信じて実装した自分のコードが、次のセッションで自分を裏切る形になりました。

## 訂正の残し方

この時、AGENT_LEARNINGS.md をどう修正するかで迷いました。

- 前のエントリを **削除** する → 履歴が残らず、なぜ書き換えたかが不明
- 前のエントリを **残したまま訂正エントリを追加** する → 2 つの記述が並んで読み手が混乱

採用した方法は **「前エントリに打ち消し線を引いて、後段エントリで訂正する」** です。

```markdown
### 2026-04-18 前段 — 著者フィールドの扱い

1. ~~`<dc:creator>` には author_login を入れる~~ **← 2026-04-18 後段で訂正**
2. ~~表示名は `<wp:author_display_name>` で指定~~ **← 同上**

### 2026-04-18 後段 — note 実装は逆だった

**観察**: 上記エントリの知見で生成した WXR を note に投入したところ
「記事の読み込みに失敗しました」で弾かれた。公式エクスポートを再解析
したところ、著者フィールドの対応が上記と **逆** だった:

- `<dc:creator>` = 表示名（`みね`）
- `<wp:author_display_name>` = login ID（`mine_unilabo`）

**根拠**: 公式エクスポート `note-mine_unilabo-1.xml` との構造比較、
[PR #62](https://github.com/s977043/my-blog/pull/62)
（fix(note): WXR item に wp:* 必須フィールド追加、著者フィールド訂正）
```

こうすることで、「学び自体も誤ることがある」という事実と、それを訂正するプロセス自体が次の学びとして残ります。

## AI 協働固有の罠: コミット履歴からの「自信」

AI エージェントは、コミット履歴や既存のドキュメントを読んで「学び」を書きがちです。そこで起きるのが、**既存の推測をそのまま再記述してしまう** ループです。

- PR コメントで同僚が「多分こうだと思う」と書いた → エージェントが「こうだと判明した」と学びに昇格させる
- ドキュメントに書いてあった推測が、エージェントには「既存の知見」として見える
- 結果として、実測していない推測が「確定した学び」として残る

この対策として、**根拠欄に何が入っているか** を運用ルールに組み込みました。

## 追記時のセルフチェック 5 項目

追記する前に、以下を自問します。

1. **根拠は実測か、推測か**
2. **推測なら、そう明示したか**（「〜と思われる」「〜の可能性が高い」）
3. **実測なら、再現方法を書いたか**（対象ファイル、コマンド、出力）
4. **関連する既存エントリとの矛盾はないか**（矛盾する場合は訂正として追記）
5. **リンクすべき PR / コミットを引用したか**

例えば、根拠欄の書き方として、次のどちらか明確にします。

```markdown
- **根拠（推測）**: WordPress 一般慣例から。note 実装未確認
- **根拠（実測）**: `note-mine_unilabo-1.xml` (commit `<sha>`) を ElementTree で解析
```

## エントリ形式のテンプレート

実運用しているテンプレは次のとおり。訂正エントリの場合は `対策/学び` を省略し、`観察` と `根拠` のみで残すことが多いです。

```markdown
### YYYY-MM-DD — <短い見出し> [カテゴリ]

**観察**: 何が起きたか。事実ベース。

**対策/学び**: どう対応したか、または今後どうするか。

**根拠**: 参照 PR / Issue / commit（あれば）。推測なら明示。
```

カテゴリは次のいずれかを使います。

- `Platform`（プラットフォーム固有の落とし穴）
- `Workflow`（作業フロー・手順系）
- `Gotcha`（ハマりどころ）
- `Performance`（パフォーマンス関連）
- `Convention`（命名・規約）
- `Tooling`（ツール運用）

## エントリの育て方

AGENT_LEARNINGS.md は **追記型**。古いエントリも履歴として残し、訂正が必要なら打ち消し線で明示。削除はしないのが原則です。

- 同じ罠を 2 回以上踏んだら追記
- 特定の成功パターンが再現できると確認できたら追記
- ユーザーから「次回はこうして」と明示的な指示があったら追記

逆に、**書かないこと** は:

- コードを読めば分かる事実（ファイルパスや関数名の羅列）
- 一度きりのタスクの詳細（メモリや会話で完結する内容）

## まとめ

AI エージェント用の lessons-learned ファイルは、以下 3 点を守ると次セッションの罠を減らせます。

- **推測と実測を根拠欄で明示区別** する
- **誤った学びは削除せず、打ち消し線で残して訂正エントリを追記**
- **追記時のセルフチェック 5 項目** で品質を担保

学びは一度書いたら終わり、ではありません。**学び自体も誤ることを前提にした運用** が、AI 協働時代のドキュメントには必要です。
