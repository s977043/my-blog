**この記事で得られること**

- AIエージェントの「自己改善」を、記録 → 棚卸し → 機械化 → 剪定 の4段ループとして設計する方法
- 何でもスキル化しないための抑制ルール（2回繰り返しで棚卸し対象、3回実施でコマンド化）
- 整備した環境が逆に劣化する実例と、使用実績カウンタに基づく剪定のやり方
:::

## はじめに：環境を整備するほど、エージェントは劣化した

AIエージェント（Claude Code / Codex）と協働するリポジトリで、学びの記録・スキル・ガードを1年近く育ててきました。

学びがあればドキュメントに追記する。繰り返す作業があればスキルにする。事故が起きたらガードを足す。どれも個別には正しい改善です。

ところが先日、セットアップ全体を棚卸しして数字を見たら、こうなっていました。

- スキル一覧は約250エントリ。エージェントが毎セッション読み込む一覧の推定サイズは、ハーネスがスキル一覧用に割り当てる文字数予算（コンテキスト予算）の4〜5倍
- 一覧が予算を超えると説明文が切り詰められ、**スキルのルーティング（どの作業にどのスキルを使うかの判断）が静かに劣化する**
- 累計622セッションで使用回数が1〜2回のスキルが19個、常駐していた
- 良かれと入れたアカウント切替ガードのスキルが、このリポジトリでは**逆方向のアカウントを強制しかねない**状態で放置されていた

つまり、「AIが自己成長するための環境」は、**足すだけの運用だと確実に太って劣化する**ということです。人間のチームで言えば、誰も読まない社内Wikiと形骸化したチェックリストが増え続けている状態に近いと思います。

この記事では、この反省を踏まえて運用している自己改善ループを、実測値と失敗談ベースで紹介します。個々の部品（学び記録の書き方、memoryの設計など）は過去記事に切り出してあるので、本記事は**ループ全体の設計と、各段をつなぐ判断基準**に絞ります。

## 自己改善ループの全体像

ループは4段です。前半2段が「増やす側」、後半2段が「増やしすぎを抑える側」で、対になっています。

| 段 | 何をするか | 頻度 | アウトプット |
| --- | --- | --- | --- |
| 1. 記録 | 学び・失敗をその場で追記する | 即時 | lessons-learnedファイル、memory |
| 2. 棚卸し | 繰り返された作業だけを部品化候補にする | 月次 | スキル・コマンド・ワークフロー |
| 3. 機械化 | 人間の注意力に頼るルールをhook/CIに落とす | 事故駆動 | pre-commit hook、CIゲート、チェッカー |
| 4. 剪定 | 使われていないもの・矛盾するものを削る | 週次〜月次 | 削除・統合・無効化 |

ポイントは、**1で書いたものが自動的に2に進むわけではない**ことと、**2〜3で作ったものは必ず4の対象になる**ことです。ここの規律がないと、冒頭の「250エントリ」に一直線です。

## 段1. 記録：判断せず、その場で書く

学びの記録は `AGENT_LEARNINGS.md` という lessons-learned ファイルに集約しています。現在70エントリ。セッション中に非自明な落とし穴や成功パターンを見つけたら、その場で追記します。

この段の設計判断は2つだけです。

- **記録段階では「これは部品化すべきか」を判断しない。** 判断は月次の棚卸し（段2）に送る。書く時点で選別すると、記録自体が億劫になって続かない
- **推測で書かない。** 根拠（PR番号・エラーメッセージ・再現手順）を添える。推測で断定した学びは次セッションの罠になる

2つ目は実際に痛い目を見ていて、詳細は別記事にまとめています。

https://zenn.dev/minewo/articles/agent-learnings-md-operation

また、リポジトリ横断で効く学び（ユーザーの好み、外部サービスの仕様）は、リポジトリ内のファイルではなくエージェントのmemoryに書きます。この使い分けも別記事があります。

https://zenn.dev/minewo/articles/ai-agent-autonomy-boundary-with-memory

## 段2. 棚卸し：「2回以上繰り返された作業」だけを部品化する

月次で、直近30日のセッションログ・memory・既存のスキル/エージェント/ワークフローを突き合わせる棚卸しをやっています。ここでの唯一のフィルタが **2回ルール** です。

> 2回以上繰り返された作業だけを、Reuse / Extend / Create / Skip に分類する。

- **Reuse**: 既存部品で対応できる → 何も作らない（これが一番多い）
- **Extend**: 既存部品に足りない観点を足す
- **Create**: 新規にスキル・コマンド・ワークフローを作る
- **Skip**: 繰り返しているが部品化の費用対効果がない

「1回しかやっていない作業」は、どんなに部品化したくなっても対象外です。AIに頼むとスキルのひな形が数分で出てくるので、作ること自体のコストがほぼゼロに感じられます。しかし本当のコストは作成時ではなく、**常駐して一覧を圧迫し続ける維持費**と、**使われず陳腐化した部品が将来のセッションを誤誘導するリスク**の方です。

実例を挙げると、直近の棚卸しでは生ログからこんな数字が出ました。

- 長時間タスク中に「続き」系の入力を人間が打った回数: 30日で55件 → 反復駆動コマンド（`/loop`）に置き換え（Create）
- セルフレビュー＋外部AI多視点検証を生プロンプトで依頼した回数: 28件 → 既存の多視点レビューコマンドに寄せる（Reuse）

「なんとなく便利そう」ではなく、**自分が実際に繰り返した操作の回数**が部品化の根拠になっているのがミソです。

## 段3. 機械化：口頭チェックリストは、必ず破られる

記録と棚卸しを回していると、「学びとして書いてあるのに、また踏んだ」というケースが出てきます。これが機械化のシグナルです。

うちのリポジトリで実際に機械化した例です。いずれも、**元はドキュメント上の注意書き（＝エージェントと人間の注意力頼み）で、破られた実績がある**ものです。

- **mainへの直接commit防止** → pre-commit hookでブロック。バイパスは環境変数の明示指定のみ
- **CLIツール専用キャッシュディレクトリのcommit混入防止** → pre-commit hookでブロック
- **プラグインが自動注入する一時ブロックの誤commit防止** → pre-commit hookでブロック
- **並列セッション由来の「古いPRがmainの変更を巻き戻す」事故** → マージ前にstate確認・fetch・mergeシミュレーションを一括実行するチェッカーで巻き戻しを検知してexit 1
- **レビュー必須指摘が未解決のままの公開** → CIゲートで検知

4つ目のチェッカーは、squash済み記事が別ベースのPRで巻き戻された実インシデントが起点になっています。

注意書きとhookの違いは、**破るのに意図が必要になる**ことです。注意書きは読み飛ばせますが、hookはバイパスフラグを打たない限り止まります。エージェント運用では「読み飛ばし」が人間より高頻度で起きるので、この差は人間チーム以上に効きます。

### コマンド化は「3回実施」まで待つ

機械化の中でも、公開作業のような**事故ると取り返しがつかない操作のコマンド化**は、さらに慎重にしています。うちの基準は **3回ルール** です。

> 同型の作業を手順書ベースで3回完走して、初めてコマンド化する。

実際、ZennとQiitaの公開パイプラインは長らく「あえてコマンド化しない」運用でした。npm scriptと手順書で3サイクルずつ完走し、手順が安定した2026年7月にようやく `/publish-zenn` `/publish-qiita` としてコマンド化しています。それでも公開とマージの実行だけは自動化せず、人間の承認ゲートを残しました。

1回やっただけの手順をコマンド化すると、手順自体の欠陥ごと固定化されます。3回やると、毎回微妙に違った部分（＝人間の判断が必要な部分）と、完全に同じだった部分（＝自動化してよい部分）が分離されるので、その境界にゲートを置けます。

このあたりの「セッション直後の振り返りを改善フローにつなげる」やり方は別記事にあります。

https://zenn.dev/minewo/articles/engineering-process-improvement-skill

## 段4. 剪定：使用実績カウンタで削る

冒頭の劣化の話に戻ります。段1〜3を1年回した結果が「スキル250エントリ・予算超過」でした。剪定なしの自己改善ループは、**改善のたびに環境が重くなる片道ループ**です。

剪定で重要なのは、印象ではなく**使用実績**で判断することです。エージェントのハーネス（Claude Code）はスキル・プラグインごとに累計使用回数と最終使用日時を記録しているので、これを直接見ます（どこで確認できるかはハーネスのバージョンや構成によって異なるため、お使いの環境の統計・管理機能を確認してください。ここに挙げる数字はうちの環境で観測した実測値です）。

直近の剪定の実測です。

- 累計622セッション中、使用1〜2回のスキル19個を無効化（設定1行なのでいつでも戻せる）
- 同一機能のスキルが3系統重複していたものを1つに統合
- 累計3回しか使われていないプラグインを無効化
- エージェント設定ファイル内の、ハーネスが自動読み込みする一覧と二重掲載になっていたツール表を削除

そして剪定で一番怖かった発見がこれです。

:::message alert
別プロジェクト用に作ったGitHubアカウント切替ガードのスキルが、ユーザーレベルに常駐していた。そのスキルの「期待アカウント」は別プロジェクトの値で、**このリポジトリで発動すると正しいアカウントを間違いとして逆に切り替えようとする**。
:::

ガードは足した瞬間が一番正しく、環境が変わるほど**逆ガード**になるリスクを持ちます。使われていないガードは「無害な余分」ではなく「文脈がズレた時限式の事故」なので、剪定の優先対象です。

memoryについても同様に、週次で重複・矛盾・陳腐化を検出して人間確認のうえ削る運用（うちでは memory-dream と呼んでいる、memoryの定期クリーニング）を入れています。記録（段1）を躊躇なく書けるのは、後段に剪定があるからです。

## ループを回すカレンダー

まとめると、運用リズムはこうなります。

- **即時**: 学び・失敗を記録する（段1）。判断はしない
- **事故が起きたら**: 注意書きで済ませず、hook/CI化できないか検討する（段3）
- **週次**: memoryの剪定（段4）
- **月次**: 30日分のログで棚卸し（段2）。2回ルールで分類し、Createは最小限。あわせて使用実績カウンタで環境の剪定（段4）

「AIが自己成長する環境」というと、自動で学習して自動で強くなる仕組みを想像しがちですが、実際に効いたのは地味な逆方向の規律でした。**増やす判断を遅らせ（2回・3回ルール）、減らす判断を定期化する（週次・月次の剪定）**。この2つがあって初めて、記録と機械化が資産として積み上がります。

## おわりに

自己改善ループの4段は、独立した施策ではなく互いの前提です。

- 剪定があるから、記録を躊躇なく書ける
- 記録があるから、棚卸しが印象論にならない
- 棚卸しの回数実績があるから、機械化の対象を絞れる
- 機械化で注意力が浮くから、剪定のような地味な作業に回せる

エージェントの性能が上がるほど「環境を足すコスト」は下がり続けます。だからこそ、**足す速度ではなく、足すものを絞る規律と減らす仕組み**の方が、これからの差になっていくはずです。

うちのループもまだ発展途上で、剪定の基準（何セッション・何回未満で削るか）は手探りです。同じようにエージェント環境を育てている方の運用も、ぜひ聞かせてください。
