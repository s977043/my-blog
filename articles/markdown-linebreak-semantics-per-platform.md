## はじめに

「スマホで読みやすくしたい」と思って、段落内で `  \n`（行末 2 スペース + 改行）を多用したら、**note で投稿した実記事が視覚的に間延び** しました。

書いたときは Markdown プレビューで問題なかったのに、note に取り込むと段落間の余白に加えて文中改行の余白が重なり、スマホ表示で呼吸が浅くなったような読み味になります。

この現象を追っていくと、**Markdown の同じ記法がプラットフォームごとに異なる解釈を受ける** という、意外と知られていないポイントに行き当たりました。この記事は、note / Zenn / Qiita で 2 スペース改行がどう扱われるかを並べて、運用ルールを整理したものです。

## CommonMark 仕様での 2 スペース改行

まず仕様の確認から。CommonMark（Markdown の事実上の標準）では、行末に 2 つ以上のスペースを置いて改行すると **hard line break** として扱われ、HTML の `<br />` に変換されます。

```markdown
この行は
このように改行されます
```

↓ `<p>` で包まれて段落内の空白改行

```html
<p>この行は このように改行されます</p>
```

一方、2 スペース改行を使うと:

```markdown
この行は  
このように改行されます
```

↓ `<br />` が入る

```html
<p>この行は<br />このように改行されます</p>
```

これは仕様として正しい動きで、Python-Markdown / marked / remark などの主要実装も同じ挙動です。問題は、この HTML が各プラットフォームで **どう視覚化されるか** が揃っていないことです。

## 実例比較: 同じ Markdown を 3 プラットフォームに投げる

同じソースを 3 箇所で表示させると差が出ます。以下はイメージです。

### Zenn

- `<br />` はインライン改行として素直に表示される
- 段落間の余白は標準（1.5em 相当）
- **期待通りに読める**

### Qiita

- `<br />` は Zenn とほぼ同じ
- 段落間の余白はやや狭め
- **期待通りに読める**

### note

- `<br />` が `<p>` 間と同等以上の余白を生み出す
- 段落間に加えて文中 `<br />` でも 1 行分の空白が入る
- 2 スペース改行を多用した記事はスマホで **間延びして見える**

note の CSS 設計は、読み物の「余韻」を優先する方向にチューニングされていて、`<br />` も段落級の余白として扱われます。これは note の UX デザインとしては妥当ですが、Zenn / Qiita と同じ感覚で書くと事故ります。

## 実測: 11 箇所の 2 スペース改行を入れた記事の末路

私が書いた記事（約 5,500 字）に、レビュー指摘に従って 11 箇所の 2 スペース改行を導入したところ:

- Zenn プレビュー: 問題なし。文のリズムが効く
- note 下書きプレビュー: スマホで間延び、段落境界が曖昧化

対処として 11 箇所全ての 2 スペースを除去したところ、note 側の余白が適正化されました。**Markdown の意図（文中の息継ぎ）を、note では空行による段落分けで表現する** のが素直です。

```python
# 行末 2 スペース除去の単純置換
import re
fixed = re.sub(r" +$", "", source, flags=re.MULTILINE)
```

## プラットフォーム別の推奨ルール

実運用の肌感覚を整理すると、こう分かれます。

### Zenn

- 2 スペース改行は **使ってよい**（リスト内や段落内での短文リズム作りに有効）
- 多用しても表示崩れは起きにくい
- ただし、note との共通ソースを使う場合は要注意

### Qiita

- Zenn と同じ感覚で OK
- 2 スペース改行は適度なら有効

### note

- 2 スペース改行は **原則避ける**
- 文中で改行したい意図があるなら、**空行で段落を分ける** 方が意図に近い表示になる
- 既存記事で 2 スペース改行を使っている場合も、スマホ実機でチェック

### 共通記事をマルチプラットフォーム展開する場合

- **ソースは「2 スペース改行なし」** で書く
- 段落分けは空行で表現
- Zenn 固有のリズム作りは Zenn 版の個別編集で追加

## AI 生成時の落とし穴

AI に「読みやすくするために改行を入れて」と頼むと、 2 スペース改行を多用した出力が返ってくることが多い印象です。理由を推測すると:

- LLM の事前学習データに GitHub Markdown / Stack Overflow / Qiita 等が多く含まれる
- これらは 2 スペース改行が問題になりにくいプラットフォーム
- note / ブログ CMS の CSS 事情は学習データで過小表現

AI 生成した Markdown を note に投入する場合は、**行末 2 スペースを全除去するフィルタ** を挟むのが安全です。

```bash
# sed で一括除去
sed -i '' 's/  *$//' articles_note/new/<slug>.md
```

## 検出と予防

運用に組み込むなら、レビューエージェントの観点に追加するのが楽です。私のリポジトリでは、記事レビュー用エージェントに以下の観点を入れています。

- 本文中の `  $`（行末スペース）をカウント
- 全角 5 行以上連続で `<br />` が入るなら警告
- 空行による段落分けで代替可能かを確認

```python
# 簡易チェック
two_space_breaks = sum(1 for line in text.split("\n") if line.endswith("  "))
if two_space_breaks > 5:
    print(f"[WARN] 2 スペース改行 {two_space_breaks} 箇所。note で余白過多の可能性")
```

## まとめ

- CommonMark の 2 スペース改行は **`<br />` に変換される** 仕様
- Zenn / Qiita は `<br />` を控えめな改行として表示、note は段落級の余白として表示
- **note 向けは原則避ける**、マルチ展開するならソースを 2 スペース改行なしで書く
- AI 生成 Markdown は 2 スペース改行を多用しがち、フィルタで除去する運用が安全

Markdown は「書き手にとって同じ記法でも、読み手の環境で表示が変わる」ことを常に意識すると、プラットフォーム横断の運用が安定します。

note で読まれる前提で書くなら、**2 スペース改行よりも空行による段落分け**。シンプルですが、ここを揃えるだけで「なんか読みづらい」が減ります。
