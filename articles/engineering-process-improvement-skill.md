## はじめに

Claude Code で開発セッションを回すようになってから、振り返りの扱いに困っていました。

複数 PR をマージした直後や、マルチエージェント品質レビューの後、リリース作業の直後。
振り返りたい対象はあるのに、**どこを範囲にして、何を残せば次に効くのか** が毎回ブレていました。

結果として、振り返りは「今回はこれが大変だった」「次は気をつけよう」のループで終わりがちでした。
仕組みとして残せず、同じ失敗を 2 〜 3 回繰り返したあたりで、さすがに運用として回していく必要を感じました。

そこで導入したのが、`engineering-process-improvement` という user-level スキルです。

## この記事で伝えたいこと

この記事は、開発セッション直後の振り返りを「感想」で終わらせず、次回に効く改善フローへつなげる運用の話です。

自分は `engineering-process-improvement` という user-level スキルを使って、**振り返り → 改善提案 → 実装 → PR 化** までを一気通貫で回しています。

本記事では、8 ステップの通常モード、軽量モード、A/B/C 分類、自動ガード優先の考え方、そして導入後に何が変わったかをまとめます。

---

## このスキルの位置付け

振り返り系のスキルはいくつかありますが、本記事で扱うのは「スプリント全体」ではなく「直前の開発セッション単位」の振り返りです。
目的は、セッション直後の気づきをその場で改善候補に変え、再発防止できる仕組みとして残すことです。

| スキル | スコープ | タイミング | 成果物 |
|---|---|---|---|
| `engineering-process-improvement` | 直前の開発セッション（複数 PR、品質レビュー、リリース等） | セッション直後 | 改善 PR（通常 1 本） |
| `retrospective-improvement` | スプリント全体 | Sprint Retrospective | 次スプリントで試す改善実験 |
| `engineer-skill-creator` | スキル自体の新規作成 | 新しい運用パターンを定着させたいとき | 新スキルの SKILL.md |

本記事で扱うのは一番上の `engineering-process-improvement` です。既存スキルの改善（meta-retrospective）と、新スキル作成（`engineer-skill-creator`）の使い分けは後述します。

## まず全体像：8 ステップ

通常モードでは、次の 8 ステップで進めます。

1. 範囲を決める
2. 振り返り表を作る
3. 改善アクションを分類する
4. 実装する項目を合意する
5. 改善を実装する
6. 検証する
7. PR を作る
8. 確認してマージする

要するに、「振り返る」だけで終わらせず、改善を次回に効く形に変換する流れです。
以降で各ステップを詳しく見ていきます。

---

## いつ使うか / 使わないとき

### 使う場面

- 複数 PR をマージした後のクールダウンフェーズ
- リリース作業完了後
- マルチエージェント品質レビュー完了後
- ユーザーから明示的に「振り返りして」と依頼されたとき
- 失敗・手戻り・force-push が発生したセッションの直後

### 使わない場面

- 単発の小さな修正の直後（振り返り対象が薄い）
- セッション開始直後（振り返る材料がない）
- ユーザーが先を急いでいる時（軽量な進捗報告で代用）

---

## 各ステップ詳細

### Step 1: 範囲特定

まず、前回の振り返り PR を境界として、そこから HEAD までを今回の対象範囲にします。初回実行や該当 PR がない場合は、フォールバック用のコミットを起点にします。

対象はコミットだけでなく、force-push、CI failure、stash 混乱などのインシデント痕跡も含めて見ます。

```bash
# 例: タイトル限定検索で前回境界を特定、なければ chore(release) にフォールバック
PREV=$(gh pr list --state merged \
  --search 'in:title retrospective OR "process-improvements" OR 振り返り' \
  --limit 1 --json mergeCommit --jq '.[0].mergeCommit.oid')

if [ -z "$PREV" ]; then
  PREV=$(git log --grep '^chore(release)' -n 1 --format=%H)
fi

git log --oneline "${PREV}..HEAD"
git reflog --date=iso | head -30
```

### Step 2: 振り返り表

2 つのテーブルを必ず作ります。

**✅ うまくいったこと（Keep doing）**

並列性、段階性、早期検知など、継続すべきパターンを列挙。

**⚠️ 改善が必要なこと（Change）**

| # | 問題 | 発生箇所 | 影響 |
|---|---|---|---|

**このステップのポイントは、「大変だった」で終わらせず、影響を時間・リスク・コストで書くことです。**
「手戻り 30 分」「リカバリに force-push 1 回」など、数字で書けると次のステップで自動ガードに落としやすくなります。

### Step 3: 改善アクションの分類

各改善項目を 3 種類に分類します。

- **A. ドキュメント追記** — AGENT_LEARNINGS.md / CLAUDE.md / README など、次回読む場所に明文化
- **B. 自動ガード** — CI / pre-commit hook / lint ルール。機械が止めてくれる仕組み
- **C. ツール運用変更** — `run_in_background` の使い分け、作業順序の固定化など

この運用で一番効いているのは、改善候補を「思いつき」で終わらせず、**A / B / C のどこに落とすかで考える**点です。

特に重要なのは、ドキュメントだけで終わらせず、**最低 1 件は B（自動ガード）に落とす** ことです。注意力に頼る改善より、機械が止めてくれる改善のほうが再発防止として強いからです。

分類前には、既存の `AGENT_LEARNINGS.md` / `CLAUDE.md` / `AGENTS.md` を通読して重複を避けます。並行セッションが同種の振り返り PR を既にマージしていることもあるので、前回境界以降のマージ済み PR 本文も確認します。

### Step 4: ユーザー合意

どの改善項目を実装するかを提示し、優先順位の確認を取ります。
全部やる・一部だけやる・順番を変える、のどれでも即応できる粒度で提示します。

### Step 5: 実装（ブランチ + コミット衛生）

実装フェーズでは、**Step 2 で見つけた失敗を自分で再発させない**ことを最優先にします。特に WIP の巻き込みや雑な `add` は避けます。

```bash
# 1) ブランチ衛生
git status --short                         # WIP の有無を確認
git branch --list 'chore/process-*'        # 衝突チェック
git checkout -b chore/process-improvements

# 2) 変更を適用
#    例: AGENT_LEARNINGS.md の追記 / .github/workflows/check.yml に CI ステップ追加 / lint ルール更新
#    手順で決めた A/B/C の改善項目だけを編集する

# 3) ステージング（add -A は使わない）
git add path/to/AGENT_LEARNINGS.md .github/workflows/check.yml
git status --short                         # コミット前再確認

# 4) コミット
git commit
```

WIP と同一ファイルを触る必要がある場合は、`git stash push -m "wip" -- path/a path/b` で先に退避し、改善 PR のマージ後に `git stash pop` で戻します。

### Step 6: 検証

検証は変更種別に応じて切り替えます。docs-only なら lint のみ、コード変更ありなら lint / typecheck / test、CI や hook の変更がある場合はローカルで同等の確認まで行います。

過剰検証は時間の無駄なので、変更スコープに合わせて絞るのがコツです。

### Step 7: PR 作成

PR 本文では、A / B / C の内訳、Test plan、関連セッション（前回振り返り PR + 対象セッションの PR 群 + 発生インシデント）の 3 点を揃えます。

**前回の振り返り PR へのリンクは必須**で、チェーンを辿れるようにしておきます。「なぜこのガードが入ったのか」を半年後にも追跡できるようにするためです。

最小テンプレートは次のとおり。

```markdown
## Summary
直前の <対象セッション> の振り返りから、N 件の改善を実装:
- A（ドキュメント）: X 件
- B（自動ガード）: Y 件
- C（ツール運用）: Z 件

## Test plan
- [x] ローカル検証コマンド

## 関連セッション
- 前回の振り返り PR: #XX
- 対象セッションのマージ済み PR: #YY, #ZZ
- 発生したインシデント: <リンク>
```

### Step 8: ユーザー確認 → マージ

承認後に `gh pr merge --squash`。auto-merge も可。

## 軽量モード

対象セッションが薄いとき、8 ステップ全部を回すと改善 PR がノイズになります。次のいずれかに当てはまるときは軽量モードに切り替えます。

- 対象コミット数 ≤ 3 かつ force-push / orphan commit / stash 混乱などのインシデントなし
- 改善アクション候補が 2 件以下、かつ自動ガード化する価値のある項目がない
- ユーザーが明示的に「振り返りだけ」「PR 不要」と伝えている

軽量モードでは、Step 1（範囲特定）と Step 2（振り返り表）だけ実施し、改善候補がゼロ〜1 件なら PR 作成は省略します。改善項目がゼロなら「健全な状態」で完了を宣言して終わります。

**このモードがあるおかげで、「振り返ったら必ず改善 PR を作る」という過剰運用を避けられます。**
改善がない状態を「改善がない」と結論付けられることも、健全なサイクルの一部です。

代わりに、**Carry-over observations（次回持ち越しメモ）** を短く残しておきます。今回対処しなかったが次回影響しそうな観察を 1 〜 2 行で残すだけで、次の振り返りで拾い直せます。

```text
ユーザーの 12d6e28 は stash→checkout --theirs の迂回を記録している。
今後の同種作業では `git reset --soft` + `git add -p` を優先候補にする。
```

## Quality Gates（運用で守るルール）

このスキルを「ただの振り返りテンプレ」で終わらせないために、いくつか強制ルールを置いています。自分の気分や記憶に依存しないためのガードです。

- 振り返り表に必ず **うまくいったこと** も含める（失敗ばかり書かない）
- 「影響」を必ず書く（時間・リスク・コスト）
- 改善項目の最低 1 件は **B（自動ガード）** に落とす
- 実装フェーズで自分が指摘した失敗を繰り返さない（`git add -A` 禁止など）
- PR 本文に対象セッションへのリンクを含める

特に最後の 2 つ。
Step 2 で「雑に `add -A` して余計な変更を混ぜた」と指摘したのに、Step 5 で同じことをやる、という悪いループをやらかしがち。`git status --short` を Step 5 の頭と末に必ず走らせる、で回避します。

## Anti-patterns

運用していて繰り返し踏みかけるアンチパターン。

- **感想文化する** — 「〜が大変でした」で終わる。必ず次回の具体的な手を書く
- **ドキュメントだけで済ます** — 忘れる前提で自動ガード化を優先する
- **全項目を一括 PR にする** — レビュー困難。独立項目は別 PR か、最低でもコミット分離
- **WIP ファイルを巻き込んで force-push** — 振り返りで指摘した失敗を実装中に繰り返す最悪パターン
- **ユーザー合意なしで実装開始** — 優先順位がズレると手戻りになる

## 導入して変わったこと

実際に運用して、自分の行動が変わった点を正直に並べてみます。

### Before

- セッション後に「今回はこれが大変だった」で終わる
- 学びは頭の中か、バラバラな場所のメモに残る
- 同じ失敗を次のセッションで踏む
- `git add -A` でコミットしがちで、たまに余計な変更が混ざる
- 改善したい項目はあるのに、PR にまで落とす気力が続かない

### After

- セッション終了時にスキルを呼び出す運用が定着
- 振り返り表で影響の大きさまで書かれる
- 改善 PR が前回境界からチェーンで辿れる
- Quality Gates に「`add -A` 禁止」が入っていて、機械的に止まる
- 改善項目が B（自動ガード）に落ちるので、次のセッションで注意力を使わずに済む

### 特に効いた 3 つの仕組み

**1. 影響を数字で書く癖**

「大変だった」が「手戻り 30 分、force-push 1 回、stash ファイル 2 個混入」のように書かれるようになりました。これが書けると CI ガードの設計ができます。感情ではなく構造で失敗を記述できるようになります。

**2. 自動ガード優先の方針**

ドキュメントに書くだけで済ませる誘惑は強い。「書いた、これで改善した」と言いたくなります。
でも、このスキルの Quality Gates に「最低 1 件は B に落とす」が入っているので、強制的に CI / pre-commit / lint のどれかを設計することになります。結果として再発防止が強くなりました。

**3. 前回境界からのチェーン**

PR 本文に「前回の振り返り PR: #XX」を書くことで、改善の積み重ねが履歴として追跡できます。
「なぜこの CI チェックが入ったのか」を後から辿れるので、運用が属人化しにくい。これは小さな設計ですが、半年後に効いてきます。

## 運用してみた気づき

スキル化して 10 回ほど回してから気づいたことをいくつか挙げます。

### 振り返りが「怖くない」感覚になった

失敗を書き出すのは、本来あまり気が進まない作業です。
でも「書いた失敗が仕組みに変換される」と分かっていると、書くことへの抵抗が減ります。
「書いた失敗は消える（自動ガード化されて再発しない）」という実感が、振り返りを前向きな作業に変えた気がします。

### 軽量モード + Carry-over（次回持ち越しメモ）の組み合わせが効く

軽量モードで残す Carry-over が、次のセッションの振り返りで驚くほど回収されます。
自分が目の前の作業に集中している間も、並行で別の情報が積まれている。それを 1 セッション持ち越すだけで、統合的な振り返りができます。

改善 PR を無理に量産しないで済むのも、軽量モードの大きなメリットです。「今回は健全だった」と結論して終われることが、振り返りサイクル自体の健全性につながります。

## Meta-retrospective（スキル自体の振り返り）

このスキル自体も改善対象にします。ただし、通常の振り返りを **3 回以上** 回してから見直すルールにしています。サンプル不足の段階で構造を変えると、運用そのものが不安定になるからです。

改善ターゲットは `~/.claude/skills/engineering-process-improvement/SKILL.md` 本体。観察項目は「どの Step で迷ったか」「どの Quality Gate を守れたか」「軽量／通常モードの比率」など、スキル構造の使われ方です。

`~/.claude/skills/` を git 管理していない場合、PR は発生せず直接編集 + チャット報告で完了します。git 管理している場合は通常の Step 5〜8 にしたがって改善 PR にまとめます。

2 回目で「このスキル改善したい」と思ったときは、だいたい実際にはスキル自体ではなく自分の使い方が悪いだけでした。3 回ルールは、その踏み込みすぎを止めるための安全装置として機能しています。

:::message
**`engineer-skill-creator` との住み分け**

- 新しい運用パターンをスキルとして立ち上げる → `engineer-skill-creator`
- 既存スキルの改善 → 本スキルの meta-retrospective モード
:::

## まとめ

`engineering-process-improvement` は、開発セッション直後の振り返りを、仕組みに変換する運用スキルです。

- 8 ステップの通常モード、軽量モード、meta-retrospective の 3 モード
- 改善は A（ドキュメント）/ B（自動ガード）/ C（ツール運用）の 3 分類
- 最低 1 件は B（自動ガード）に落とす
- 前回境界からチェーンを辿れる PR 本文

**この運用で大事なのは、振り返りを丁寧に書くことそのものではありません。書いた気づきを、次回の事故率を下げる仕組みに変えることです。**

運用し始めてから、振り返りが感想で終わることがなくなり、同じ失敗を踏む頻度が目に見えて減りました。
改善サイクルが、注意力ではなく設計で回るようになっている感覚があります。

気合いではなく、仕組みで事故率を下げる。
これを実現するための土台として、かなり手応えのあるスキルになっています。
