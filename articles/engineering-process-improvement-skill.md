## はじめに

Claude Code で開発セッションを回すようになってから、振り返りの扱いに困っていました。

複数 PR をマージした直後や、マルチエージェント品質レビューの後、リリース作業の直後。
振り返りたい対象はあるのに、**どこを範囲にして、何を残せば次に効くのか** が毎回ブレていました。

結果として、振り返りは「今回はこれが大変だった」「次は気をつけよう」のループで終わりがちでした。
仕組みとして残せず、同じ失敗を 2 〜 3 回繰り返したあたりで、さすがに運用として回していく必要を感じました。

そこで導入したのが、`engineering-process-improvement` という user-level スキルです。
開発セッション直後に呼び出し、**振り返り → 改善提案 → 実装 → PR 化** までを一気通貫で進めます。

この記事は、その実運用と、導入前後の行動変容をまとめたものです。

---

## このスキルの位置付け

振り返り系のスキルは複数あるので、まず使い分けを整理しておきます。

| スキル | スコープ | タイミング | 成果物 |
|---|---|---|---|
| `engineering-process-improvement` | 直前の開発セッション（複数PR、品質レビュー、リリース等） | セッション直後 | 改善 PR（通常 1 本） |
| `retrospective-improvement` | スプリント全体 | Sprint Retrospective | 次スプリントで試す改善実験 |

本記事で扱うのは前者です。
**セッション単位の振り返りで、チームが再発防止できる仕組みを PR として残す** のが目的。

---

## いつ使うか / 使わないとき

### 使う場面

- 複数 PR をマージした後のクールダウンフェーズ
- リリース作業完了後
- マルチエージェント品質レビュー完了後
- ユーザーから明示的に「振り返りして」と依頼されたとき
- 失敗・手戻り・force-push が発生したセッションの直後

### 使わない場面

- 単発の小さな修正の直後（振り返り対象が薄い）
- セッション開始直後（振り返る材料がない）
- ユーザーが先を急いでいる時（軽量な進捗報告で代用）

## 8 ステップの中身

通常モードは 8 ステップで動きます。

### Step 1: 範囲特定

前回の振り返り PR を境界として、そこから HEAD までを対象にします。

```bash
# 前回の振り返り PR を特定（タイトル限定検索）
PREV=$(gh pr list --state merged \
  --search 'in:title retrospective OR "process-improvements" OR 振り返り' \
  --limit 1 --json mergeCommit --jq '.[0].mergeCommit.oid')

# 初回実行や該当なしのフォールバック
if [ -z "$PREV" ]; then
  PREV=$(git log --grep '^chore(release)' -n 1 --format=%H)
fi

if [ -z "$PREV" ]; then
  echo "前回境界が特定できません。範囲を手動指定してください。" >&2
  exit 1
fi
echo "前回境界: $PREV"

# そこから HEAD までのコミットが対象範囲
git log --oneline "${PREV}..HEAD"

# インシデント痕跡の確認
git reflog --date=iso | head -30
```

対象 PR、コミット、発生インシデント（force-push、orphan commit、CI fail など）をリスト化します。
`in:title` は OR 節全体に効くよう先頭に置くのが `gh pr list --search`（GitHub Search API）の慣習です。

### Step 2: 振り返り表

2 つのテーブルを必ず作ります。

**✅ うまくいったこと（Keep doing）**

並列性、段階性、早期検知など、継続すべきパターンを列挙。

**⚠️ 改善が必要なこと（Change）**

| # | 問題 | 発生箇所 | 影響 |
|---|---|---|---|

ここで重要なのは **影響の大きさ** を必ず書くことです。
「大変だった」ではなく、「手戻り30分」「リカバリに force-push 1回」など、時間・リスク・コストで記述する。

### Step 3: 改善アクションの分類

各改善項目を 3 種類に分類します。

- **A. ドキュメント追記** — AGENT_LEARNINGS.md / CLAUDE.md / README など、次回読む場所に明文化
- **B. 自動ガード** — CI / pre-commit hook / lint ルール。機械が止めてくれる仕組み
- **C. ツール運用変更** — `run_in_background` の使い分け、作業順序の固定化など

A だけで済ますのは弱い。**最低 1 件は B（自動ガード）に落とす** のがルールです。

分類前に、既存の `AGENT_LEARNINGS.md` / `CLAUDE.md` / `AGENTS.md` を通読して重複を避けます。
並行セッションが同種の振り返り PR を既にマージしていることもあるので、前回境界以降のマージ済み PR 本文も確認します。

### Step 4: ユーザー合意

どの改善項目を実装するかを提示し、優先順位の確認を取ります。
全部やる・一部だけやる・順番を変える、のどれでも即応できる粒度で提示する。

### Step 5: 実装（ブランチ + コミット衛生）

ここが一番大事で、**自分が Step 2 で指摘した失敗を、実装中に絶対再発させない** ようにします。

```bash
git status --short                         # WIP の有無を確認
git branch --list chore/process-*          # 衝突チェック
git checkout -b chore/process-improvements

# 変更を適用

git add <明示パス>                          # add -A は禁止
git status --short                         # コミット前再確認
git commit
```

WIP と同一ファイルを触る必要がある場合は、`git stash push -m "wip before process-improvement" -- path/a path/b` でユーザー WIP を先に退避し、改善 PR のマージ後に `git stash pop` で戻します。

### Step 6: 検証

変更種別に応じて検証スコープを切り替えます。過剰検証は時間の無駄。

- **docs-only PR**（*.md / コメント更新のみ）: `npm run lint` だけで十分
- **コード変更あり**: lint + typecheck + test の 3 点セット必須
- **CI 追加ステップあり**: ローカルで同じスクリプトを dry-run
- **husky/hook 変更あり**: 実際のコミット操作でフックが期待どおり発火/bypass するか手動確認

### Step 7: PR 作成

本文テンプレート:

```markdown
## Summary
直前の <対象セッション> の振り返りから、N 件の改善を実装:

### A. ドキュメント追記（X件）
- 項目1: Why

### B. 自動ガード（Y件）
- CI ステップ追加: 何を検知するか

### C. ツール運用（Z件）
- ルール: 何を避け、何を使うか

## Test plan
- [x] ローカル検証コマンド
- [x] CI ステップの dry-run（該当時）

## 関連セッション
- 前回の振り返り PR: #XX
- 対象セッションのマージ済み PR: #YY, #ZZ
- 発生したインシデント: <force-push / orphan commit / CI fail 等へのリンク>
```

**前回の振り返り PR へのリンクは必須** で、チェーンを辿れるようにしておく。
「なぜこのガードが入ったのか」を後から追跡できます。

### Step 8: ユーザー確認 → マージ

承認後に `gh pr merge --squash`。auto-merge も可。

## 軽量モード

対象セッションが薄いとき、8 ステップ全部を回すと改善 PR がノイズになります。
次のいずれかに当てはまるときは軽量モードに切り替えます。

- 対象コミット数 ≤ 3 かつ force-push / orphan commit / stash混乱 などのインシデントなし
- 改善アクション候補が 2 件以下、かつ自動ガード化する価値のある項目がない
- ユーザーが明示的に「振り返りだけ」「PR 不要」と伝えている

軽量モードの構成:

1. Step 1（範囲特定）と Step 2（振り返り表）は実施
2. Step 3（分類）は簡略化。候補がゼロ〜1 件なら項目名のみで OK
3. Step 4（合意）は候補提示のみ、「振り返りだけで終える」を明示選択肢として提示
4. ユーザーが PR 作成を選んだ場合のみ Step 5 以降を実施
5. 改善項目がゼロなら「健全な状態」で完了を宣言して終わる
6. **Carry-over observations** を短く残す（次回の振り返りで回収するため）

**なぜ必要か**: 「振り返り依頼 → 必ず PR 作成」と決め打ちすると、対象が薄いセッションでも意味の小さい PR を生んでノイズになる。改善がない状態を「改善がない」と結論付けられるのが健全なサイクル。

Carry-over の例:

```
ユーザーの 12d6e28 は stash→checkout --theirs の迂回を記録している。
今後の同種作業では `git reset --soft` + `git add -p` を優先候補にする。
```

## Quality Gates

スキルを運用するうえで、必ず守るゲート。

- 振り返り表に必ず **うまくいったこと** も含める（失敗ばかり書かない）
- 「影響」を必ず書く（時間・リスク・コスト）
- 改善項目の最低 1 件は **B（自動ガード）** に落とす
- 実装フェーズで自分が指摘した失敗を繰り返さない（`git add -A` 禁止など）
- PR 本文に対象セッションへのリンクを含める

特に最後の 2 つ。
Step 2 で「雑に `add -A` して余計な変更を混ぜた」と指摘したのに、Step 5 で同じことをやる、という悪いループをやらかしがち。
`git status --short` を Step 5 の頭と末に必ず走らせる、で回避する。

## Meta-retrospective（スキル自身の改善）

このスキルには、自分自身を改善するためのモードがある。

「改善フローのスキルを改善して」と依頼されたときの扱い:

1. 対象範囲は **直近の複数ラウンド**（単一セッションでなく、過去 3 〜 5 回の運用を俯瞰）
2. 観察項目は **スキル構造**（どの Step で迷ったか、どの Quality Gate を守れたか、軽量/通常モードの比率）
3. 改善ターゲットは `~/.claude/skills/engineering-process-improvement/SKILL.md` 本体
4. ユーザーレベルスキルは git 管理外のため、**PR は発生しない**（直接編集 + チャット報告で完了）
5. **engineer-skill-creator との住み分け**: 新スキル作成は engineer-skill-creator、既存スキル改善は本スキルの meta-retrospective モード

過剰適用を避けるため、meta-retrospective は **通常の振り返りを 3 回以上実施した後** に限定します。
サンプル不足の状態で構造を変えると、振り回されるだけになるからです。

## Anti-patterns

運用していて繰り返し踏みかけるアンチパターン。

- **感想文化する** — 「〜が大変でした」で終わる。必ず次回の具体的な手を書く
- **ドキュメントだけで済ます** — 忘れる前提で自動ガード化を優先する
- **全項目を一括 PR にする** — レビュー困難。独立項目は別 PR か、最低でもコミット分離
- **WIP ファイルを巻き込んで force-push** — 振り返りで指摘した失敗を実装中に繰り返す最悪パターン
- **ユーザー合意なしで実装開始** — 優先順位がズレると手戻りになる

## 導入前後の行動変容

実際に運用して、自分の行動が変わった点を正直に並べてみます。

### Before

- セッション後に「今回はこれが大変だった」で終わる
- 学びは頭の中か、バラバラな場所のメモに残る
- 同じ失敗を次のセッションで踏む
- `git add -A` でコミットしがちで、たまに余計な変更が混ざる
- 改善したい項目はあるのに、PR にまで落とす気力が続かない

### After

- セッション終了時にスキルを呼び出す運用が定着
- 振り返り表で影響の大きさまで書かれる
- 改善 PR が前回境界からチェーンで辿れる
- Quality Gate に「`add -A` 禁止」が入っていて、機械的に止まる
- 改善項目が B（自動ガード）に落ちるので、次のセッションで注意力を使わずに済む

### 特に効いたポイント

**1. 影響を数字で書く癖**

「大変だった」が「手戻り30分、force-push 1 回、stash ファイル 2 個混入」のように書かれるようになった。
これが書けると、CI ガードの設計ができる。感情ではなく構造で失敗を記述できるようになる。

**2. 自動ガード優先の方針**

ドキュメントに書くだけで済ませる誘惑は強い。
「書いた、これで改善した」と言いたくなる。
でも、このスキルの Quality Gate に「最低 1 件は B に落とす」が入っているので、強制的に CI / pre-commit / lint のどれかを設計することになる。結果として再発防止が強くなった。

**3. 前回境界からのチェーン**

PR 本文に「前回の振り返り PR: #XX」を書くことで、改善の積み重ねが履歴として追跡できる。
「なぜこの CI チェックが入ったのか」を後から辿れるので、運用が属人化しにくい。
これは小さな設計だけど、半年後に効いてくる。

**4. 軽量モードという逃げ道**

改善 PR を無理に量産しないで済む。
「今回は健全だった」と結論して終われるのが、振り返りサイクル自体の健全性につながる。
意味の薄い PR でチームのレビューコストを食わない。

## 運用してみた気づき

スキル化して 10 回ほど回してから気づいたことをいくつか挙げます。

### 振り返りが「怖くない」感覚になった

失敗を書き出すのは、本来あまり気が進まない作業です。
でも「書いた失敗が仕組みに変換される」と分かっていると、書くことへの抵抗が減る。
「書いた失敗は消える（自動ガード化されて再発しない）」という実感が、振り返りを前向きな作業に変えた気がします。

### Carry-over observations の価値

軽量モードで残す carry-over が、次のセッションの振り返りで驚くほど回収される。
自分が目の前の作業に集中している間も、並行で別の情報が積まれている。それを1セッション持ち越すだけで、統合的な振り返りができる。

### Meta-retrospective の発動タイミング

「スキル自身を改善」は、誘惑が強いが早すぎると壊す。
3 回以上運用してから、というルールは自分の運用では正解だった。
2 回目で「このスキル改善したい」と思ったときは、だいたい実際にはスキル自体ではなく自分の使い方が悪いだけだった。

## まとめ

`engineering-process-improvement` は、開発セッション直後の振り返りを、仕組みに変換する運用スキルです。

- 8 ステップの通常モード、軽量モード、meta-retrospective の 3 モード
- 改善は A（ドキュメント）/ B（自動ガード）/ C（ツール運用）の 3 分類
- 最低 1 件は B（自動ガード）に落とす
- 前回境界からチェーンを辿れる PR 本文
- ユーザーレベル user-level スキル、git 管理外で維持

運用し始めてから、振り返りが感想で終わることがなくなり、同じ失敗を踏む頻度が目に見えて減りました。
改善サイクルが、注意力ではなく設計で回るようになっている感覚があります。

気合いではなく、仕組みで事故率を下げる。
これを実現するための土台として、かなり手応えのあるスキルになっています。
