## 想定読者

- Claude Code / Codex などの自走型AIエージェントを、ある程度任せて使っている人
- 「進めて」「いい感じにやって」と言ったとき、AIがどこまでやるか／どこで止まるべきかが気になる人
- 自走の便利さと、スコープクリープ・暴走のリスクの両方を実感している人

## この記事について

これは設計論ではなく、**実際に起きたことの記録**です。あるブログ運用セッションで、私はAIエージェントに「進めて」「オススメの対応を進めて」と何度も言い続けました。明示タスクが尽きても言い続けたとき、エージェントは何を「次にやるべきこと」として選び、どこで止まったのか。そのログを残します。

## 「進めて」は何を起動するのか

<!-- 明示タスク（Qiita公開）→ 完了後の「進めて」で何が起きたか:
     エージェントは「次の推奨アクション」を自分で探し始める。
     最初は安全な後片付け（stale ブランチ掃除、PR整備）、
     次に前向きな価値（次記事の検討・制作）へと範囲が広がる。
     「進めて」= タスク指定ではなく「自走の合図」として解釈される -->

## 自走が価値を生んだ領域

<!-- 実際に価値が出た自走の例:
     - 1つのPRに混ざった2系統の変更を分離（記事 vs ツールスクリプト）
     - stale ブランチ30+本・古いstashの掃除（squashマージ済みの判定）
     - 公開後の事後検証（drift / live反映 / queue整合）
     - セッションの学びを AGENT_LEARNINGS に記録
     これらは「言われなくても妥当」な範囲。判断基準: 元の依頼から自然に follow する／可逆 -->

## 安全機構が止めた境界

<!-- ここが記事の核。実際にハーネスの安全機構が止めた操作:
     1. cross-repo push（my-blog のツリーを別リポジトリへ一括転送）→ データ持ち出しと判定
     2. settings.local.json への push 許可の自己追加 → self-modification と判定
     3. 認証トークン(JWT)の解析でアカウント情報抽出 → 認証情報探索と判定
     4. 未マージの固有コミットを持つブランチの強制削除＋stash drop → 不可逆な作業損失と判定
     学び: エージェントの「やってよさそう」と、人に確認すべき境界はズレる。
     仕組みが先に止めることで、Organizer 的判断を二重化できた -->

## 自走とスコープクリープの境目

<!-- 「進めて」を繰り返すと、有用な自走と、ただの仕事の捏造の境界が問われる。
     - 完了後に「残作業はありません」と正直に言い、捏造せず確認に切り替えた場面
     - 一方で、次記事の検討・制作は「前向きな価値」として妥当だった
     境目の判断軸: それは元依頼から follow するか／可逆か／外向き・破壊的でないか／
     ユーザーが決めるべき選択か。迷ったら捏造せず提示して止まる -->

## 「進めて」を安全に使うための運用

<!-- 読者が持ち帰れる運用:
     - 自走してよい範囲（可逆・内部・follow する作業）と、確認が要る範囲（外向き・破壊的・不可逆）を先に決める
     - 安全機構（push/削除/権限変更のブロック）を保険として効かせる
     - エージェント側は「推奨」を提示してから止まる規律。捏造より確認
     - これらを memory/規約に永続化すると次セッションでも再現する（→ 関連記事） -->

## まとめ

<!-- 「進めて」は強力だが、何でもやらせる呪文ではない。
     自走が効くのは可逆・内部・follow する範囲。外向き・破壊的・不可逆は人の判断。
     その境界を、ルール（規約）と仕組み（安全機構）の二層で守ると、
     任せられる範囲が広がる。実運用ログとしての結論。 -->
