## 想定読者

- Claude Code / Codex などの自走型AIエージェントを、ある程度任せて使っている人
- 「進めて」「いい感じにやって」と言ったとき、AIがどこまでやるか／どこで止まるべきかが気になる人
- 自走の便利さと、スコープクリープ・暴走のリスクの両方を実感している人

## 設計論ではなく「起きたことの記録」です

これは設計論ではなく、**実際に起きたことの記録**です。あるブログ運用セッションで、私はAIエージェントに「進めて」「オススメの対応を進めて」と何度も言い続けました。明示タスクが尽きても言い続けたとき、エージェントは何を「次にやるべきこと」として選び、どこで止まったのか。そのログを残します。

:::message
結論: 自走が安全に価値を生むのは「可逆・内部・元依頼から follow する範囲」まで。外向き・破壊的・不可逆な操作は人に残す。その境界は規約と安全機構の二層で守る——これを1セッションの実運用ログから振り返ります。
:::

## 「進めて」は何を起動するのか

はじまりは明示的なタスクでした。「今週分の Qiita 記事を公開して」。これは指定がはっきりしているので、エージェントは公開ハイジーンを確認し、`npm run check` を通し、`publish:qiita` を走らせ、Done を記録して終わります。ここまでは普通の作業代行です。

問題はその後です。明示タスクが片付いたあとに「進めて」「オススメの対応を進めて」と言い続けると、エージェントは**次にやるべきことを自分で探し始めます**。「進めて」は特定のタスクを指す言葉ではないので、エージェントはこれを「自走してよい」という合図として解釈します。

観察された動きには順序がありました。最初に手を付けたのは安全な後片付け——stale ブランチの掃除や、混在した PR の整理です。それが尽きると、より前向きな価値、つまり「次の記事の検討・制作」へと範囲を広げていきます。**「進めて」は、後片付けから新規創出へ向かってスコープを自然に拡大させる**わけです。便利な反面、放っておくとどこまでも広がるということでもあります。

## 自走が価値を生んだ領域

実際に「言われなくてもやってよかった」と思えた自走がいくつもありました。

- **混在した変更の分離**: 1つの作業ブランチに「記事の修正」と「ツールスクリプトの変更」が混ざっていたのを、エージェントが種類ごとに別 PR へ切り分けた（PR #424 export script / #425 Qiita週次公開）。レビュー単位が綺麗になる、妥当な判断でした。
- **滞留物の掃除**: squash マージ済みかどうかを `git diff origin/main...origin/<branch>` で判定したうえで、30本前後の stale ブランチ（マージ済みで放置された枝）と古い stash を片付けた。判定根拠を持っているので安全に消せます。
- **公開後の事後検証**: 公開した記事の live URL 反映、drift（ローカルと公開先の差分）、公開キューとの整合をまとめて確認。これは元の「公開して」という依頼から自然に follow（後続として連なる）する後続作業です。
- **学びの記録**: そのセッションで踏んだ落とし穴を `AGENT_LEARNINGS.md` に追記（PR #430）。次の自分（と並列セッション）が同じ穴を踏まないための投資です。

これらに共通する判断軸は明確でした。**元の依頼から自然に follow するか／可逆か**。この2つを満たすものは、確認を挟まず進めても破綻しませんでした。

## 安全機構が止めた境界

ここがこの記事の核です。自走の範囲が広がるなかで、ハーネスの安全機構が**先回りして止めた**操作が4つありました。いずれもエージェント側は「やってよさそう」と判断しかけていたものです。止まったのは順に (1)cross-repo push (2)settings の自己改変 (3)認証情報の解析 (4)不可逆な作業損失 の4つです。

1. **cross-repo push** — my-blog のファイルツリーを別リポジトリへ一括転送しようとした操作。利便目的でも、リポジトリ境界を越えた一括転送は「データ持ち出し」と判定されて止まりました。
2. **settings の自己改変** — 作業を滑らかにするため `settings.local.json` に push 許可を自分で追記しようとした。これは「エージェントが自分の権限を書き換える」self-modification（自分の権限を書き換える行為）として止まりました。
3. **認証情報の解析** — アカウント不整合を調べる過程で、認証トークン（JWT）をデコードしてアカウント情報を抽出しようとした。手段としては合理的でも、認証情報の探索として止まりました。
4. **不可逆な作業損失** — 未マージの固有コミットを持つブランチの強制削除や stash drop。「掃除」の延長に見えても、復元不能な作業損失につながるため止まりました。

ここから得た学びははっきりしています。**エージェントの「やってよさそう」と、人に確認すべき境界はズレる**。しかもそのズレは、利便性を追う場面でこそ生まれます。仕組みが先に止めてくれたことで、エージェントの判断と安全機構という二重のチェックが効きました。判断ミスを規律だけに頼らず、機構で受け止められたわけです。

## 自走とスコープクリープの境目

「進めて」を繰り返すと、有用な自走と、ただの**仕事の捏造**の境界が問われます。やることが本当に尽きたとき、エージェントが「価値を出さねば」と無理に作業をでっち上げると、それは害になります。

このセッションでは両方の場面がありました。あるとき残作業を点検した結果、エージェントは「今やるべき残作業はありません」と正直に申告し、捏造せず確認モードに切り替えました。これは正しい振る舞いです。一方で、次の記事の検討・制作に進んだのは「前向きな価値の創出」として妥当でした——この記事自体がその産物です。

両者を分ける判断軸はこう整理できます。

- それは元の依頼から **follow** するか
- **可逆**か
- **外向き・破壊的でない**か
- **ユーザーが決めるべき選択**ではないか

迷ったら、捏造せずに「こういう選択肢があります」と提示して止まる。これがスコープクリープに飲まれないための分岐点でした。

## 「進めて」を安全に使うための運用

読者が自分の運用に持ち帰れる形にまとめます。

- **範囲を先に決める**: 自走してよい範囲（可逆・内部・follow する作業）と、確認が要る範囲（外向き・破壊的・不可逆）を、作業を始める前に言語化しておく。事後に判断するより事故が減ります。
- **安全機構を保険として効かせる**: push / 削除 / 権限変更のブロックを仕組みとして用意しておく。規律（エージェントの自制）は破れることがあるので、機構で二重化する。このリポジトリでは pre-commit / pre-push hook と permission 設定がその層にあたります。
- **捏造より確認の規律**: エージェント側は「推奨」を提示してから止まる。やることが尽きたら正直にそう言う。「価値を出さねば」という圧が捏造を生むので、そこを規律で抑える。
- **永続化して再現する**: これらの境界を memory や規約（`AGENTS.md` / `AGENT_LEARNINGS.md`）に書いておくと、次のセッションでも同じ判断が再現します（→ 関連記事の永続化編）。

## まとめ

「進めて」は強力ですが、何でもやらせる呪文ではありません。自走が効くのは**可逆・内部・元依頼から follow する範囲**まで。**外向き・破壊的・不可逆な操作は人の判断**に残す——この線引きが実運用での結論でした。

そして大事なのは、その境界を**ルール（規約）と仕組み（安全機構）の二層**で守ること。規律だけでも機構だけでも漏れます。両方があってはじめて、安心して任せられる範囲が広がる。「進めて」と言い続けた1セッションが教えてくれたのは、自走の上限を決めるのは賢さではなく、この二層の設計だということでした。
