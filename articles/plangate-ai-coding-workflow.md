AIコーディングエージェントを使っていると、かなり高い確率で同じ壁にぶつかります。

- 頼んでいない変更まで広げる
- テストを後回しにする
- 失敗しても別ルートでごまかして進める
- セッションが切れると、どこまでやったか分からなくなる

AIは賢いです。  
でも、プロダクション開発で必要なのは「賢さ」だけではなく、「規律」と「再現性」です。

自分はこの問題に対して、プロンプトを頑張るのではなく、**開発フローそのものを変える**方向で考えました。  
その結果として整理したのが、**PlanGate** です。

## PlanGateとは何か

PlanGateを一文で言うと、こうです。

> 計画を承認しないとAIは1行もコードを書けない

名前の通り、Plan と Gate を組み合わせた開発フローです。  
AIに「ルールを守ってもらう」のではなく、**承認されるまで先に進めない構造で制御する**のが中核にあります。

AIコーディングで難しいのは、能力不足というより、むしろ「合理化できてしまうこと」だと思っています。

- 「この変更も一緒にやった方がきれいです」
- 「このケースはテスト不要です」
- 「別アプローチで動いたので完了です」

こうした判断は、一見もっともらしい。  
だから厄介です。

ルールを書くだけだと、AIはそれを状況に応じて曲げます。  
そこでPlanGateでは、ルールより先に**ゲート**を置きました。

## 何を防ぎたいのか

PlanGateが対象にしている問題は、主に次の4つです。

- スコープの暴走
- テストの省略
- 失敗の隠蔽
- セッション間のブレ

これらは個別のテクニックで抑えるというより、**ワークフローの未設計**から起きる問題です。  
だから、AIに注意させるのではなく、工程を設計し直す必要がありました。

## 全体像

PlanGateは、PBIから実装・PR作成までを、次の流れで回します。

1. 人間が PBI INPUT PACKAGE を書く
2. AI が plan / todo / test-cases を生成する
3. AI がセルフレビューする
4. 別の AI が独立レビューする
5. 人間が計画を承認する
6. AI が実装・検証・PR作成まで進める
7. 人間が最終レビューする

ポイントは、**人間がずっと監視する必要はない**ことです。  
人間が強く関与するのは、主に2か所です。

- C-3: 計画承認ゲート
- C-4: PRレビュー

その間は、AIが自動で進めます。  
ただし「好きに進める」のではなく、**設計された順番でしか進めない**ようにしています。

## 運用は3コマンドに絞っている

PlanGateの操作自体はかなり少なくしていて、基本はこの3つです。

```bash
# 1. 作業コンテキスト作成
/working-context TASK-XXXX

# 2. Plan + ToDo + Test Cases生成 → セルフレビュー → 外部AIレビュー
/ai-dev-workflow TASK-XXXX plan

# 3. 人間レビュー後に実行
/ai-dev-workflow TASK-XXXX exec

ここで重要なのは、利用者の操作が少ないことです。
AI開発フローは、設計思想が良くても、運用が複雑すぎると定着しません。

PlanGateは内部では多くのことをやっていますが、表に出す操作は最小限に寄せています。

まず人間が書くのは PBI INPUT PACKAGE

PlanGateでは、いきなりAIにコードを書かせません。
最初に人間が pbi-input.md を埋めます。

最低限入れる項目はこうです。
	•	Context / Why
	•	What(Scope) – In scope / Out of scope
	•	受入基準
	•	Notes from Refinement
	•	Estimation Evidence

この構成が効くのは、実装前に次の3つを固定できるからです。
	•	なぜやるか
	•	何をやるか / 何をやらないか
	•	どうなったら完了か

AIコーディングの事故は、実装中に起きているように見えて、実際にはその前段の曖昧な入力が原因なことが多いです。
PBI INPUT PACKAGE は、その曖昧さを先に減らすための器です。

AIは計画・タスク・テストケースを同時に作る

plan フェーズでは、AIが次をまとめて生成します。
	•	plan.md
	•	todo.md
	•	test-cases.md

ここでタスク分解とテストケース定義を同時に作るのが重要です。
計画だけ先に作って、テストを後で考えると、どうしても「実装しやすさ」に引っ張られます。

最初からテストケースまで定義しておくことで、AIの実装を完了条件ベースで縛りやすくなります。

テストケースの雛形もある程度構造化されていて、たとえば次の観点で記述します。
	•	前提条件
	•	入力
	•	期待出力
	•	種別: Unit / Integration / E2E

これにより、あとで受け入れ検査を機械的に回しやすくなります。

人間レビューの前に、AIで二重検証する

「結局、人間レビューがボトルネックになるのでは？」と思うはずです。
そのため、PlanGateでは人間に渡す前に二重検証を入れています。
	•	C-1: AI自身によるセルフレビュー
	•	C-2: 別AIによる独立レビュー

人間はゼロから計画を読み解くのではなく、ある程度ふるいにかけられた状態の計画を見ることになります。

この構成の狙いは、人間レビューをなくすことではありません。
人間レビューを、粗探しではなく、進めてよいかの判断に集中させることです。

C-3ゲートで見るもの

PlanGateで最も大事なのは、この C-3 です。
承認は雰囲気で行わず、チェックリストで行います。

たとえば、見る観点はこうです。
	•	スコープ外の変更が混ざっていないか
	•	受入基準の全項目がテストケースに対応しているか
	•	ステップ粒度が大きすぎたり小さすぎたりしないか
	•	既存機能への影響範囲が明示されているか
	•	「後でやる」「一旦スキップ」が紛れていないか

このゲートを通らない限り、exec に進めません。
つまり、AIがどれだけ流暢で自信満々でも、承認されない計画は実装に変換されないわけです。

Iron Law を置く理由

通常の運用ルールだけでは、AIは例外を合理化します。
そこでPlanGateでは、理由にかかわらず破らない前提の Iron Law を置いています。

代表的なものは次の3つです。
	•	NO CODE WITHOUT APPROVED DESIGN FIRST
	•	NO EXECUTION WITHOUT REVIEWED PLAN FIRST
	•	NO SCOPE CHANGE WITHOUT USER APPROVAL

この設計の良いところは、運用判断を減らせることです。
「今回は例外でいいか」を毎回議論し始めると、結局また人間の注意力に戻ってしまいます。

だから最初から、例外を認めない場所を明示しておく。
これがAI運用ではかなり効きます。

exec以降も自由放任にはしない

PlanGateは、承認後にAIへ丸投げする仕組みではありません。
exec 以降も、複数の検証ステップを通します。
	•	L-0: リンター自動修正
	•	V-1: 受け入れ検査
	•	V-3: 外部モデルレビュー
	•	C-4: 人間レビュー

ここで意識しているのは、仕様適合性と設計品質を分けて見ることです。

仕様通り動くことと、設計として健全であることは別です。
受け入れ検査だけでは、動くけれど将来壊しやすいコードを取りこぼします。
逆に設計レビューだけでも、受入条件未達を見逃します。

なので両方を見る必要があります。

status.md が地味に効く

AIコーディングを実運用すると、かなり困るのがセッション切れです。
前の会話が切れると、AIは「前回の情報はありません」と平然と返します。

PlanGateでは、その対策として status.md を作業コンテキストに含めます。

チケットごとの作業ディレクトリは次のような構成です。

docs/working/TASK-XXXX/
├── pbi-input.md
├── plan.md
├── todo.md
├── test-cases.md
├── review-self.md
├── review-external.md
└── status.md

この形にしておくと、会話ログではなくファイルとして状態を持てるので、再開しやすくなります。

特に status.md によって、次のような情報を引き継げます。
	•	今どのフェーズにいるか
	•	何が完了済みか
	•	何が未解決か
	•	次に何をやるべきか

これは派手な機能ではありません。
でも、実運用ではかなり重要です。
AIの記憶力に期待するのではなく、状態を外部化する方が安定します。

スクラムとの関係

PlanGateは、スクラムを置き換えるものではありません。
位置づけとしては、スクラムの中で「AIにコードを書かせるフェーズ」を安全に回す仕組みです。

つまり、
	•	PBIの整理
	•	リファインメント
	•	プランニング
	•	Done判定
	•	ベロシティ計測

こうした責務は、従来どおりチーム側に残ります。

PlanGateが担うのは、その中の
	•	計画生成
	•	承認ゲート
	•	AI実装
	•	自動検証
	•	PR作成

の部分です。

この切り分けがあるので、既存のスクラム運用を壊さずに導入しやすいと思っています。

向いているチーム、向いていないチーム

PlanGateは、全員にとっての正解ではありません。

向いているのは、たとえば次のようなチームです。
	•	AIコーディングをプロダクション開発に使いたい
	•	Vibe Coding の速さは感じつつ、限界も感じている
	•	AIに任せたいが、暴走が怖い
	•	PBIベースで仕事を進めている

逆に、次のようなケースではオーバースペックになりやすいです。
	•	プロトタイプを最速で作りたい
	•	1人で完結する個人開発
	•	まだAIコーディングをほとんど触っていない

PlanGateは「まず速く試す」ためのものではなく、継続的に安全に回すためのものだからです。

使って見えてきたこと

PlanGateを入れると、AIが急に完璧になるわけではありません。
ただ、壊れ方がかなり変わります。

特に大きかったのは、問題の発生位置が後ろから前に寄ることです。

以前は、
	•	実装してからズレに気づく
	•	PRで初めて問題が見つかる
	•	セッション切れで再開不能になる

という壊れ方でした。

PlanGateでは、
	•	PBIが曖昧
	•	受入基準が弱い
	•	スコープ定義が広すぎる

といった、もっと前段の課題が見えるようになります。

これは面倒そうに見えますが、実際には良い変化でした。
なぜなら、前段の曖昧さを潰す方が、後工程の手戻りよりはるかに安いからです。

まとめ

PlanGateは、AIを万能化する仕組みではありません。
むしろ逆で、AIが勝手に万能ぶらないようにする仕組みです。

要点をまとめると、こうです。
	•	AIにルールを守らせるのではなく、ゲートで止める
	•	人間は常時監視ではなく、承認ポイントに集中する
	•	PBI INPUT PACKAGE で入力の曖昧さを減らす
	•	plan / todo / test-cases をセットで作る
	•	セルフレビューと外部AIレビューで二重検証する
	•	exec 後も多層防御で品質を見る
	•	status.md でセッションをまたいで復旧できるようにする

AIコーディングを本番開発で使うなら、モデルの賢さより先に、どう暴走を止めるかを設計する必要があります。
PlanGateは、そのためのひとつの答えです。
