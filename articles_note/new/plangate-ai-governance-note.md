# AIにコードを書かせる前に、人間が承認する場所を作る

AIコーディングエージェントを使うようになってから、開発の速度は確かに変わりました。

以前なら、実装方針を考え、関連ファイルを読み、細かい修正を積み上げていた作業を、AIがかなりの速度で進めてくれます。

これは大きな変化です。

ただ、最近は「AIがどれくらいコードを書けるか」よりも、「AIにどこまで書かせてよいか」のほうが気になっています。

コードが速く書けるようになると、失敗も速くなります。

スコープが少し広がる。テスト観点が曖昧なまま実装される。レビューで初めて、今回やりたかったことと少し違うと気づく。そういう小さなズレが、AIを使うほど目立つようになりました。

そのために作っているのが PlanGate です。

PlanGateの考え方は、かなり単純です。

> 承認なし、コードなし。

AIがコードを書く前に、まず計画を作る。人間がそれを読む。承認してから実装に進む。

たったそれだけです。

でも、この「たったそれだけ」が、AIをチーム開発に入れるときにはかなり効きます。

最初に言ってしまうと、PlanGateはAIを疑うための仕組みではありません。

AIをチームで使える形にするために、**人間が判断する場所を先に決める仕組み**です。

## この記事で書きたいこと

この記事では、PlanGateの細かいコマンドやHookの仕様よりも、その背景にある考え方を書きます。

- なぜAIにいきなりコードを書かせないのか
- なぜ「承認」という少し重い言葉を使うのか
- なぜプロンプトだけではなく、HookやCLIで検査したくなったのか
- なぜOSSとして公開しているのか

技術的な説明はZenn向けの記事に寄せるとして、ここではもう少し個人の実感に近い話をします。

## AIは親切にスコープを広げる

AIに実装を頼むと、こちらが頼んだこと以上に気を利かせてくれることがあります。

たとえば、小さな画面修正をお願いしたつもりなのに、周辺コンポーネントの命名も整えてくれる。テストを追加するだけでよかったのに、テスト基盤も少し直してくれる。関連する画面にも同じ改善を広げてくれる。

それ自体が悪いとは限りません。

むしろ、単体で見れば良い変更のこともあります。

でも、チーム開発では「良い変更かどうか」と「今回入れるべきかどうか」は別です。

レビューする人は、その変更が今回の目的に必要なのか、別PRに分けるべきなのか、後方互換に影響しないのかを判断しなければなりません。

AIが速く実装してくれるほど、人間が判断する量が増えてしまうことがあります。

だから、実装前に境界を置きたいと思いました。

## 計画を先に見る

PlanGateでは、AIにまず計画を作らせます。

そこには、何をやるかだけでなく、何をやらないかも書かせます。

自分が特に大事にしているのは、次のような項目です。

- 今回の目的
- 変更する範囲
- 変更しない範囲
- テスト観点
- 既知のリスク
- 完了条件

この中でも、「変更しない範囲」が一番効きます。

人間同士でも、ついでの改善はよく起きます。AIの場合、その速度と量が増えます。

だからこそ、先に「今回はここまで」と決める必要があります。

これはAIを不自由にするためではありません。

人間がレビューできる形に保つためです。

## 実際に何が変わるのか

PlanGateを使うと、AIとの会話のタイミングが変わります。

以前は、実装後にこういう確認をしていました。

- この変更は今回の目的に必要だったのか
- このファイルまで触る必要があったのか
- テスト観点は本当に満たしているのか
- レビューでどこを重点的に見ればよいのか

PlanGateでは、同じ問いを実装前に移します。

- この計画で進めてよいか
- 今回やらないことは明確か
- テスト観点は判断できる粒度か
- リスクは人間が見られる形になっているか

実装後に差分を戻すのは重いです。

でも、実装前に計画を直すのは軽いです。

この差が、AIをチーム開発に入れるときには大きいと感じています。

## 承認はブレーキではなく、合意点

「承認」という言葉を使うと、少し重く見えるかもしれません。

でも、PlanGateでの承認は、開発を遅くするためのブレーキではありません。

むしろ、先に合意しておくための場所です。

実装後に差分を見ながら、

「これは今回必要だったのか」

「このテスト観点は考えていたのか」

「このファイルまで触ってよかったのか」

と話すより、実装前に計画を見て、

「この範囲で進めよう」

「ここはやらないことにしよう」

「このリスクは人間がレビューしよう」

と決めておくほうが楽です。

AIが速くなったことで、実装前の会話の価値が上がったのだと思います。

## プロンプトだけでは足りない

最初は、プロンプトに「承認前に実装しないでください」と書けば十分だと思っていました。

でも、実際にはそれだけだと不安が残ります。

長いセッションでは、指示が薄くなることがあります。人間側も、急いでいると「まあこのまま実装して」と進めてしまうことがあります。

そのため、PlanGateではHookやCLIで検査する方向に寄せています。

たとえば、計画がない状態でコードを触ろうとしていないか。承認後に計画が変わっていないか。テスト観点なしで検証に進んでいないか。スコープ外のファイルを編集していないか。

こうしたことを、プロンプトのお願いではなく、実行時に見られるようにします。

もちろん、最初から全部を強制する必要はありません。

最初はwarningで十分です。

大事なのは、AIの作業がどこで境界を越えそうになるのかを観測できることです。

## AIを信じるためではなく、人間が判断するため

PlanGateは、AIを信用するための仕組みではありません。

どちらかというと、人間が判断しやすくするための仕組みです。

AIが作った計画が正しいとは限りません。

AIが作ったテスト観点も抜けることがあります。

だからこそ、実装前に人間が読みます。

ここで大切なのは、人間がすべてを細かく指示することではありません。

人間は、目的、境界、リスク、完了条件を見る。AIは、その範囲の中で具体的な実装を進める。

この分担にすると、AIの速度を活かしながら、チームとしての判断も残せます。

## OSSとして公開する理由

PlanGateは、自分の開発フローの中から生まれたものです。

最初は、自分がAIエージェントを使うための作業ルールでした。

でも、使っているうちに、これは個人の工夫というより、これから多くのチームが直面する問題に近いのではないかと思うようになりました。

AIはもっと速く、もっと自然にコードを書くようになります。

そうなったとき、必要になるのは「AIに全部任せる勇気」ではなく、「どこで人間が判断するかを設計する力」だと思います。

PlanGateは、そのための小さな提案です。

計画を作る。承認する。実装する。検証する。レビューする。

昔からある当たり前の工程を、AIエージェントの速度に耐えられる形で組み直したい。

そのためにOSSとして公開しています。

## まずは一つのタスクからでいい

こういう仕組みは、きれいに作ろうとすると重くなります。

最初から全社標準にする必要はありません。

まずは一つのタスクで、AIにこう頼むだけでも十分です。

```markdown
まず実装計画を作ってください。
まだコードは変更しないでください。
計画には、今回やること、やらないこと、テスト観点、リスクを書いてください。
```

そして、その計画を人間が読む。

納得できたら、実装に進める。

それだけでも、AIとの開発はかなり扱いやすくなります。

ツールとしてのPlanGateを使う前に、この考え方だけ試してもよいと思います。

## 興味があれば

PlanGateはOSSとして公開しています。

まずはリポジトリを眺めて、READMEの `plan → approve → exec` の流れだけ見てもらえれば十分です。

実際に使わなくても、自分の開発フローに「AIがコードを書く前に人間が見る場所」を一つ置くだけで、似た効果は試せます。

もし考え方に近いものを感じたら、GitHubでスターしておいてもらえると、あとで試しやすくなります。

そして実際に使ってみて、違和感や困った点があればIssueで教えてもらえると嬉しいです。

PlanGateは、AIに全部任せるためのものではなく、AIと人間の判断をどう並べるかを考えるための道具です。

## おわりに

AIコーディングエージェントの進化は、とても面白いです。

ただ、コードを書く能力が上がるほど、チーム側の設計も必要になります。

どこまで任せるのか。

どこで止めるのか。

何を証拠として残すのか。

誰が承認するのか。

PlanGateは、その問いに対する自分なりの答えです。

AIにコードを書かせる前に、人間が承認する場所を作る。

それは、AIの力を弱めるためではなく、チームで使える形にするための工夫だと思っています。
