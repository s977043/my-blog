## 1. はじめに

人が仲介して複数のAIエージェント（**Claude Code／Gemini Code Assist**）を往復させる運用は、**観点が散り、前提がぶれやすい**。本稿は、**二層ドキュメント**（決定＝`仕様.md`／経緯＝`対話ログ.md`）と**役割分担（主＝Claude／副＝Gemini）で、合意形成のスピードと再現性**を高める実務ガイドです。
評価はPBIの作成数（量）ではなく、**変更提案ドラフトまでのリードタイム**に一本化します。

### 1.1 記事の範囲（PBI本文は別記事）

- 本稿：ワークフロー／運用ルール／入力設計／品質担保。
- 別稿：PBI本文の型、受け入れ条件の書き方。

### 1.2 想定読者

- PdM／Webディレクター／Webサイト編集者／Webエンジニア。

### 1.3 得られるもの（ゴール）

- 二層ドキュメント運用と役割分担の型。
- アクセスデータを使った**対象選定**。
- **ドラフト到達**までのKPI・チェックリスト。

## 2. 背景と課題

- 人が仲介してエージェントを往復させると、**観点の散逸・前提ズレ・手戻り**が発生。
- レビュー観点が毎回変わり、**学びの再利用**が効きづらい。
- KPIが散らばると、活動が\*“**量産**”に流れがち\*で成果が見えにくい。

---

## 3. 解決アプローチ：二層ドキュメント

- **決定の器＝`仕様.md`**：単一の真実源（Single Source of Truth）。**合意内容のみ**を集約。
- **経緯の器＝`対話ログ.md`**：やり取り・**前提／矛盾／未決事項**を記録・可視化。
- **対象選定**はアクセスデータ（PV／sessions 等）で**高インパクト領域**に集中。
- **コード参照**（GitHub等）は**必要時のみ**。該当ロジックや依存関係を特定して確認。

---

## 4. 役割分担とプロンプト戦略

### 4.1 主＝Claude Code の責務

- **統合**：参考URL／アクセス表／指摘を受けて**仕様ドラフト**を作成・更新。
- **構造化**：決定／未決／前提／依存／リスクを**分離記述**。
- **反映**：合意候補は\*`**仕様.md**`のみ更新\*（`対話ログ.md`は経緯の記録に限定）。

### 4.2 副＝Gemini Code Assist の責務

- **観点提示**：抜け・矛盾・前提ズレの**検出と反証**。
- **測定観点の補完**：Core Web Vitals（**LCP／CLS／INP／TTFB**）の網羅性を確認。
- **根拠の明示**：アクセス表の**根拠行**を指摘。必要時のみ\*コード参照（GitHub等）\*を指示。

### 4.3 ハンドオフ（合意と反映）

> **主がドラフト → 副が観点提示 → `仕様.md`へ反映 → 担当者が確認 → PdMが最終判断**

- 反映対象は\*合意**候補**\*のみ。**未決／矛盾**は `対話ログ.md` に残す。
- **担当者**：スコープ逸脱・実装可否・依存関係を確認（必要に応じ追記）。
- **PdM**：最終判断。差し戻し時は `仕様.md` を更新し、理由を `対話ログ.md` にタグ付き（`[unresolved]` 等）で記録。
- **KPI**：**差し戻しはKPI（リードタイム）から除外**。

### 4.4 ミニ・プロンプトセット（主／副）

そのままコピペで利用可。出力は日本語想定。

```
［主（Claude）］
目的：速度改善（対象URL：/a、/b）。仕様.mdを単一の真実源として更新。
	1.	仕様ドラフト：目的／対象／計測条件／受け入れ条件（初版・たたき）
	2.	未決・前提・依存・リスクを分離し、タグ付け（[unresolved]／[assumption]／[risk]）
	3.	次に確認すべき質問を3点（各50字以内）

［副（Gemini）］
仕様ドラフトをレビューし、抜け・矛盾・前提ズレを指摘。
	1.	Core Web Vitals（LCP／CLS／INP／TTFB）の網羅性を確認
	2.	計測条件・対象範囲・依存の妥当性を検証
	3.	アクセス表の根拠行を明示し、代表性を評価
	4.	（必要時）コード参照箇所（GitHub等）を特定し、確認観点を指示

出力：仕様へ昇格候補／保留理由／リスクのメモ
```

## 5. 入力設計（速度改善ケース）

### 5.1 参考URL／アクセスデータ／対象ページの束ね方

- **参考URL**：社内ナレッジ・外部ベストプラクティスを**1〜3本**に限定（出典を短く併記）。
- **アクセス表（最低列）**

  - `url`：計測・検証の対象ページのURL（テンプレ／レイアウトの代表URL）。
  - `pv` **または** `sessions`：期間内のページビュー数 **または** セッション数（どちらか1列）。
  - `device\_ratio\_mobile`：モバイル閲覧の割合（**0〜1で記載**。例：0.72＝72%）。
  - `note`：選定理由／除外理由など（簡潔に1行）。
- **束ね方の基準**

  - **テンプレ／レイアウト単位**で束ねる（例：記事詳細、LP、料金ページ）。
  - ABテスト中・実験フラグ有りは `note` に明記し、対象外または別バケットへ。
- **選定の書き方（`仕様.md`）**

  - 例：「対象：/pricing、/feature-a」「根拠：アクセス表 row#1、row#2」「除外：/blog/\*（テンプレ異なる）」。

**最小サンプル（CSV）**

```
url,pv,device_ratio_mobile,note
/pricing,95000,0.68,“改修影響が広い（代表テンプレ）”
/feature-a,120000,0.72,“上位流入の代表URL”
/blog/post-xyz,30000,0.85,“対象外：テンプレが異なる”
```

### 5.2 計測方針とメトリクスの置き場所

- **主指標**：**LCP／CLS／INP／TTFB**（Core Web Vitals）。
- **計測条件の固定（例）**

  - デバイス：モバイル相当（Mid端末）、ネットワーク：4G、初回表示（first-view）。
  - キャッシュ無効、ログアウト、A/B実験・拡張は**OFF**。
  - 3回以上の計測 → **中央値**採用（外れ値は備考）。
- **ツール（自由）**：Lighthouse／PageSpeed Insights／WebPageTest 等。
- **置き場所**

  - `仕様.md` の **Metrics** セクション（指標・条件・閾値）。
  - 詳細ログ：`/measurements/YYYY-MM-DD/`（ファイル名規約は 5.3）。

### 5.3 データ保管と命名規約（ツール自由）

- **ディレクトリ例**

```
/spec/                ← 仕様.md（単一の真実源）
/logs/                ← 対話ログ.md（経緯・観点・未決）
/measurements/        ← 計測一式
└─ 2025-08-18/
├─ pricing_lighthouse.json
├─ feature-a_wpt.csv
└─ summary.md
```

- **ファイル名規約（例）**

  - `<page-key>\_<tool>.<ext>`（例：`pricing\_lighthouse.json`）。
  - まとめは `summary.md`（中央値・条件・備考のみ）。
- **参照ルール**

  - `仕様.md` から**相対パスでリンク**（エビデンスの追跡性を担保）。
  - 誤差や前提の変更は `対話ログ.md` に **`[assumption]`／`[assumption-conflict]`** で記録。

## 6. 運用フロー（改訂版）

### 6.1 ドラフト→観点提示→仕様反映→担当者確認→PdM最終判断

1. **主（Claude）がドラフト**を作成（仕様ドラフト＋未決／前提／依存／リスクの分離）。
2. **副（Gemini）が観点提示**（抜け・矛盾・前提ズレ・リスクの指摘）。
3. **`仕様.md`へ反映**：合意“候補”のみ反映。未決や矛盾は\***`対話ログ.md**`\*へ。
4. **担当者が確認**：スコープ逸脱・実装可否・依存関係を確認し、必要なら追記。
5. **PdMが最終判断**：承認または差し戻し。

- 差し戻し時：`仕様.md` を更新、理由を `対話ログ.md` にタグ付き（`[unresolved]` 等）で記録。
- **差し戻しはKPI（リードタイム）から除外**。

### 6.2 アクセスデータに基づく対象選定（仮→合意）

- **仮選定**：アクセス表（`url / pv または sessions / device\_ratio\_mobile / note`）から**高インパクト×代表テンプレ**を抽出（原則3件以内）。
- **レビュー**：副が**代表性と偏り**をチェック（根拠行を明示）。
- **合意**：`仕様.md` に**対象／根拠／除外**を1行ずつ記載、検討過程は `対話ログ.md` に残す。

### 6.3 仕様反映→チェック項目フィードバック

- 反映対象は**決定事項のみ**（`仕様.md`＝単一の真実源）。
- 観点は `対話ログ.md` で**タグ管理**：`[metric]`／`[assumption]`／`[assumption-conflict]`／`[risk]`／`[scope]`／`[unresolved]`。
- 次サイクル開始時、ログから**観点チェックリスト**を更新（再利用で漏れ防止）。

### 6.4 変更提案ドラフト生成（Draft PR またはレビュー用ドラフト）

- **到達条件（最小）**

  - 受け入れ条件の**数値・対象・計測条件**が明記。
- **クリティカル未決が実装ブロッカー**として残っていない。
- **`仕様.md` ↔ `対話ログ.md`** に相互リンク。

  - 影響範囲と**ロールバック方針メモ**（1〜3行）。
- **対象選定の根拠行**（アクセス表）を示す。
- **出力先**
- **Git利用**：**Draft PR（Pull Request）**。
- **非Git**：**レビュー用変更提案ドラフト**（Docs／チケット等）。
- **KPI**

  - 評価は\*変更提案ドラフトまでのリードタイム（中央値）\*のみ。
- **差し戻しはKPIから除外**（学びはログに残す）。

## 7. 品質担保

### 7.1 出力フォーマット固定化（ディレクトリ／命名）

- 配置ルール・命名・相対リンクは**5.3**の規約に準拠。
- 変更時は `対話ログ.md` に**理由付き**で残す（例：`[metric] LCP閾値 2.5s→2.7s 暫定`）。

### 7.2 “観点lint”によるチェック（最小セット）

- **タグ定義**：`[metric]`／`[assumption]`／`[assumption-conflict]`／`[scope]`／`[risk]`／`[unresolved]`。
- **使い方（`対話ログ.md`）**：

  - 例：`[scope] /pricing のみ対象。/blog はテンプレが異なるため除外`。
  - 例：`[metric] LCP はモバイル・4G・初回表示で計測、中央値採用`。
- **運用**：サイクル終了時にタグ付きログを**チェックリスト化**して更新。`仕様.md`には**決定事項のみ**反映。

### 7.3 完了の定義（DoR/DoD相当）

- **着手条件（DoR）**

  - 参考URLが**1〜3本**に絞られている。
- **アクセス表**（`url／pv または sessions／device\_ratio\_mobile／note`）が用意済み。

  - 目標・許容の**閾値**と**計測条件**が暫定合意。
- **完了条件（DoD）**
- \*`**仕様.md`**\*に対象／根拠／除外／受け入れ条件（数値・対象・計測条件）が明記。
- **未決の棚卸し**が完了（残る場合は`[unresolved]`で明示）。
- **計測エビデンス**が `/measurements/YYYY-MM-DD/` に格納され、仕様からリンク済み。
- **変更提案ドラフト**が作成済み（Git＝Draft PR／非Git＝レビュー用ドラフト）。

  - KPIに必要な**開始・終了時刻**が記録済み（リードタイム算出可）。

## 8. 実行例（抜粋）

### 8.1 入力例（アクセス表・抜粋）

プロンプトは**4.4**を参照。アクセス表。

```
url,pv,device_ratio_mobile,note
/pricing,95000,0.68,“改修影響が広い（代表テンプレ）”
/feature-a,120000,0.72,“上位流入の代表URL”
/blog/post-xyz,30000,0.85,“対象外：テンプレが異なる”
```

### 8.2 対話ログ要点（タグ付きサマリ）

- `[metric]` 計測：モバイル（Mid）／4G／初回表示／3回→中央値。
- `[assumption]` `/pricing` と `/feature-a` は同一テンプレートで主要導線を共有。
- `[assumption-conflict]` `/blog/\*` は別テンプレートのため除外（代表性が低い）。
- `[scope]` **静的要素の最適化**に限定（画像最適化・フォント・CSS分割）。
- `[risk]` 画像遅延読込の閾値変更で**LCP悪化**の可能性（ファーストビュー画像は遅延対象外）。
- `[unresolved]` キャンペーンバナーABの有無（本番で一時OFFにできるか要確認）。
- **昇格候補**：LCP目標2.5s（許容3.0s）／CLS 0.1以下／INP 200ms以下／TTFB 0.8s以下。

> ログ本文は `対話ログ.md` に保持。`仕様.md`には**決定事項のみ**反映。

### 8.3 仕様確定スナップショット（diff要約）

**Diff要約（3行）**

```
Before：対象＝全体／閾値未定
After ：対象＝/pricing・/feature-a／LCP＜2.5s（モバイル）
理由　：上位PVかつ代表テンプレで効果検証がしやすい
```

**`仕様.md` 抜粋（確定時点）**

```
目的
主要導線のCV影響が大きいページでCore Web Vitalsを改善し、離脱率の低減を狙う。

対象と根拠
	•	/pricing（アクセス表 row#1：PV 95k、mobile 0.68）
	•	/feature-a（アクセス表 row#2：PV 120k、mobile 0.72）
除外：/blog/*（テンプレ差異・導線非代表）

指標と閾値（目標／許容）
	•	LCP：2.5s／3.0s（モバイル・4G・初回表示・中央値）
	•	CLS：0.10／0.15
	•	INP：200ms／250ms
	•	TTFB：0.8s／1.0s

計測条件
モバイル相当（Mid端末）、4G、初回表示、ログアウト、AB/拡張OFF、3回測定→中央値。
```

## 9. 運用で得たこと

- **二層ドキュメント**で前提ズレと観点漏れが減る（決定＝`仕様.md`／経緯＝`対話ログ.md`）。
- **主＝Claude／副＝Gemini**で観点が定着・再利用される。
- **アクセス表**で高インパクトなURLに集中し、検証の歩留まりが改善。
- **観点lintタグ**により、未決・前提・リスクの棚卸しが楽になる。
- **追跡性**：アクセス表 → 仕様 → 変更提案ドラフトが相互リンクで説明コストが低下。
- **小さく回せる**：テンプレ／チェックリスト導入で2サイクル目以降が速い。

## 10. ハマりどころと対処

### 10.1 観点が多すぎて収束しない

- **到達条件**を先に置く（数値・対象・計測条件が埋まればドラフトへ）。
- 指摘は**タグ付け**して `対話ログ.md` に残し、**次サイクルで読む**。

### 10.2 参照URL・アクセスデータの品質

- 期間は**直近28日**など明記、PVかsessionsか**片方に統一**。
- **代表テンプレ**以外は除外 or 別バケット（`note`列に理由を1行）。
- **モバイル割合**（`device\_ratio\_mobile`）が高い順に優先。

### 10.3 粒度調整（仕様が膨らむ問題）

- `仕様.md`は**決定事項だけ**。経緯・検討は`**対話ログ.md**`へ退避。
- 受け入れ条件は**短文＋数値**で固定。
- 迷ったら\***Diff要約（3行）**\*で収束：Before／After／理由。

## おわりに（締め）

二層ドキュメントは、AI出力を**量産する仕組み**ではなく、合意と変更提案を**ぶらさず早く**進めるための**運用設計**です。
