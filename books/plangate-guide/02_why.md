> 検証バージョン: **PlanGate v8.10.0**（2026-05）。

前章のクイックスタートで「承認なし実装が止まる」挙動を体験しました。本章では、なぜそんな制約をわざわざ設けるのか ―― その必然性を、本命2章（Plan / Exec）に入る前に押さえます。

## AI が速く書くほど起きること

AI コーディングを実務で使うと、誰もが一度はこうした目に遭います。

- 頼んでいない既存コードを「ついでに」リファクタリングし始める
- 触ってほしくないファイルを書き換えて、無関係な機能を壊す
- 受入基準が曖昧なまま「それらしく動くが要件を満たさない」実装を量産する
- 何をどう判断して実装したのか、後から追えない（ブラックボックス化）

これらは AI が「悪い」のではなく、**指示の曖昧さと、それを止める仕組みの不在**が原因です。人間が手で書いていた頃は、書く速度自体が遅く、その遅さが事実上のブレーキになっていました。AI はそのブレーキを外します。だから、別のブレーキ ―― 承認ゲートと強制力 ―― を明示的に設計する必要があります。

### よくある事故のかたち

具体的な例で考えてみましょう。「ログイン画面にパスワード強度のバリデーションを追加して」と AI に頼んだとします。期待は `LoginForm` に数行のチェックを足すことだけ。しかし返ってきた差分はこうでした。

- `LoginForm.tsx` にバリデーション追加（← これは正しい）
- ついでに `auth/session.ts` の「古い書き方」をリファクタリング（← 頼んでいない）
- リファクタの過程で `useEffect` の依存配列を「最適化」し、セッション更新が一度しか走らなくなった（← 既存機能の破壊）
- テストは緑（← バリデーションのテストは通るが、壊れたセッション更新にはテストがない）

レビュアーは「パスワード強度のチェック」を見に来ているので、`session.ts` の変更は視界の外です。差分が大きいほど、本来の依頼と無関係な変更ほど、人間の目はすり抜けます。そして**テストが緑だから OK**に見えてしまう。これが「速く大量に書く」ことの怖さです。

ここで効くのが「計画」と「強制力」です。事前に「触ってよいのは `LoginForm.tsx` だけ」と計画（スコープ）を宣言しておけば、`session.ts` への編集はその瞬間に検知できます。AI を賢くするのではなく、**逸脱できない枠**を先に作る、という発想です。

### なぜ AI 時代に問題が「増幅」されるのか

これらの問題自体は AI 登場前からありました。スコープ逸脱も、受入基準の曖昧さも、昔からあるアンチパターンです。AI が変えたのは**速度と量**です。

人間が手で書く時代は、1 つの変更に分単位〜時間単位かかり、その間に「あれ、これ余計かな」と立ち止まる余地がありました。書く遅さが、考える時間を強制的に確保していたのです。AI はこの「考える時間」ごと圧縮します。10 分で 500 行が生成され、レビューが追いつく前に次のタスクへ進む。**問題が生まれる速度が、人間が気づく速度を超える**。これが増幅の正体です。

だから対策も「人間がもっと頑張ってレビューする」では追いつきません。生まれる速度に対抗できるのは、同じ速度で働く**機械的なチェック（Hook）**だけです。

## ゲートがない開発 / ある開発

```mermaid
flowchart TB
    subgraph After["✅ ゲートがある開発"]
        direction LR
        A1[指示] --> A2[計画+受入基準]
        A2 -->|C-3 承認| A3[実装]
        A3 -->|Hook が逸脱を検知| A4[計画どおりの成果]
    end
    subgraph Before["❌ ゲートがない開発"]
        direction LR
        B1[曖昧な指示] --> B2[即実装]
        B2 --> B3[スコープ逸脱<br/>受入基準なし]
        B3 --> B4[事故・手戻り]
    end
```

違いは「速さ」ではありません。ゲートがある開発は、実装に入る前に**何を作るかを確定**させ、実装中は**計画からの逸脱を機械が止める**。結果として手戻りが減り、トータルでは速くなります。

## 既存の方法では何が足りないか

「AI の暴走対策なら、すでにある仕組みで足りるのでは？」という疑問は当然です。それぞれが埋める範囲を整理します。

| 手段 | 埋める範囲 | 足りないこと |
|------|-----------|--------------|
| Claude Code の hooks / permissions | ツール実行の許可制御 | 「承認された計画」という概念がなく、計画との整合は見ない |
| CI / branch protection | マージ前の最終関門（テスト・レビュー） | 事後。AI が書く**その瞬間**の逸脱は止められない |
| SpecKit 系のワークフロー | 仕様駆動の手順 | 規約・プロンプト依存で、逸脱を**機械的に強制**する層が弱い |
| 規約ドキュメント（CLAUDE.md 等） | 守ってほしいことの明文化 | 守られたかを検査する強制力がない（読まれない・無視される） |

PlanGate が埋めるのは、これらの**隙間**です。具体的には 2 点。

1. **強制力（enforcement）** — 「計画を承認したか」「スコープ内か」を Hook が実装の瞬間に機械検査する。規約を「お願い」でなく「不変条件」にする。
2. **段階導入（Phase 0〜3）** — いきなり全部を強制せず、チームの習熟に応じて警告 → 強制へ上げられる。

つまり PlanGate は既存ツールの**置き換えではなく補完**です。Claude Code の hooks も CI も併用しながら、「承認された計画からの逸脱」という、どのツールも単独では見ていない軸を機械的に守ります。

## よくある反論への回答

### Q. 承認ゲートを挟むと開発が遅くなりませんか？

短期的には「承認待ち」の分だけ遅く見えます。しかし計測すると、**事故と手戻りの減少がそれを上回る**ことが多いです。スコープ逸脱で壊れた機能のデバッグ、大きすぎる差分の再レビュー、「なぜこうなったか」の調査 ―― これらに比べれば、計画を承認する数分は安い投資です。なお PlanGate には低リスクタスクの承認を自動化する仕組み（C-3 Autonomous APPROVE、第 3 章）もあり、人間の承認帯域は本当に判断が要るものに集中できます。

### Q. うちのチームはそこまで厳格な統制は要らないのでは？

全部を一度に入れる必要はありません。PlanGate は段階導入（Phase 0〜3）で、まず「警告だけ」から始められます。いきなりブロックせず、「どこで何が引っかかるか」を 1 週間観察するところから始める ―― 厳格さはチームの習熟に比例させればよいのです（第 6 章）。

### Q. AI がもっと賢くなれば、こうした統制は不要になりませんか？

賢さと統制は別の問題です。どれだけ賢い実装者（人間でも AI でも）でも、「何を作るべきか（要件）」を取り違えれば誤ったものを作ります。承認ゲートは AI の能力を疑っているのではなく、**人間とAIの認識を一致させる場**です。賢い AI ほど、曖昧な指示を「もっともらしく」埋めてしまうため、むしろ計画の明示が重要になります。

## 統制を「入れないコスト」を見積もる

承認ゲートのコスト（計画を書く時間、承認を待つ時間）は目に見えます。一方で「入れないコスト」は見えにくく、後から分散して襲ってきます。先ほどの `session.ts` 破壊の例で、どこにコストが積もるかを並べてみます。

| 局面 | 入れないコスト |
|------|----------------|
| レビュー | 依頼と無関係な差分まで読む。差分が大きいほど見落とす |
| マージ後 | 壊れたセッション更新が本番で再現、問い合わせ対応 |
| デバッグ | 「いつ壊れたか」を git blame で遡る。原因が無関係な PR だと特定が難航 |
| 再発防止 | 「なぜこの変更が入ったか」を誰も説明できず、教訓化できない |

計画を書く数分・承認を待つ数分は**前払いの固定コスト**です。対して入れないコストは**確率的に発生し、発生すると大きい**。AI で生成量が増えるほど、後者の期待値は上がります。承認ゲートは、見えにくい後払いコストを、見える前払いコストに**付け替える**仕組みだと捉えると腑に落ちます。

## ゲートが「向かない」ケースもある

公平のために書いておくと、承認ゲートが常に正解ではありません。PlanGate 自身も「[使わない方がいい領域](https://github.com/s977043/PlanGate/blob/main/docs/when-not-to-use.md)」を明示しています。

- **30 分で書き捨てる検証コード・スパイク** — 計画の前払いコストが見合わない
- **データ探索・ノートブック** — 試行錯誤そのものが目的で、計画が固定できない
- **インライン補完・リアルタイム支援** — 秒単位の体験に承認は挟めない

これらは「曖昧なまま速く書く」ことに価値がある領域で、ゲートはむしろ邪魔です。逆に言えば、本書が対象とするのは「**チームで継続的にプロダクションコードを育てる**」場面 ―― 事故のコストが高く、後から説明責任が問われる開発です。自分のタスクがどちら側かを見極めるのも、立派な設計判断です。
