title: "計画フェーズを設計する（Plan）— 本書の核"
---

前章では「AI が速く書くほど事故が増える」こと、そしてそれを承認ゲートで止める発想を見ました。本章はその中心 ——「**良い計画とは何か**」を掘り下げます。

## なぜ Plan が最も大切か

本書の主張を一言で言えば、こうです。

> **計画（Plan）の精度が、開発の成否の大半を決める。**

AI コーディングでは、人間が手で書いていた頃よりも「計画の粗さ」がそのまま増幅されます。曖昧な指示を渡せば、AI は曖昧なまま、しかし猛烈な速度でコードを生成します。受入基準が決まっていなければ「それらしく動くが要件を満たさない」実装が一瞬で積み上がります。

逆に言えば、**計画が正しく精度が高ければ、実装はその計画どおりに進み、設計どおりの実装で大きな問題が起きない状態に近づきます**。だから、AI 開発でまず投資すべきは「速く書かせる工夫」ではなく「正しく計画する工夫」です。

ただし誤解しないでください。これは「計画万能論」ではありません。計画が要件そのものを取り違えていれば破綻しますし、実行時のずれもゼロにはなりません。計画の精度は成否の**大半**を決める ―― 残りの差分を埋めるのが次章以降の Exec（強制力）と Verify（検証）です。本書は「計画への投資が最も費用対効果が高い」と主張しているのであって、計画だけで十分とは言っていません。

## 精度の高い計画とは何か

PlanGate は、実装に入る前に次の 4 つの成果物を**同時に**生成します。

- **PBI INPUT PACKAGE**（`pbi-input.md`）— 何を・なぜ作るか（Why / What / 受入基準 / 制約）を人間が書く
- **plan.md** — 実装計画（後述の必須セクション）
- **todo.md** — 作業分解
- **test-cases.md** — 受入基準をテストケースとして先に固定する

ポイントは「**受入基準を実装前に test-cases として確定させる**」ことです。受入基準が後出しになると、AI は「動いたかどうか」を自己判断できず、人間のレビューも「何をもって OK とするか」が曖昧になります。テストケースを先に書くことは、計画の精度をそのまま検証可能性に変換する行為です。

`plan.md` には少なくとも次のセクションを持たせます（PlanGate の `examples/sample-task/plan.md` の構造）。

| セクション | 役割 |
|------------|------|
| Goal | 達成すべき目標 |
| Constraints / Non-goals | 制約と「やらないこと」 |
| Approach Overview | アプローチの概要 |
| Work Breakdown | ステップ・出力・責任・リスク・チェックポイント |
| Files / Components to Touch | 変更対象ファイル一覧（= スコープ宣言） |
| Testing Strategy | Unit / Integration / E2E のテスト戦略 |
| Risks & Mitigations | リスクと対策 |
| Questions / Unknowns | 未解決事項 |
| Mode判定 | ultra-light / light / standard / high-risk / critical |

特に **「Files / Components to Touch」はスコープの宣言**であり、次章で AI に守らせる「触ってよい範囲」の根拠になります。計画の段階でここを曖昧にすると、強制力（Exec）も効きません。

## Plan の「型」と分量の目安

開発者が最も迷うのは「**どの粒度で計画を書けばいいのか**」です。重すぎれば速度を殺し、軽すぎれば事故ります。PlanGate はこれを **Mode** で切り替えます。

| Mode | 目安 | plan の重さ |
|------|------|------------|
| **ultra-light** | 30 分以内の書き捨て・検証コード | plan 省略可・ゲートなし |
| **light** | 小さな機能追加・低リスク変更 | 簡易 plan + C-1 簡易（7 項目） |
| **standard** | 通常の機能開発 | フル plan + C-1（17 項目）+ C-3 |
| **high-risk / critical** | スキーマ変更・破壊的変更・セキュリティ | フル plan + 外部レビュー + 人間 C-3 必須 |

つまり「全部のタスクに重い計画を書け」とは言いません。**リスクに比例して計画の精度要求を上げる**のが PlanGate の型です。書き捨てコードに PBI INPUT を強制するのは過剰であり、公式も「[使わない方がいい領域](https://github.com/s977043/PlanGate/blob/main/docs/when-not-to-use.md)」として明記しています（探索的プロトタイピング、リアルタイム補完など）。

## C-3 承認ゲート — 計画が正しいと認めるまで実装させない

計画を書いたら、実装に入る前に **C-3 ゲート**を通します。C-3 は人間が計画を 3 値で判定するチェックポイントです。

| 判定（c3_status） | 意味 | 次のアクション |
|------|------|----------------|
| **APPROVED** | 計画に問題なし | exec 開始 |
| **CONDITIONAL** | 骨格は有効だが修正が必要 | 指摘を plan に反映 → 簡易 C-1 再実行 → exec |
| **REJECTED** | 根本的な問題あり | plan を再生成（PBI 修正に戻る） |

判定は `approvals/c3.json` の `c3_status`（`APPROVED` / `CONDITIONAL` / `REJECTED`）に記録されます。`CONDITIONAL` なら修正内容（`conditions`）、`REJECTED` なら却下理由（`rejection_reason`）の記載がスキーマ上必須です。「なんとなく OK」を許さず、**承認の根拠を構造化して残す**のが C-3 の本質です。

そしてこの承認記録は次章の強制力につながります。C-3 が APPROVED でなければ、Hook が実装をブロックします（EH-2）。承認は「気持ち」ではなく「機械が参照する状態」なのです。

## C-3 がボトルネックにならない工夫

ここで必ず出る反論が「**人間の承認を挟むと遅くなるのでは？**」です。これは正しい懸念で、放置すると C-3 が開発の渋滞点になります。

PlanGate の答えは 2 つです。

1. **人間の判断を Yes/No に追い込む** — AI に計画段階で確認質問（clarification）を尽くさせ、曖昧さを潰しきった状態で C-3 に出す。人間は「ゼロから考える」のではなく「整った計画を承認/差し戻す」だけにする。
2. **C-3 Autonomous APPROVE**（v8.10.0 で明文化）— 低リスクなタスクは、一定条件下で AI の自律承認を許す。

Autonomous APPROVE の判定はリスクで分かれます。

| 条件 | 自律 APPROVE |
|------|--------------|
| Mode = ultra-light / light | ✅ 可（C-1 PASS のみ） |
| standard + 受入基準 ≤ 5 + 影響が plan の Files に閉じる | ✅ 可（C-1 PASS のみ） |
| standard + 受入基準 > 5 または影響が plan の Files を超える | ⚠️ 条件付き（C-2 必須・重大指摘なし） |
| high-risk / critical | ❌ 不可（人間 C-3 必須） |
| スキーマ変更 / 破壊的変更 / セキュリティ | ❌ 不可（人間 C-3 必須） |

要は「**人間の承認帯域を、本当に判断が必要な高リスクタスクに集中させる**」設計です。低リスクなものまで人間を待たせない。これが「承認ゲート＝遅い」という思い込みへの回答です。

## ハルシネーションを含む Plan を見破る — C-3 チェックリスト

C-3 で人間が見るべき最大のポイントは「**この計画は本当に実在する前提の上に立っているか**」です。AI は一見完璧な計画を、存在しないファイルや API を参照したまま書くことがあります。承認とは、この「もっともらしい嘘」を弾く作業でもあります。

C-3 レビューで最低限確認するチェックリスト：

- [ ] **Files / Components to Touch のファイルは実在するか**（存在しないパスを編集対象にしていないか）
- [ ] **受入基準と test-cases が 1 対 1 で対応しているか**（テストされない受入基準はないか）
- [ ] **スコープ（触る範囲）は PBI の意図に収まっているか**（勝手に広がっていないか）
- [ ] **Non-goals が書かれているか**（「やらないこと」が明示され、暴走の余地を塞いでいるか）
- [ ] **Risks に「未知」が正直に書かれているか**（Questions / Unknowns が空＝危険信号）
- [ ] **Mode 判定はリスクに見合っているか**（破壊的変更を light にしていないか）

「計画が綺麗に書けている」ことと「計画が正しい」ことは別です。C-3 は前者ではなく後者を見るゲートです。

## 計画が担保する範囲の線引き

最後に、計画への過信を戒めておきます。確認質問と C-3 が担保できるのは「**渡された要件に対して、計画が整合的で実行可能か**」までです。要件そのものが間違っていれば（garbage-in）、どれだけ精度の高い計画を立てても出力は間違います。

だから PBI INPUT PACKAGE の Why / 受入基準は人間が責任を持って書く必要があります。PlanGate は「正しい要件かどうか」までは保証しません ―― そこは人間の領分として残します。計画フェーズが担保するのは「**要件 → 実装の翻訳精度**」であって「要件の正しさ」ではない、という線引きを忘れないでください。

## 実装途中で計画の不備に気づいたら（Re-plan）

完璧な計画を一度で立てられることは稀です。実装に入ってから「この設計では無理だ」と分かることは普通に起きます。重要なのは、**そのとき計画を黙って逸脱しないこと**です。

PlanGate では、承認後に `plan.md` を書き換えると Hook（EH-3）が `plan_hash` の不一致を検知します。これは「承認した計画と、いま実装している計画が食い違っている」ことを機械的に捕まえる仕組みです。だから計画を変えたいときは、こっそり直すのではなく：

1. 一度立ち止まり、計画の不備を `plan.md` に反映する
2. 変更の規模に応じて C-3 に**戻って再承認**を得る（軽微なら CONDITIONAL の範囲で）
3. `c3.json` の `plan_hash` を更新し、承認と計画を再び一致させる

ガバナンスを「足かせ」にしないコツは、**Re-plan を例外でなく正規ルートとして設計しておく**ことです。計画は変わってよい。ただし「変えたことを承認境界に通す」のが PlanGate の流儀です。

## まとめ

- 計画の精度は開発の成否の大半を決める。AI 開発ではまず計画に投資する。
- 精度の高い計画とは、受入基準を test-cases として先に固定し、スコープを宣言した計画。
- 粒度は Mode でリスクに比例させる。重い計画を全タスクに強制しない。
- C-3 は「綺麗さ」でなく「正しさ」を見るゲート。チェックリストでハルシネーションを弾く。
- Autonomous APPROVE で人間の承認帯域を高リスクに集中させ、渋滞を防ぐ。
- 計画は変わってよい。ただし Re-plan は承認境界を通す。

次章では、この「承認された計画」を **実行時に守らせる強制力（Exec）** を見ていきます。
