## 0. 本書の主張（背骨）

本書を貫く一本の主張は次の通り。章はすべてこの順序を支えるために存在する。

> **計画（Plan）の精度が、開発の成否の大半を決める。**
> 計画が正しく精度が高ければ、実装はその計画どおりに進み、設計・計画どおりの実装で大きな問題が起きない状態に近づく。
> だからこそ **Plan が最も重要**。
>
> ただし、どんなに良い計画でも**実行時に守られなければ意味がない**。
> よって、計画を実装フェーズで守らせる **Exec（強制力）が second pillar** になる。
>
> そして「計画どおりに守られたこと」を後から証明し、計画の精度を上げ続けるために **Verify / Scale** が残差を埋める。

この2本柱（**Plan = 何を作るかの精度 / Exec = それを逸脱させない強制力**）が本書の本命。
Why はこの主張の導入、Verify / Scale は「守られたことの証明と、計画精度の継続改善」の補強と位置づける。

> ⚠️ **過剰主張をしない**: 「計画が完璧なら検証は不要」とは言わない。計画の精度は成否の**大半**を決めるが、残差（要件自体の誤り・実行時のずれ）を Exec と Verify が埋める。本書は「計画万能論」ではなく「計画への投資が最も費用対効果が高い」という主張である。

PlanGate の用語に対応づけると:

- **Plan の精度** = PBI INPUT → 確認質問 → plan / todo / test-cases の同時生成 → **C-3 承認ゲート**（人間が計画を承認するまで実装させない）
- **計画を守らせる Exec** = **Hook 強制**（EH-1〜EH-10 + EHS-1〜EHS-3）による scope / approval / evidence の不変条件検査（「計画外のファイルを触る」「承認なしに実装する」を機械的にブロック）＋ **C-4**（PR レビューゲート）による人間の最終確認

### 本命2章の書き分け（最重要）

- **03 Plan（本命①）** — 「なぜ Plan が最も大切か」を本書の中心命題として正面から論じる。
  - **精度の高い計画とは何か**: PBI INPUT の埋め方、確認質問で曖昧さを潰す、test-cases で受入基準を先に固定する、plan / todo / test-cases の同時生成。
  - **C-3 承認ゲート**で「計画が正しいと人間が認めるまで実装に進ませない」運用を具体例で示す。
  - **C-3 がボトルネックにならない工夫**（必須）: 「AI に確認質問を尽くさせ、人間は Yes/No を判断するだけの状態に追い込む」負荷低減テクニックを書く。重い承認＝速度を殺す、という反論への回答。
  - **Re-plan（計画修正のライフサイクル）**（必須）: 実装途中で計画の不備に気づいた時、安全に C-3 へ戻って再承認を得る変更管理。ガバナンスが「足かせ」にならない設計。
  - **計画が担保する範囲の線引き**（必須）: 「計画通りでも要件自体が間違っていれば破綻する（garbage-in）」への先回り。確認質問が要件妥当性をどこまで担保し、どこからは担保しないか。
  - **Plan の「型」と分量の目安**（必須 / Gemini 指摘）: 開発者が最も迷うのは「どの粒度の計画を書けば AI も人間も困らないか」。「機能追加ならこの程度 / リファクタならこの程度」の型と分量目安を示す。
  - **ハルシネーションを含む Plan の見破り方**（必須 / Gemini 指摘）: 一見完璧でも存在しないファイルを参照している等の Plan を、人間が C-3 でどうレビューするか。**C-3 チェックリスト**を提供する。
  - 素材記事は「テンプレート紹介」止まりなので、**「計画の精度が実装品質を決める」という因果の説明を加筆**する。
- **04 Exec（本命②）** — 「良い計画も守られなければ無意味」を起点に、**計画を実行時に逸脱させない強制力**を主役にする。
  - Hook 強制（EH-1〜EH-10 + EHS-1〜EHS-3）で scope 外編集・承認なし実装・evidence なし PR が**機械的にブロックされる**挙動を、**コンソール出力つき**で再現。
  - **記号の羅列でなく「何を防ぐか」を軸に**（C-3/C-4/L-0/V-1〜V-4/EH/EHS の暗記を読者に強いない。記号は表に集約し本文は意味で書く＝陳腐化耐性）。
  - C-4（PR レビューゲート）で人間が最終確認する流れも含める。
  - **既存 CI との共存**（必須 / Gemini 指摘）: ローカル Hook（PlanGate）と GitHub Actions 等の CI（branch protection）の役割分担と二重投資の防ぎ方。
  - Next.js TDD の詳細（`ai-driven-tdd-nextjs`）は分量過大（単体 1277 行）なため**本文に抱えず既存記事へ外部リンク**し、本文は「計画を守らせるループ」に絞る。

> 既存記事はすべて Zenn 公開済み（`published: true`）。Book 化では**重複コンテンツ扱いにならないよう**、記事は「単発の入口」、Book は「体系的な通し読み」と役割分担し、相互リンクで補完する。**再編集は本文コピペでなく、各章で「本命主張（Plan→Exec）への接続文」を新規に書き起こす**。稼ぎ頭 entry-point 記事（`ai-legible-repository-design` / `plangate-v86-hook-enforcement`）は canonical を記事側に置き、Book 章は冒頭で元記事へリンクする（自作 Book が本家記事の検索順位を希釈する自家中毒を防ぐ）。
