# レビューゲート 記事クラス別基準

多段 AI レビュー（3ペルソナ → Codex 二次 → Gemini 検証）は強力だがコストが重い。**全記事に一律適用しない**。記事クラスでレビュー強度を分岐する基準を定める。

- 由来: Codex 方針検討 + Gemini 検証（マルチAI連鎖, 2026-05-17）の提案 3/3
- 関連: [`multichannel-rollout-template.md`](./multichannel-rollout-template.md) の §4、[`content-channel-strategy.md`](./content-channel-strategy.md)

## レビュークラス

| クラス | 適用記事 | レビュー構成 | 目安コスト |
|---|---|---|---|
| **Full** | 中核テーマ / 高リスク / 事実主張が多い / 多媒体展開の正本 | 3ペルソナ reviewer → Codex 二次 → Gemini 検証 | 高（3エージェント連鎖） |
| **Standard** | 通常の技術記事（単一媒体・既知テーマ） | 3ペルソナ reviewer のみ。判定 NEEDS_FIX 時のみ Codex 二次を追加 | 中 |
| **Light** | Qiita 短文 Tips / 小修正 / 既存記事の軽微更新 | 単一レビュー（セルフ or 1エージェント）＋下記「Light 必須チェック」 | 低 |

## クラス判定基準（上から評価し最初に該当したクラスを採用）

1. 次のいずれかに該当 → **Full**
   - 多媒体展開の**正本**記事（誤りが全媒体に波及）
   - 外部事実・引用・ベンチマーク等の検証可能な主張を複数含む
   - セキュリティ / 法務 / 公開ポリシーに触れる
   - シリーズの起点（後続記事が前提にする）
2. 次のいずれかに該当 → **Light**
   - 1,500 字未満の Tips / トラブルシュート単発
   - typo・リンク・体裁のみの修正
   - 既存記事の事実関係を変えない軽微更新
3. 上記以外 → **Standard**

## Light 必須チェック（軽量でも省略しない）

Gemini 検証指摘：軽量化すると「AI 臭さ（無難で退屈な表現）」が残り、付加価値なし判定で検索順位が下がる。Light でも以下は必須。

- [ ] 技術的な誤り・古い情報がないか（ファクトチェック）
- [ ] 執筆者の一次体験・独自見解（I-message）が最低1つ入っているか
- [ ] AI 生成特有の冗長・一般論で終わっていないか（独自性）
- [ ] 媒体の AI 生成コンテンツポリシー（Qiita 等）に適合しているか

## Goal ベース Definition of Done（全クラス共通の合格条件）

Gemini 検証指摘：クラス定義だけだと疲労時に「Standard：なんとなく」「全部 Light」へ収束し、レビュー強度の形骸化が起きる。**レビュー強度は手段であって合格条件ではない**。クラスに関わらず、公開前に次の Goal-DoD を1行で言語化して満たすことを必須とする。

- **この記事は「誰の・どの課題」を解決するか**を1行で書けるか（書けないなら公開しない）
- その課題に対し、**読者が次にとる行動 / 持ち帰り**が明確か
- 競合（自分の既存記事含む）にない**一次体験・独自洞察**が最低1つあるか

記事 PR 本文の Summary 直下に以下を1行で記載する（CI 強制はしない・著者レビューで確認）:

```
レビュークラス: <Full|Standard|Light> / 根拠: <1行>
Goal-DoD: 誰の<対象>の<課題>を解決し、読者は<行動>できる
```

レビュークラスはこの Goal を満たすための手段選択にすぎない。Goal-DoD が言語化できていれば、クラスが Light でも合格しうる。

## 運用メモ

- 迷ったら1段上のクラスを選ぶ（過小レビューより安全側）
- Full の Codex/Gemini は `codex:codex-rescue` / Gemini CLI をバックグラウンド起動し、完了通知駆動で統合する（対話を待ち時間で消費しない）
- クラスは記事単位で `multichannel-rollout-template.md` §4 に記録し、判断根拠を残す
- レビュー強度の基準自体が陳腐化したら本ファイルを改訂（改訂日と根拠を残す）
