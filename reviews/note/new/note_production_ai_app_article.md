### 該当箇所 6
L79-L100（`agents/` の定義が記事内で独自に狭く使われている）

```
### `agents/` は知的判断を担当する層

`agents/` には以下のような構成がある。

- `document_grader.py`
- `query_decomposer.py`
- `adaptive_router.py`
- `tools/vector_search.py`
- `tools/web_search.py`
- `tools/code_search.py`

このあたりは、単純なRAGを超えて、より柔軟に判断するための層だと思う。

たとえば、

- 複雑な質問を分解する
- 取得した文書が本当に有効か判定する
- ベクトル検索、Web検索、コード検索のどれを使うか選ぶ

といった処理は、固定ロジックだけではなく、LLMを補助的に使って判断させると相性がいい。
```

### 問題点
想定読者視点での技術的正確性の論点。一般に LLM エージェントは「自律的にツールを選択・実行する主体（Planning + Tool Use + Memory + 観測ループ）」として定義されることが多い（Anthropic / OpenAI の最新 SDK も同様の枠組みで整理している）。本記事の `agents/` 定義「知的判断を担当する層」「LLMを補助的に使って判断させる」は、エージェントの一機能（LLM-as-Judge、LLM-assisted routing）に寄せた狭い定義になっている。

図のファイル名（`document_grader.py` `query_decomposer.py` `adaptive_router.py` + `tools/`）を見ると、これは **adaptive/corrective RAG（例: CRAG / Self-RAG）で使う「LLMを補助的に使う判断モジュール群 + ツール群」を `agents/` にまとめた構成**と解釈するのが自然で、一般的な「自律エージェント」（ReAct / AutoGPT 系）とはスコープが異なる。

記事内の `agents/` 定義を狭く使うこと自体は構わないが、**一般的なエージェント定義との違いに触れないまま進むと、読者は「これが agent の一般像」と誤認しかねない**。L98「図の右側にある `self-correcting retrieval` や `LLM-driven source selection` という説明」という記述があるので、このキーワードを手がかりに、「一般的な自律エージェントではなく、adaptive RAG 的な判断モジュール群として `agents/` を定義している図である」と1行注記するのが親切。

### 提案
L90-L91 の間、あるいは章頭に以下を挿入:
```

```
