# 個人開発でAIエージェントチームを作るために、ヘルメスを導入した レビュー

対象: `articles_note/new/hermes-introduction-note.md`

## 総評

方向性は合っている。現行稿は「個人開発で複数AIを使うと、依頼管理が必要になる」という主張に寄っており、前回の「会社・人間チーム導入」に見えるズレは解消されている。

一方で、まだ「ヘルメス導入の記事」としては固有性が弱い。読者には「それはNotion、GitHub Issues、Linear、テンプレート管理でもよいのでは」と見える可能性がある。次の改稿では、ヘルメスを何として扱うのか、導入済みなのか導入前の仮説なのか、最初にどの依頼で検証するのかを明確にするとよい。

## 優先修正

### 1. 冒頭をドラフト宣言ではなく読者の痛みから始める

現状の冒頭は「いったんドラフト」「あとで差し込む前提」から始まるため、読者向け記事として温度が下がる。

修正方針:

- 冒頭2段落の制作メモは削除、または末尾の執筆メモへ移す
- 最初は「AIエージェントを増やすほど、依頼管理が散らかる」という問題提起から入る

例:

```md
個人開発でAIエージェントを使うほど、開発は速くなります。

一方で、依頼そのものは散らかり始めます。

実装相談、レビュー依頼、記事構成の相談を別々のチャットで進めているうちに、前提や判断理由を探し直す時間が増えてきました。
```

### 2. ヘルメスが何かを早い段階で定義する

本文ではヘルメスを「依頼窓口」「司令塔を外に出す場所」として扱っているが、読者にはツールなのか、自作システムなのか、管理基盤なのかが分からない。

修正方針:

- 冒頭付近に「この記事でいうヘルメス」を1段落で置く
- 正式仕様が未確定なら、機能紹介ではなく「この記事内での扱い」として限定する

例:

```md
ここでいうヘルメスは、個人開発で複数のAIエージェントに依頼を出すための窓口として使おうとしている仕組みです。この記事では機能紹介ではなく、AIエージェントチームを運用するための依頼管理として整理します。
```

### 3. 「導入した」と「導入前」の時点をそろえる

タイトルは「導入した」だが、本文には「導入前の時点では」「導入後に見たい」があり、実際のステータスが曖昧。

修正方針:

- 導入済みなら「導入直後」と明記する
- まだ検討中ならタイトルを「導入を考えている」に寄せる

導入済みの場合の例:

```md
導入直後の時点では、まず「AIに何を頼むか」を置いておく場所として使い始めています。
```

### 4. 具体例を入れて、抽象論から依頼文の変化へ落とす

現行稿は概念中心で、「ヘルメスに置くと何が変わるのか」がまだ見えにくい。

修正方針:

- 「チャットに直接投げる依頼」と「ヘルメスに置く依頼」の比較を1セット入れる
- 悪い例と良い例で、目的、前提、制約、完了条件、停止条件の差を見せる

例:

```md
悪い依頼: このIssue進めて。

ヘルメスに置きたい依頼:
- 目的: Issueの実装前に、変更範囲とリスクを整理する
- 前提: まだコードは変更しない
- 制約: 既存設計を優先し、新しい依存は提案だけに留める
- 完了条件: 変更対象ファイル、懸念点、実装方針案が出ている
- 停止条件: 認証、決済、本番データに触れる可能性が出たら止める
```

### 5. AIエージェントチームの条件を定義する

「複数AIを使っている状態」と「AIエージェントチームとして運用している状態」の境界がまだ曖昧。

修正方針:

- チームとして扱う条件を短く定義する
- 流れを1つ示す

例:

```md
自分の中では、次の4つが揃ったらAIエージェントチームとして扱えると考えています。

- 役割が分かれている
- 依頼と完了条件が残る
- 成果物を次のAIに渡せる
- 最後に自分の判断点がある

たとえば、調査AIが前提を整理し、設計AIが方針を出し、実装AIが差分を作り、レビューAIが確認し、最後に自分が採否を決める、という流れです。
```

## 追加で効く修正

### 安全設計を少し足す

個人開発ではレビュー者が自分だけなので、AIに任せない範囲を明示すると実用性が上がる。

入れるとよい項目:

- 認証、決済、秘密情報を扱う変更
- 本番データ操作
- 破壊的なGit操作
- 公開済みコンテンツの無断変更
- 新しい依存追加

### 検証方法を少し具体化する

「読み返しやすくなったか」だけでは成功判定が曖昧。

例:

- 2週間、ヘルメス経由の依頼件数を数える
- 再利用できた依頼の件数を見る
- AI間の引き継ぎで説明を書き直した回数を見る
- 人間レビューで止めた依頼の件数を見る

### PlanGate / River Reviewer の説明を補う

固有名詞を残すなら、一言説明を入れる。説明を増やしたくないなら削る。

例:

```md
これは、これまで PlanGate（実装前の計画確認）や River Reviewer（レビュー支援）を触ってきて感じていることともつながります。
```

## 採用優先度

1. 冒頭の差し替え
2. ヘルメス定義の追加
3. 導入ステータスの統一
4. 依頼例の追加
5. AIエージェントチーム条件の追加
6. 安全設計と検証指標の補強

## 現時点で残してよい強み

- 「個人開発でもAIが増えるとマネジメントが必要になる」という主張は良い
- 「自分が司令塔であることは変わらない」という線引きは必要
- 「どこまで任せられるかではなく、どこで止められるか」は締めとして強い
- note向けに、技術仕様ではなく体験と運用設計に寄せている点は媒体方針に合っている
