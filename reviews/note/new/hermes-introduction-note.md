# 個人開発でAIエージェントチームを作るために、ヘルメスを「依頼窓口」として導入し始めた レビュー

対象: `articles_note/new/hermes-introduction-note.md`

## 総評

方向性は良い。この記事の訴求は「ヘルメスの機能紹介」ではなく、「個人開発で複数AIを使うほど、依頼・前提・判断が散らかる。その整理先としてヘルメスを依頼窓口にする」という点にある。

この訴求は残すべき。読者に刺さるのは、ツール名そのものではなく、AIを使えば使うほど自分の管理コストが増えるという実感だから。

公開前に強化すべきなのは、次の4点だった。

- ヘルメスを万能な自動化ツールに見せない
- 読者が真似できる依頼テンプレートを増やす
- 読後に最初の一歩を踏み出せる最小フローを示す
- 画像や図が必要かを判断し、必要ならどこに入れるかを決める

今回の改善では、記事の訴求を変えずに、導入理由、運用設計、安全設計、検証項目、最初の運用フローを補強した。

## ループ1: 初見読者視点レビュー → 改善

### 観点

初見読者は「ヘルメスとは何か」を知らない可能性が高い。

元記事は、あえて機能紹介に寄せず「依頼管理」として整理していた。この方向性は良いが、冒頭だけ読むと「ヘルメスに何を期待しているのか」が少し抽象的だった。

### 指摘

- ヘルメス固有の説明が少なく、読者が置いていかれる可能性がある
- ただし、機能紹介に寄せすぎると記事の主旨がぼやける
- 「ツール導入記事」ではなく「個人開発の運用設計記事」として読みたい

### 改善

冒頭に、ヘルメスを次のように定義し直した。

```md
この記事でいうヘルメスは、個人開発で複数のAIエージェントに依頼を出すための窓口として使おうとしている仕組みです。
```

さらに、すぐ後に次の補足を追加した。

```md
ただし、ここで書きたいのはヘルメスの機能一覧ではありません。
むしろ、チャットに流れがちな依頼、前提、制約、判断理由を、あとから読み返せる形で残すための運用設計として整理したいと思っています。
```

これにより、読者は「ヘルメスの機能解説」ではなく「AIエージェントチーム運用の入口」として記事を読める。

## ループ2: 実用性レビュー → 改善

### 観点

note記事として読むだけでなく、読者が自分の個人開発に持ち帰れるかを見た。

元記事には「このIssue進めて」と「目的・前提・制約・完了条件・停止条件」の比較があり、実用性はすでにあった。ただし、AIエージェントチーム運用として見ると、引き継ぎ先と人間の確認点まで含めるとより強くなる。

### 指摘

- 依頼テンプレートは良いが、AI間の引き継ぎまで考えるなら項目を増やしたい
- 「人間がどこで見るか」がテンプレート内にあると、記事の主張と一致する
- 検証項目は「まず検証したいこと」より「最初の2週間で見る項目」の方が行動に移しやすい

### 改善

依頼テンプレートに次の2項目を追加した。

```md
引き継ぎ先: 実装方針を確認後、実装AIに渡す
人間の確認点: 変更範囲、リスク、停止条件の妥当性を確認する
```

また、検証セクションを「最初の2週間で見る項目」に変更し、導入直後に見るべき指標を明確にした。

```md
- ヘルメス経由で出した依頼の件数
- 再利用できた依頼テンプレートの件数
- AI間の引き継ぎで説明を書き直した回数
- 人間レビューで止めた依頼の件数
- 依頼後に完了条件を修正した回数
```

これにより、記事が単なる思想ではなく、導入後の観察項目まで持つ内容になった。

## ループ3: 訴求強化・構成レビュー → 改善

### 観点

ユーザーへの訴求点を変えずに強化できているかを確認した。

この記事の核は「AIに丸投げする話ではなく、AIへ渡す仕事の形を整える話」である。ここが弱まると、ヘルメス導入が単なる流行ツール導入に見えてしまう。

### 指摘

- 「ヘルメスを入れれば自動で回る」という誤解を先に潰した方がよい
- 記事全体を通して「どこまで任せるか」より「どこで止めるか」が重要だと伝えるべき
- PlanGate / River Review との接続は強みなので残す
- 末尾に続編への余地を作ると、導入し始めた記事として自然に閉じられる

### 改善

冒頭直後に、次のセクションを追加した。

```md
## ヘルメスを「万能な自動化ツール」として見ない
```

このセクションでは、ヘルメスを魔法の自動化ツールとして扱わず、依頼の型や停止条件を置く場所として使うという期待値を明確にした。

また、末尾に次の一文を追加した。

```md
実際に運用してみて、依頼の型がどこまで再利用できたか、どこで破綻したかは、あらためて整理したいです。
```

これにより、「導入し始めた」というタイトルと本文の検証前提がそろった。

## ループ4: 読後行動レビュー → 改善

### 観点

3回目までの改善で、記事の問題提起と思想はかなり明確になった。

一方で、読者が読み終わった後に「では、自分は最初に何をすればよいか」まで自然に移れるかを見ると、まだ少し弱かった。

この記事は導入手順の記事ではない。ただし、読者が自分の運用に持ち帰るには、最初に回す小さなフローが必要になる。

### 指摘

- 「依頼の型」はあるが、それをどの順番で使うかがまだ弱い
- 「最初の依頼は小さくする」だけだと、行動の粒度としては少し粗い
- 最初から自動化しない、という主張をより実践に落としたい
- 5ステップ程度の最小運用フローがあると、読者が真似しやすい

### 改善

`## 依頼文を「仕事」として残す` の後に、次のセクションを追加した。

```md
## まずは5ステップの最小フローで回す
```

追加した5ステップは次の通り。

```md
1. 思いついた依頼を、そのままAIに投げない
2. 目的、前提、制約、完了条件、停止条件に分ける
3. 調査、設計、実装、レビューのどこまで任せるかを決める
4. 実行前または次のAIに渡す前に、自分が確認する
5. 結果、判断、次に渡す材料を残す
```

この追加により、記事の主張が次のように自然につながった。

```text
AIを使うほど依頼が散らかる
↓
ヘルメスを依頼窓口として使う
↓
依頼文を仕事として残す
↓
まずは5ステップの最小フローで回す
↓
導入時のルール、最初の依頼、検証項目へ進む
```

### 効果

4回目の改善で、記事が「考え方の提示」から「最初の行動まで示す記事」に一段進んだ。

導入手順に寄せすぎず、読者が明日から試せる粒度を足せた点が良い。

## 画像挿入の検討

### 判断

本文中に画像を無理に挿入する必要はない。

理由は、この記事の主役がツール画面や手順ではなく、「依頼が散らかる → 依頼窓口を作る → 停止条件を決める → 最小フローで回す」という考え方だから。

ただし、公開時に差し込み画像を1枚入れるなら、次の位置が最も効果的。

### 推奨する差し込み位置

`## まずは5ステップの最小フローで回す` の直後。

ここで、読者が「依頼窓口」と「AIエージェントへの分岐」を同時に理解できる図があると、本文の行動導線が強くなる。

### 推奨する画像内容

タイトル案:

```text
AIエージェントチームへの依頼窓口
```

構図:

```text
User / Owner
  ↓
Hermes: 依頼・前提・制約・完了条件・停止条件
  ↓       ↓       ↓       ↓
調査AI   設計AI   実装AI   レビューAI
  ↓       ↓       ↓       ↓
判断・再利用・次の依頼へ
```

デザイン方針:

- 人物イラストは不要
- 抽象的なUI図にする
- 白背景または淡いグレー背景
- 黒文字＋アクセント1色
- note本文中の横長差し込み画像として使う
- 文字量は少なくする

### 今回の対応判断

今回は本文への画像参照は追加しない。

理由は、画像ファイル未作成のまま `![]()` だけを入れると、noteインポート時に壊れた参照になるため。

このリポジトリでは、note記事の画像参照は `../assets/<name>.png` などの形式に揃える必要がある。画像を入れる場合は、PNGを作成して `articles_note/assets/` に配置してから本文へ参照を追加するのが安全。

## 最終レビュー

### 良くなった点

1. 冒頭の期待値調整が強くなった
   - ヘルメスを万能ツールではなく、依頼窓口として扱う前提が明確になった

2. 記事の独自性が強くなった
   - 単なるツール紹介ではなく、個人開発におけるAIエージェント運用設計の記事になった

3. 読者が真似しやすくなった
   - 依頼テンプレートに、引き継ぎ先と人間の確認点が入った

4. 読後行動が明確になった
   - 5ステップの最小フローにより、最初に何を試せばよいかが見えるようになった

5. 安全設計が残っている
   - 認証、決済、個人情報、本番データ、破壊的Git操作などの停止条件が明示されている

6. 検証可能になった
   - 最初の2週間で見る項目が入り、導入後の観察ポイントが明確になった

### まだ気になる点

- ヘルメス自体の詳細な導入手順はない
- 実際の画面や運用ログはまだない
- 導入後の結果は別記事が必要

ただし、この記事は「導入し始めた」段階の記事なので、この粒度で問題ない。導入手順や運用結果まで入れると、記事の焦点がぼやける。

## 公開判断

公開前ドラフトとしては十分に読める。

特に、次の読者には刺さりやすい。

- 個人開発でAIエージェントを複数使い始めた人
- ChatGPT、Claude、Codex、Geminiなどを使い分けているが、依頼履歴が散らかっている人
- AIに任せる範囲と止める範囲を整理したい人
- 個人開発を「AIエージェントチーム運用」として扱いたい人

今回の改善では、ユーザーへの訴求点は変えずに、以下を強化できた。

- 問題提起: AIを使うほど依頼が散らかる
- 解決方針: ヘルメスを依頼窓口として使う
- 実用性: 依頼テンプレート、最小フロー、検証項目を提示する
- 安全性: 任せない範囲と停止条件を明確にする
- 続編性: 運用結果を次の記事につなげる
