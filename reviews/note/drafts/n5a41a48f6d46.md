### 該当箇所 4
L19-L23（箇条書きで問題事象を列挙しているが、「AIが」の主語が省略されていてスマホで読むと誰の行動か一瞬詰まる）

```
- 頼んでいない変更まで広げる
- テストが後回しになる
- それっぽい代替案で進めてしまう
- セッションが切れると、どこまでやったか分からなくなる
```

### 問題点
2項目めだけ主語が「テスト」になっており、1・3・4 項目は AI が主語、5 項目は「人間（チーム）」が主語と、主語が項目ごとに揺れる。共感喚起を狙う箇条書きでは主語が揃っていないと読者の自分ごと化が阻害される。スクロール読みされる note では特に顕著。

### 提案
```
- AIが頼んでいない変更まで広げる
- AIがテストを後回しにする
- AIがそれっぽい代替案で進めてしまう
- セッションが切れると、AIはどこまでやったか分からなくなる
- PR の段階で、人間が初めてズレに気づく
```
主語を各行冒頭に明示し、AI と人間のどちらの問題かをひと目で判別できるようにする。

### 該当箇所 5
L50-L51（長文が1段落に詰まっており、スマホで3行を超える／並列構造も読みにくい）

```
アジャイル開発やスクラムでは、何をやるかだけでなく、何をやらないか、どうなったら終わりか、既存機能にどう影響するかが大事です。
でもAIは、そうした制約が弱いと、善意で範囲を広げたり、実装しやすい形に問題を変形したりします。
```

### 問題点
L50 は全角で約60文字、読点4つでつながっており、スマホ（iPhone 標準幅）では4〜5行に折り返される。オピニオン記事の核心部分にあたる箇所で、視線が停滞すると主張の輪郭がぼやける。

```
- 何をやるか
- 何をやらないか
- どうなったら終わりか
- 既存機能にどう影響するか

でもAIは、こうした制約が弱いと、善意で範囲を広げたり、実装しやすい形に問題を変形したりします。
```

```
## 具体的なテンプレートはZennにまとめた

この記事では、考え方と設計意図を中心に書きました。

```

### 問題点
note の想定読者（非エンジニア寄りの読者も含む）に対して、記事の山場直後で外部プラットフォーム（Zenn）への誘導を行うのは離脱リスクが高い。また「Zennにまとめた」という受動的見出しは note の検索・おすすめ流入の観点でクリックスルーを誘発しにくい。note 内で完結する読後体験を先に提示し、Zennは補足リンクとする構成が望ましい。

```
<https://zenn.dev/minewo/articles/plangate-ai-coding-workflow>
```
読後の行動喚起を note 内で完結させ、外部誘導は補足に回す。

## 総合評価

### 良い点
- 主張（「ハーネスなしではフィットしにくい」）が冒頭・中盤・結論で一貫しており、オピニオン記事として骨格が明快
- 「個人では気持ちよく回るがチームだと危うい」という共感点から入り、読者の自分ごと化に成功している
- スクラムの PBI 構造を AI 実行向けの制御情報として再解釈する視座は新規性があり、EM／テックリード層の関心に刺さる
- PlanGate という固有名の提示＋ GitHub／Zenn への導線で、記事を単なる読み物でなく実践資産として位置付けている

### 改善点
- 中核語（AI駆動開発／AIコーディング／AIレビュー）の関係がリードで未整理
- 強調（bold）の多用で主張の優先順位が平坦化
- L165 の `[plan.md](http://plan.md)` など、実在しないURLへのリンクが生成されている（note 公開時の信頼性リスク）
- 箇条書きの主語揺れ（L19-23）
- 結びに読者への問いかけ／CTA がなく、note 内エンゲージメント機会を取り逃している

### 推奨アクション
1. **最優先**: L165 のファイル名リンクをコードスパンに置換し、note 側でプレビューして実表示を確認する
2. L11-15 のリードで中核語の関係を1文定義し、敬体に統一する
3. L19-23 の箇条書きを主語明示形に整形する
4. 強調箇所を 2箇所程度に絞り込む（L28, L81, L93, L109 を解除）
5. 結び（L185 以降）に読者への問いかけ段落を1つ追加する
6. draft 反映時は note 側の本文・見出し・リンクカード展開を必ず現物確認する（`drafts/` の状態整合ルール）
