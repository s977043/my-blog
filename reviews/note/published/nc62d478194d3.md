```

### 該当箇所 6
L74 / L222（JTFスタイル違反：全角ダブルクォート「“ ”」と疑似ダッシュ記号）

```
3. **AI実装**：Claude Code/Gemini CLIに**“Redをにする最小変更”**を促す。
…
→ **対策**：**振る舞い（ユーザー視点）**で書く。DOMはRole/Label、APIは入出力契約を固定。
```

### 問題点
本文中で `“ ”`（U+201C/U+201D）が複数箇所で使われている（例：L74 の "Redをにする最小変更"、L258 の "Red"、L268 の "Red" など）。JTFスタイルでは和文引用はカギ括弧「」を用い、欧文引用符は用いないのが原則。加えて、箇条書きのインデントに用いられている `→` は矢印記号として許容範囲だが、ハッシュ `＝` と半角 `=` が混在している（例：L18「AITDD＝」、L262「1PR＝」）ため、表記統一が望ましい。

### 提案
```
# 全角ダブルクォート “ ” はカギ括弧「」に統一
#   例: 「Red を Green にする最小変更」
# 「＝」は全角に統一（"AITDD＝""1PR＝" 既に全角で統一済み、OKだが、
#   本文中の英数字は半角、記号は和文と結合する場合は全角、という原則を明示すると編集が楽）
```

## 総合評価

### 良い点
- 「AITDD」という造語で主張の核をワンワード化できており、タイトル・リード・まとめで一貫して繰り返されているため、記事の記憶残存性が高い。
- Quick Start、ワークフロー、テンプレ、失敗パターン、測定、FAQ、用語辞書、参考リンク、CTA まで、note 系ハウツー記事の定番構成を網羅しており、読者タイプ別に拾い読みできる設計になっている。
- 「先にテストを書く」という主張を、抽象論で終わらせず Issue テンプレ・Jest コード・CI YAML・プロンプト例まで具体物で示している。技術的正確性（Vitest/Jest 両対応の姿勢、`actions/checkout@v4` `setup-node@v4` 等の現行バージョン）も概ね適切。

### 改善点
- 章構成レベルの重複が2箇所（「5)6)」実装/CI 章、「まとめ」章）あり、読者体験を大きく損ねている。編集時のコピペミスと推定されるため、公開後の修正優先度は最高。
- 本文の中核となる「Redを Green にする」のキーワードが L74 で欠落している。主張の中核語が抜けており、発見次第の差し替えが必須。
- 太字記法 `**…**` の開閉不一致、バックスラッシュエスケープの残存、全角ダブルクォートとカギ括弧の混在、といった Markdown/JTFスタイル準拠レベルの瑕疵が複数ある。note の編集画面でプレビューしながら一括チェックすることを推奨。
- 「Quick Start」で React Testing Library をインストールするが、例示される `sum.test.ts` は純関数のテストであり、DOM 環境設定が無いまま進むと将来的な混乱を招く。最短導線では依存を最小化すべき。
- 画像キャプション（L58 "「テスト→チェック→実装」の順序を示す抽象アイコン"）が画像直下の通常段落として記述されており、note の画像キャプション機能を使えば見栄えが統一される。

### 推奨アクション
1. **重複章の削除（最優先）**：L184-L213 と L266-L274 を削除する。
2. **L74 の脱字修正**：「Red を **Green** にする最小変更」に修正し、全角ダブルクォートをカギ括弧に置換する。
3. **Markdown 装飾の総点検**：太字 `**` の開閉、バックスラッシュエスケープ（`\_` `\*`）、引用符の向き `“” "` を一括置換で確認。note エディタにペーストしてプレビューで検証する。
4. **リード段落の誤字修正**：L17「出力せたくなる」→「出力させたくなる」。
5. **Quick Start の依存最小化**：`@testing-library/react` を Quick Start から外し、React コンポーネントテストを扱う続編記事に回す。
