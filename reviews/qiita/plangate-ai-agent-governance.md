# Qiita記事レビュー: plangate-ai-agent-governance

対象: `Qiita/public/plangate-ai-agent-governance.md`

## レビュー方針

Qiita向けに、技術仕様よりも体験談・チーム開発・実務の変化が伝わるかを重視した。

PlanGateの訴求点は削らず、以下を強める方針で3ラウンド実施した。

- AIに実装させる前に人間が判断する場所を作る
- Non-goals / test-cases によってレビュー可能性を高める
- Hook enforcement は最初から全部使わず、小さく始められる

## レビュー視点

| 視点 | 見ること |
| --- | --- |
| チーム開発者 | 自分の現場でも同じ困りごとがあると思えるか |
| EM / スクラムマスター | 導入前後の変化が説明できるか |
| Qiita読者 | 体験談として自然で、最後まで読みやすいか |
| PlanGate利用候補者 | 何から試せばよいか分かるか |

## Round 1: 冒頭と対象読者の明確化

### 改善点

- 冒頭の問題提起は良いが、結論まで少し距離があった。
- PlanGateを使うと何が変わるのかを先に出した方が、Qiita読者には読みやすい。
- 対象読者を明示すると、記事の保存価値が上がる。

### 実施した改善

- タイトルを「チームで使うために」に調整し、過度に強い言い方を避けた。
- `先に結論` を追加。
- 対象読者リストを追加。
- `plan.md` のコードブロックのインデントを整えた。

### 品質評価

| 観点 | 評価 |
| --- | --- |
| 共感導入 | 4.0 → 4.4 |
| 記事の対象明確性 | 3.2 → 4.4 |
| 読み始めやすさ | 3.6 → 4.3 |

## Round 2: 導入前後の変化を明確化

### 改善点

- 記事の中で変化は語られていたが、一覧で見える形ではなかった。
- Qiita読者には「実際にどう変わったのか」の表があると刺さりやすい。
- PlanGateの導入価値を、AIの出力品質ではなく人間側の確認タイミングの変化として整理したい。

### 実施した改善

- `導入前後で変わったこと` を追加。
- 実装開始、スコープ、テスト観点、レビュー、失敗時の比較表を追加。
- 「チームが判断しやすい状態を先に作る」という表現で価値を強化。

### 品質評価

| 観点 | 評価 |
| --- | --- |
| 実務イメージ | 3.7 → 4.6 |
| Qiitaらしい体験談性 | 4.1 → 4.5 |
| PlanGate導入の納得感 | 4.0 → 4.6 |

## Round 3: 読後アクションの強化

### 改善点

- 最後に参考リンクはあるが、読者が明日から試す具体アクションがもう少し欲しい。
- PlanGateを全部導入する前に、依頼文だけ試せる導線があると心理的ハードルが下がる。

### 実施した改善

- `明日から試すなら` を追加。
- まずAIにplanを作らせる依頼文を追加。
- 人間が確認する3点を追加。
- PlanGateリポジトリへの自然な導線を追加。
