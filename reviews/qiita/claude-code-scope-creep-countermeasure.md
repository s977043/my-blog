## Must-fix（公開前に必須）

### 1. 既存公開記事とのカニバリ・相互導線の欠如

`Qiita/public/ai-coding-preflight-checklist.md`（公開予定、`id: null`）が **Goal / Scope / Non-goals / Test / Risks の5項目をタイトルに据えた記事**で、本記事の「対策3: 確認する観点を5つに絞る」「すぐ使える依頼テンプレート」と内容がほぼ重複している。

- L104-120（対策3）の表と、preflight 記事の本体（5項目の各節）が同一テーマ
- L160-186 の依頼テンプレートと、preflight 記事 L144-186 のテンプレートがほぼ同型
- どちらも PlanGate を結論導線にしている

**問題点**: 2 記事が同一検索クエリ（"Non-goals AI" "AIコーディング 計画" 等）で競合し、Qiita 内でカニバる。読者から見ても「同じ著者が同じ話を 2 回している」印象になり保存価値が下がる。

**提案**: 役割分担を明示し、相互リンクを張る。
- 本記事 = 「スコープクリープという**症状**と、その**対策の全体像（5対策）**」
- preflight 記事 = 「5項目フレームの**詳細な書き方リファレンス**」
- 本記事「対策3」末尾（L120 付近）に「5項目それぞれの書き方は別記事で詳しく解説しています」と preflight 記事への導線を追加（公開順が前後する場合は、後発側にのみリンクを張る運用メモを作業メモに記載）。
- 「関連リンク」セクション（L206-210）に Qiita 内の preflight 記事 / `plangate-ai-agent-governance`（id: 5ebff79112ecf1af872c）への内部リンクを追加。Qiita は内部回遊で滞在時間が伸び、検索評価にも寄与する。

### 2. frontmatter `updated_at` と公開フローの申し送り

L10 `updated_at: '2026-05-13T00:00:00+09:00'`。本日時点で過去日。

**問題点**: Qiita は `updated_at` を記事日時として表示する。公開日より古い日付のまま publish すると新着で埋もれる。

**提案（作業メモへ明記）**:
- `ignorePublish: true` / `private: false` / `id: null` は現状維持で正しい（初回 publish で id 採番、公開は来週）。
- 公開当日に `ignorePublish: false` へ変更し、同時に `updated_at` を**公開当日の日時**へ更新する、を作業メモ（L17-23）に追記。

### 3. HTML コメント作業メモ（L17-23）の扱い

L17-23 の作業メモは社内運用情報。Qiita プレビュー/公開時に本文へは描画されないが、Markdown ソースとして残る。

**問題点**: Qiita 記事のソースは公開後も編集画面で閲覧可能。"PlanGate記事群への導線" など社内運用語が残ると体裁が悪い。

**提案**: 公開当日に L17-23 のブロックごと削除する（Must-fix 2 の作業メモ更新と同じタイミング）。これを作業メモ自身に明記しておく。

## Recommended（推奨）

### 4. タイトルの検索キーワード網羅性

L2 `Claude CodeでAIが勝手に実装範囲を広げるときの対策`。

**問題点**: 共感は強いが、検索語「スコープクリープ」「実装範囲 暴走」「AI 勝手にリファクタリング」とのテキスト一致が弱い。slug は `scope-creep` だがタイトルに語が無い。

**提案**: 「スコープクリープ」をタイトルに含める案を検討。
- 例: `Claude Codeが勝手に実装範囲を広げる（スコープクリープ）ときの対策`
- はじめに（L25-43）にも「いわゆるスコープクリープ」と一度言い換えを入れると、本文内の検索ヒット率が上がる。

### 5. PlanGate / River Review 導線の非対称

対策5（L144-158）で PlanGate を本文導線として説明しているが、River Review は L209 の関連リンクに URL があるだけで本文言及がゼロ。

**問題点**: 本文に文脈がないリンクはクリックされにくく、関連リンクの River Review が唐突。

**提案**: いずれか。
- (a) River Review に軽い一文を添える（例: 対策5 付近で「実装後のレビュー側を仕組み化する River Review もあります」）。
- (b) 本記事のスコープ（実装"前"の境界づくり）に River Review（レビュー側）が合わないなら、関連リンクから River Review を外し PlanGate と Growth Lab に絞る。記事の主張（実装前に止める）と整合するのは (b)。

### 6. 冒頭フックの結論到達距離

はじめに L25-43 は箇条書きの具体例が良い一方、「この記事で持ち帰れる5対策」を冒頭で先出ししていない。L43 で「対策をまとめます」止まり。

**問題点**: Qiita 読者は冒頭数行で読む/離脱を判断する。既存 `plangate-ai-agent-governance` レビューでも「先に結論」追加が高評価だった（preflight 記事も同方針）。

**提案**: L43 の後に 5 対策の一覧（まとめ L194-198 と同じリスト）を「先に結論」として配置。詳細は各章で読める構成にすると保存価値が上がる。

### 7. タグの最適化

L3-8: `ClaudeCode` / `生成AI` / `AI駆動開発` / `チーム開発` / `開発プロセス`。

**問題点**: 妥当だが、本記事の固有性（レビュー観点・スコープ管理）が弱い。`コードレビュー` は preflight / plangate 記事で使われており回遊に効くタグ。`開発プロセス` よりフォロワー母数が大きい。

**提案**: `開発プロセス` → `コードレビュー` への差し替えを検討（既存 Qiita 公開記事群とタグを揃えると著者ページ内の関連表示・回遊が強化される）。`ClaudeCode`/`生成AI`/`AI駆動開発`/`チーム開発` の 4 本は維持で良い。
