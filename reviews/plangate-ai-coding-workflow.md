### 該当箇所 1
L22-L36 （「なぜPlanGateにたどり着いたのか」節と冒頭課題の重複）

```markdown
## なぜPlanGateにたどり着いたのか

最初は、プロンプトを工夫すれば改善できると思っていました。
でも実際には、次のような問題が繰り返し起きました。

- スコープが少しずつ広がる
- テストが後付けになる
- 実装後に仕様のズレが見つかる
- 会話が切れると作業状態が消える
```

### 問題点
L9-L14の冒頭の箇条書き（「頼んでいない変更まで広げる／テストを後回しにする／失敗しても別ルートでごまかして進める／セッションが切れると、どこまでやったか分からなくなる」）と、この節の4項目が実質的に同一内容である。読者は冒頭で読んだ内容を再度読まされる形になり、情報密度が低下する。

### 提案
L22-L36の箇条書きは削除し、「最初はプロンプトの工夫で改善できると考えたが、冒頭に挙げた4つの問題は個別の注意では防げず、根本原因はワークフローの未設計だった」と1-2文で接続する構成に差し替える。

```markdown
## なぜPlanGateにたどり着いたのか

最初は、プロンプトを工夫すれば改善できると思っていました。
しかし冒頭に挙げた4つの問題は、個別の注意では防ぎきれません。
根本原因は、AIの能力不足というより**ワークフローが未設計なこと**でした。
```

```markdown
背景や問題意識をもう少し長く共有した上で、`PBI -> 計画 -> レビュー -> 実行` の流れをどう設計するか、そのまま使えるテンプレートと運用手順をまとめます。
```

### 該当箇所 3
L64-L73 （ディレクトリ構成のコードブロック）

```text
docs/working/TASK-XXXX/
├── pbi-input.md
├── plan.md
├── todo.md
├── test-cases.md
├── review-self.md
├── review-external.md
└── status.md
```

### 問題点
PBI INPUT PACKAGE節（L78以降）で提示している見出しは `# PBI INPUT PACKAGE` だが、ファイル名は `pbi-input.md` と表記されている。一方、記事中で後述するテンプレート言及では "PBI INPUT PACKAGE のテンプレート" と書かれており、ファイル名と論理名の対応関係が初出時に明示されていない。読者が「pbi-input.md = PBI INPUT PACKAGE」である点を推測する必要がある。

### 提案
ディレクトリ構成直後、またはL78の節冒頭で以下のような注記を1行加える。

```markdown
## PBI INPUT PACKAGE

`pbi-input.md` に記録するテンプレートです。AIにいきなり実装させず、まず人間が入力を揃えます。
```

### 該当箇所 4
L131-L155 （plan / todo / test-cases の合体コードブロック）

```markdown
# plan.md

- 変更の目的
- 実施ステップ
- 依存関係
- リスク

# todo.md

- [ ] 1. 仕様確認
- [ ] 2. 実装
- [ ] 3. テスト
- [ ] 4. レビュー

# test-cases.md

- Case 1: 正常系
- Case 2: 異常系
- Case 3: 境界値
```

### 問題点
3つの異なるファイル（`plan.md` / `todo.md` / `test-cases.md`）の内容が単一のMarkdownコードブロックに連結されている。`#` で始まる行が3つ連続する形になり、実際のファイル運用時にコピーしてそのまま使えない（どこからどこまでが1ファイルかが不明瞭）。また `# plan.md` のような「ファイル名そのものをH1見出しに使う」書き方は、実際のドキュメントとしては推奨されない（Zennプレビューでも単一の大きな見出しが連続する形で描画される）。

### 提案
ファイルごとにコードブロックを分離し、各ブロックの上にファイル名をテキストで明示する。

````markdown
**plan.md**

```markdown
# 変更概要

- 変更の目的
- 実施ステップ
- 依存関係
- リスク
```

**todo.md**

```markdown
- [ ] 1. 仕様確認
- [ ] 2. 実装
- [ ] 3. テスト
- [ ] 4. レビュー
```

**test-cases.md**

```markdown
- Case 1: 正常系
- Case 2: 異常系
- Case 3: 境界値
```
````

### 該当箇所 5
L162-L180 （レビュー用テンプレートのフロー説明）

```markdown
セルフレビューと外部レビューは、同じ観点を別々の視点で見るために置いています。
`plan` を作った直後に `review-self.md` を作り、`exec` に進む前に `review-external.md` を通すと、承認前の見落としを減らしやすくなります。

...

人間が見るのは `review-self.md` ではなく、`plan.md` と `review-external.md` を通した後です。
```

### 問題点
L180の文「人間が見るのは `review-self.md` ではなく、`plan.md` と `review-external.md` を通した後です。」が読みづらい。主語・目的語・時点が入り混じっており、「人間がレビューするタイミング」を述べたいのか「人間が何を見るか」を述べたいのかが曖昧。L162で説明した「セルフは先、外部は次、最後に人間」という流れと、L180の記述が呼応していない。

### 提案
```markdown
セルフレビューと外部レビューは、同じ観点を別々の視点で見るために置いています。
`plan` を作った直後に `review-self.md` を作り、`exec` に進む前に `review-external.md` を通すと、承認前の見落としを減らしやすくなります。

人間が最終レビューを行うのは、`review-self.md` と `review-external.md` の両方が出揃い、`plan.md` と合わせて確認できる状態になってからです。
```

### 該当箇所 6
L200-L214 （3コマンド運用のコマンド例）

```bash
# 1. 作業コンテキスト作成
/working-context TASK-XXXX

# 2. Plan + ToDo + Test Cases生成
/ai-dev-workflow TASK-XXXX plan

# 3. 承認後に実行
/ai-dev-workflow TASK-XXXX exec
```

### 問題点
これらのコマンドは Claude Code などのスラッシュコマンドを前提としているが、記事本文ではどのエージェント／CLI 上で実行するのかが明示されていない。初見の読者はこれが bash コマンドなのか特定ツールのコマンドなのかを判断できない。コードブロックの言語指定が `bash` だが、`/working-context` は bash のコマンドではない（シバン扱いされる可能性もある）。

### 提案
言語指定を `text` に変更し、直前に前提ツールを明示する。

````markdown
実際の操作はこの3つに絞ると運用しやすいです。
以下は Claude Code のスラッシュコマンド（`.claude/commands/` に配置）として実行する想定です。

```text
# 1. 作業コンテキスト作成
/working-context TASK-XXXX

# 2. Plan + ToDo + Test Cases生成
/ai-dev-workflow TASK-XXXX plan

# 3. 承認後に実行
/ai-dev-workflow TASK-XXXX exec
```
````

### 該当箇所 7
L229-L234 （「この記事で持ち帰れること」節）

```markdown
## この記事で持ち帰れること

- AI 駆動開発をアジャイルに載せるときは、自由度より先に制御点を設計する
- PBI の情報は、そのまま AI 実行の入力と承認基準に変換できる
- PlanGate は、既存のスクラム運用を壊さずに AI 駆動開発を入れるための実装例として使える
```

### 問題点
L54-L58「この記事で扱うもの」と、L229-L234「この記事で持ち帰れること」は、Zenn記事の構成として役割が重複しがち。前者は目次的、後者はまとめ的であるが、Zennの標準的な記事構成では「TL;DR → 本文 → まとめ」の3段構成が一般的で、冒頭と末尾それぞれで要点を並べる形になっていない。読者が記事全体の骨子を掴みにくい。

### 提案
L54-L58を削除して「この記事で持ち帰れること（=まとめ）」のみ残すか、逆に冒頭に TL;DR ブロックを設けて本記事の結論を先出しする。SEO 観点でも、記事冒頭の 150-200 文字以内に結論・読者利益を示すと、検索結果のスニペット表示に有利。

```markdown
:::message
**この記事で持ち帰れること**

- AI駆動開発をアジャイルに載せるときは、自由度より先に制御点を設計する
- PBI の情報は、そのまま AI 実行の入力と承認基準に変換できる
- PlanGate は既存のスクラム運用を壊さず AI 駆動開発を組み込む実装例として使える
:::
```
を冒頭近く（L19付近）に配置し、L229-L234は純粋な「まとめ」に書き換える。

### 該当箇所 8
L19 （キーワード「ハーネスエンジニアリング」の初出）

```markdown
自分はこの発想を `ハーネスエンジニアリング` と捉えています。そして、その考えをフローにしたのが PlanGate です。
```

### 問題点
「ハーネスエンジニアリング」は筆者の造語もしくは業界用語だが、定義が示されないまま記事全体で繰り返し登場する（L47にも「AIコーディングに対するハーネスエンジニアリングの一例」と出現）。初出時にインラインコードで囲む形は視認性は出るが、用語定義にはなっていない。SEO観点では、独自用語を記事の前半で明確に定義すると、その用語名での検索流入が狙いやすくなる。

### 提案
L19-L20付近で1-2文の定義を挿入する。

```markdown
自分はこの発想を **ハーネスエンジニアリング**（AIを自由に走らせる前に、止まる場所・確認する場所・自動で進める範囲を先に設計する考え方）と呼んでいます。そして、その考えをフローにしたのが PlanGate です。
```

## 総合評価

### 良い点
- **明確な一文サマリ**: 「計画を承認しないとAIは1行もコードを書けない」という要約が強く、記事の主張が伝わる
- **ディレクトリ構成が具体的**: `docs/working/TASK-XXXX/` 配下のファイル群が明示され、そのまま真似できる
- **アジャイル／スクラムとの接続**: PBI・受入基準・見積もり根拠など、既存運用との橋渡しが意識されている
- **関連記事・自社メディアへの導線**: 記事末尾で内部リンクが整理されている

### 改善点
- **冒頭と章中盤で内容の重複がある**（L9-L14 と L22-L36）
- **テンプレートのコードブロック構造**: 複数ファイルを1ブロックに詰めており、コピペ運用しづらい
- **独自用語「ハーネスエンジニアリング」の定義不足**
- **3コマンド運用のコマンド例の前提ツールが不明瞭**

### 推奨アクション
1. **重複削減**: L22-L36の箇条書きを冒頭と呼応させる形でリファクタ
2. **テンプレート分割**: plan/todo/test-cases のコードブロックを個別に分離
3. **用語定義の追加**: ハーネスエンジニアリングの定義を初出直後に明記
4. **コマンド前提の明示**: 3コマンド運用が Claude Code のスラッシュコマンドである点を注記

### SEO観点での改善提案
- **タイトル**: 「PlanGate」は造語のため単体での検索流入は弱い。「AI駆動開発 テンプレート」「AIコーディング ワークフロー」など一般検索クエリをサブタイトル的にタイトル後半に含めると流入が増えやすい
- **冒頭200文字のリード文**: 現行は問題提起で始まるため、検索スニペット上で「この記事で何が得られるか」が見えにくい。TL;DRや「この記事で持ち帰れること」を冒頭近くに配置する
- **独自用語の定義**: 「ハーネスエンジニアリング」の定義文を記事前半に置くと、そのクエリでの検索流入・被リンクが期待できる
- **内部リンクのアンカーテキスト**: 関連記事セクションのアンカーテキストが記事タイトルそのままで長い。「仕様駆動開発の3原則（関連記事）」のように短縮した形も併記するとクリック率が上がりやすい
