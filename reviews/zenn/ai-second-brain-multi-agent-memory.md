### 該当箇所 1
L186-L189 （既存記事との差別化・内部リンクが弱い／重要度: 高）

```
## 参考

- 関連: [AIエージェントの『進めて』問題：自走と確認の境界をmemoryで永続化する](...)
- 関連: [推測で書いた学びが次の罠になった — AGENT_LEARNINGS.md の運用設計](...)
```

### 問題点
同一著者の `obsidian-supermemory-mcp.md`（Obsidian × 共有メモリの比較記事）が公開済みにもかかわらず、本記事から一切参照されていない。両記事は「Obsidian を AI の記憶基盤にする」という重複テーマを扱うため、読者・検索エンジンの双方が「Supermemory MCP 案（外部サービス・横断共有）」と「本記事の自前 Vault 正本案（指示ベース集約・Hermes 司令塔）」の違いを判別できない。差別化の明示は本タスクの主要観点であり、内部リンクが無いと SEO 上もカニバリ（共食い）リスクがある。

### 提案
```
## 参考

- 関連: [ObsidianとSupermemory MCPをつなぐ知識管理ワークフロー](https://zenn.dev/minewo/articles/obsidian-supermemory-mcp) - 外部 MCP サービスで横断共有する案。本記事は「自前 Vault を正本に固定し指示ベースで集約する」点が異なる
- 関連: [AIエージェントの『進めて』問題：...](...)
- 関連: [推測で書いた学びが次の罠になった — ...](...)
```
あわせて本文（例: L120 付近「ツールごとの接続方法」節の冒頭）に「MCP サーバで横断共有する手もあるが、本記事は MCP に依存せず Markdown を正本に置く構成を取る」と一文足すと差別化が前段で立つ。

### 該当箇所 3
L122 （技術的正確性: `autoMemoryDirectory 相当` の設定名が曖昧／重要度: 高）

```
- **Claude Code**: 公式設定（autoMemoryDirectory 相当）で、メモリーディレクトリを自動接続できる
```

### 問題点
`autoMemoryDirectory` という設定名が「相当」と濁されており、Claude Code の公式設定キーとして存在するか本文から確証が取れない。エンジニア読者は設定ファイルにこの名前で書こうとして失敗する恐れがある。技術記載の正確性に直結し、誤用なら読者の実装可能性を損なう。

### 提案
正確なキー名を確認し before/after で確定させる。実在しない／名称が異なる場合は「相当」表現を避け、実際の設定方法（例: `CLAUDE.md` への記載 or 設定ファイルの該当項目名）に置換する。
```
# before
- **Claude Code**: 公式設定（autoMemoryDirectory 相当）で、メモリーディレクトリを自動接続できる
# after（例・要検証）
- **Claude Code**: メモリーディレクトリを設定で指定でき、起動時に自動で読み込める（具体的な設定キーは公式ドキュメント参照）
```
公式ドキュメントで裏取りできない場合は、断定を避けた表現にする。

### 該当箇所 5
L26-L34, L177-L182 （導入と結論の重複・主張の三度出し／重要度: 中）

```
（L30-34 先に結論：ポイント3つ）
記憶の正本を 08 Agent Context/memory/<repo>/ に固定し...
（L179-182 まとめ：ほぼ同一の3点）
記憶の正本は 08 Agent Context/memory/<repo>/。人間もAIも同じ Markdown を見る
```

### 問題点
「先に結論」（L30-34）と「まとめ」（L179-182）の 3 点がほぼ逐語的に同一。さらに L177 でも同じ主張が繰り返され、結論→本文→まとめで主張が 3 回反復している。冒頭の先出しは有効だが、まとめが冒頭のコピーだと読後の持ち帰り感が薄れる。削除候補の判定基準「同じ主張が 3 回以上」に該当。

### 提案
まとめ側は冒頭と差分を付け、「運用して分かったこと」寄りに振る。例: 各行に冒頭には無い一次経験（「淘汰前提にしたら書くハードルが下がった」等）を足し、冒頭と語彙を変える。完全な箇条書きの再掲は片方を要約文に圧縮する。


「昇格」という比喩がセクションタイトルと本文・まとめで多用されるが、初出時に「何が昇格なのか（Inbox の雑メモ → 長期記憶への格上げ）」の言い換えが弱く、やや硬い書き言葉として narrative に浮く（観点 8）。読者が「昇格＝具体的に何の操作か」を掴むのにワンクッション要る。

## 総合評価

### 良い点
- 「先に結論」→全体像（Think→Capture→Organize→Publish→Reuse 図）→詳細→運用→つまずき→まとめ、と価値を前に出す構成が一貫しており、Zenn 読みやすさ観点 1・2・6・7 を満たしている。
- 「記憶の分断は保存先を減らすことではなく正本を1か所に決めること」という主張が冒頭・本文・まとめで芯として通り、タイトルと締めが呼応している。
- Role 表（L145-151）が並列情報を table 化しており観点 11 に適合。長尺 bash や PR テンプレートが無く、narrative 優先で仕様書感が薄い。
- 関連リンク 2 本は WebFetch で存在を確認済み（リンク切れなし）。

### 改善点
- 中心概念 Hermes が未定義（指摘2）と、既存 Supermemory 記事との差別化リンク欠落（指摘1）が最優先。この 2 点で記事の独自価値が読者に伝わりきらない。
- `autoMemoryDirectory` の設定名（指摘3）は技術的正確性の要。公式裏取りが取れなければ断定を避ける。
- 対象ツール一覧と実行者リストの不整合（Grok・Gemini、指摘7・8）で世界観がぶれる。

### 推奨アクション
1. 指摘1（差別化リンク）・指摘2（Hermes 定義）を本文に反映 — 記事の独自性に直結。
2. 指摘3（設定名）を公式ドキュメントで検証し、確証なければ表現を緩める。
3. 指摘4・6（PARA/carryover/昇格の言い換え）と指摘5（結論とまとめの重複圧縮）で narrative を整える。
4. 指摘7・8（ツール整合）を冒頭一覧と表で揃える。

### SEO観点での改善提案
- 既存 `obsidian-supermemory-mcp.md` とのキーワード共食い回避のため、本記事は「multi-agent memory」「司令塔」「自前 Vault 正本」を軸に、相互内部リンクで棲み分けを明示する（指摘1）。
- `type: "idea"` のため検索流入は限定的だが、想定読者見出しの語（Claude Code / Codex CLI / Gemini CLI 併用）が topics と一致しており良好。タイトル候補（L10-13）は公開前に 1 つへ確定する旨が記載済みで運用上問題なし。
- 「Hermes」は一般語のため単独では検索意図が拾えない。h2 見出し（L129）に「役割分担」など補助語が入っており可検索性は確保されている。
