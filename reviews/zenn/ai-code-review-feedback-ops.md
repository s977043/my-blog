```
## はじめに

## TL;DR

- AIレビューが不安定なのは、モデルの賢さより評価基準の不足が大きい
- まずは `Must Fix / Should Fix / Nice to Have` の3分類を決める
```

### 問題点

`:::message`（L9-L15）の「この記事で得られること」と TL;DR（L36-L41）がほぼ同じ主張を繰り返している。さらに「はじめに」（L17-L34）も同じ3点を課題提起として述べており、読者は本題に入るまでに同じ内容を3回読む構造になっている。

### 提案

`:::message` を「想定読者・前提条件」用途に切り替え、結論は TL;DR に集約する。

```
:::message
**想定読者**
- AIコードレビュー（Claude Code / GitHub Copilot 等）を導入済みで、安定性や精度に課題を感じているチーム

- AIレビューの不安定さは評価基準の不足が主因（モデルの問題ではない）
- `Must Fix / Should Fix / Nice to Have` の3分類から始める
- 繰り返し出るズレを記録し、3回出たらルールまたはCIに昇格させる
- CIのpromptに観点・除外範囲・出力形式を固定すると品質が揃いやすい
```

```
## 5. CIに組み込むと、レビュー観点を毎回そろえやすい

ここまでの話を実運用に落とすと、レビュー基準は手元のメモだけでなく、CIから呼ばれるAIレビューの prompt にも入れたほうが安定します。

自分たちの運用でも、GitHub Actions から AIレビューを走らせるときに、単に「このPRをレビューして」と投げるのではなく、レビュー手順と観点を prompt に固定するようにしました。

### 先にPR状態とCI状態を確認する
```

### 問題点

第5章はこの記事で最も情報量が多い章（5サブセクション・60行）だが、章に入る前に「何を何ステップで説明するか」の全体像が示されていない。読者は読み進めながら構造を把握し直す必要があり、認知負荷が高い。Zenn 読みやすさ観点 2（詳細群の前に全体像を提示）の違反。

```
1. PR状態とCI状態を先に確認する
2. 変更種別に応じた review_focus（重点観点）を切り替える
3. レビューを Upstream / Midstream / Downstream の3フェーズに分ける
4. 静的検査で拾える問題はAIに指摘させない
5. 出力形式（スコア・severity順・件数上限）を固定する

それぞれの詳細を説明します。
```

```
### 変更内容に応じて review focus を切り替える

変更コンテキストから `review_focus` を渡して、重点観点を切り替えます。
```

### 問題点

`review_focus` が prompt の変数名なのか概念名なのかが初出時点で不明。また変更種別と重点観点の対応は並列・対称な情報であり、箇条書きより表のほうが一覧性が高い。Zenn 読みやすさ観点 5（英語ラベルへの日本語副題）・観点 11（並列情報の table 化）の違反。

```
### 変更内容に応じて review_focus（重点観点）を切り替える

| 変更種別 | review_focus の重点観点 |
| -------- | ----------------------- |
| migration を含む | 後方互換性 |
| API変更を含む | スキーマ互換性 |
| shared-components 変更 | cross-site impact |
| 認証変更 | auth / security |
| フロント変更 | stale state / hydration / responsive |
```

### 該当箇所 5
L163-L166 （コマンド断片がインラインのまま、返却値の説明がない）

```
- `gh pr view --json mergeable,mergeStateStatus` でコンフリクト確認
- `gh pr checks` で CI 状態を確認
```

### 問題点

コマンドがインラインコードで箇条書きに埋め込まれており、コードブロックとして整形されていない。`--json mergeable,mergeStateStatus` が何を返すか（`CONFLICTING` などの値）が示されておらず、読者が実際に使おうとしたときに判定ロジックが不明。Webエンジニア読者視点での再現性の問題。

### 提案

```bash
# コンフリクト確認（mergeable: CONFLICTING なら人間が対処してからレビュー）
gh pr view --json mergeable,mergeStateStatus

# CI 状態確認（failed があればコードレビュー前に対処）
gh pr checks
```

### 該当箇所 6
L200-L212 （`minor/info` の件数上限と `auto-approve` の適用基準が曖昧）

```
指摘は `critical / major / minor / info` の severity 順に並べ、`minor` と `info` は合計件数も絞ります。
この制約がないと、AIは「たくさん言うほど良いレビュー」になりがちです。

スコア運用そのものが正解とは限りません。
ただ、`auto-approve / human-review-recommended / human-review-required` のように出口を固定すると、レビュー結果を人間が扱いやすくなります。
```

### 問題点

「合計件数も絞ります」とあるが件数の目安がない。また `auto-approve / human-review-recommended / human-review-required` はこの箇所が初出で、どのseverityがどの出口に対応するかが示されていない。読者が自チームに導入する際に判断できる具体情報が不足している。

### 提案

```
指摘は `critical / major / minor / info` の severity 順に並べ、`minor` と `info` は合計5件以内を目安に絞ります。

出口の判定も3種類に固定しています。

| 判定 | 適用条件 |
| ---- | -------- |
| `auto-approve` | critical / major 指摘なし。自動マージ可 |
| `human-review-recommended` | major 指摘あり。人間が確認推奨 |
| `human-review-required` | critical 指摘あり。必ず人間がレビュー |
```

### 該当箇所 7
L275-L283 （まとめが「はじめに」と TL;DR の再掲になっている）

```
## まとめ

AIコードレビューが不安定に見えるのは、モデルが信用できないからではありません。
レビュー基準、重要度の分類、採否の記録が仕組みとして固定されていないからです。

まずは `Must Fix / Should Fix / Nice to Have` の3分類から始めて、同じズレが繰り返し出たらルールやCIに戻す。
さらに、CIから呼ぶAIレビュー prompt に観点、除外範囲、出力形式まで固定すると、毎回のレビュー品質をかなり揃えやすくなります。
```

### 問題点

まとめの内容が「はじめに」（L27-L34）と TL;DR（L36-L41）で述べた内容の再掲に留まっている。「まとめ」としての新たな整理（読後に何をするか）がなく、タイトルと呼応した締めの一文もない。Zenn 読みやすさ観点 7（まとめ前の主張再掲・タイトル呼応）に照らすと、再掲はするが「行動への橋渡し」が欠けている。

### 提案

読後の行動ベースの締めに切り替える。

```
## まとめ

運用として始めやすい順序を整理します。

1. **今週**: `Must Fix / Should Fix / Nice to Have` の3分類をチームで合意する
2. **来週以降**: 繰り返し出るズレを記録し、3回出たらルール文書または CI に昇格させる
3. **余裕が出たら**: CI から呼ぶ AI レビュー prompt に観点・除外範囲・出力形式を固定する

AIレビューを安定させるのはモデルの改善ではなく、チーム側の基準を育てることです。
仕組みとして回し始めると、AIレビューは単発のコメント生成から、チームの知識を育てる工程に変わります。
```

## 総合評価

### 良い点
- 課題提起（L19-L33）が具体例（`useMemo` とReact Compiler方針）を使って明快で、読者の「あるある」を的確に捉えている
- `Must Fix / Should Fix / Nice to Have` の3分類は読者がすぐ実践できる粒度であり、記事の核として機能している
- 第6章の記録テーブル（L219-L225）は実運用で参照できる形式で、情報密度と読みやすさのバランスが良い
- 第7章「改善ループは小さく回す」が「3回出たら仕組み化」という明確な判断基準を提示しており、読後の行動につながりやすい
- 関連記事リンク（L285-L289）が同著者のシリーズ記事を内部リンクとして機能させており、読者の回遊とSEOの両面で効果的

### 改善点
- 冒頭で `:::message`・はじめに・TL;DR の3箇所が同じ主張を繰り返しており、本題に入るまでのコストが高い
- 第5章（CI連携）が最も情報量が多い章にもかかわらず、章全体の全体像が示されないまま各サブセクションが並んでいる
- 英語ラベル（`Upstream/Midstream/Downstream`、`review_focus`、`auto-approve` など）が初出時に日本語説明なしで登場している
- `minor/info` の件数制約や `auto-approve` の適用基準など、読者が自チームに導入する際に必要な具体的数値が欠けている
- まとめが「はじめに」と TL;DR の再掲に留まり、読後の行動指針が示されていない

### 推奨アクション
1. `:::message`（L9-L15）を「想定読者・前提条件」に書き換え、TL;DR（L36-L41）に結論を集約する（指摘 1）
2. 第5章冒頭（L155直後）に5ステップの番号付き一覧を追加する（指摘 2）
3. `review_focus`（L169）と `Upstream/Midstream/Downstream`（L182）に日本語副題を付け、前者は表に変換する（指摘 3・4）
4. コマンド断片（L163-L166）をコードブロックに切り出し、返却値の判定ロジックを添える（指摘 5）
5. まとめ（L275-L283）を行動ベースの締めに書き換える（指摘 7）

### SEO観点での改善提案
- タイトルは具体的で検索意図に合っているが、`topics` に `githubactions` を追加するとCI連携記事としてのクロール対象が広がる（現状は `claudecode` があるが記事本文との表記揺れあり：本文は `Claude Code`）
- 第5章の見出し「CIに組み込むと、レビュー観点を毎回そろえやすい」は検索クエリとして機能しにくい。「GitHub Actions でAIコードレビューを自動化する方法」など検索意図に近い表現に変更するとインデックス効果が上がる
- `Must Fix / Should Fix / Nice to Have` というフレーズは記事内に複数回登場しており、このフレーズを検索した読者の流入が期待できる。現在の h2 見出しは「3. 指摘は3分類で扱う」なので、このラベルを見出しに含める形（例: 「3. Must Fix / Should Fix / Nice to Have で指摘を3分類する」）に変更すると検索露出が上がる
