### 該当箇所 5
L111-L116 （grep パターンの `-r` 挙動と `scripts/` 前提への補足不足）

```bash
# LLM 呼び出しの痕跡を幅広く探す（Claude CLI / Anthropic SDK / OpenAI SDK / Gemini）
grep -rEi \
  "claude +-p|curl.*(anthropic|openai|generativelanguage)|anthropic-ai/sdk|import +Anthropic|from +anthropic|import +openai|from +openai|google-generativeai" \
  scripts/
```

### 問題点

前回レビューの指摘を受けて grep パターンが大幅に改善されているが、実運用でコピペされる際に **3 つ残課題**がある。

1. **`scripts/` ディレクトリ固定で、多くのスキルが持つ `SKILL.md` 本体とルート直下のヘルパースクリプトを走査しない**。`engineer-expertise-extractor` の構成（L34-L38）では `scripts/` 配下のみだが、一般的な Claude Code スキルでは SKILL.md 内に bash ブロックや Python スニペットを埋め込むパターンも存在する。`grep -rEi "..." .`（カレント直下全走査）のほうが、取りこぼしが減る。
2. **`claude +-p` のパターンで `claude --print` を検出できない**。Claude CLI は `-p` と `--print` が同義の短/長オプションだが、長オプション形式は拾えない。`claude\s+(-p|--print)` にすると両対応になる。
3. **検出ゼロの場合の「次にすること」が明記されていない**。L118 は「分析系の claim は自動で疑う」で終わっており、疑ったあとどう対応するか（対応策 A/B/C のどれに進むか）への導線が無い。セルフチェックとしては「検出ゼロ → 疑う → 対応策 C を検討する」まで一文で繋ぐほうが、運用手順としての完結性が出る。

### 提案

```bash
# LLM 呼び出しの痕跡を幅広く探す（Claude CLI / Anthropic SDK / OpenAI SDK / Gemini）
# スキルのルート直下すべてを走査する（SKILL.md 内の埋め込みスクリプトも拾う）
grep -rEi \
  "claude\s+(-p|--print)|curl.*(anthropic|openai|generativelanguage)|anthropic-ai/sdk|import\s+Anthropic|from\s+anthropic|import\s+openai|from\s+openai|google-generativeai" \
  .
```

```markdown
（L118 Before）
スクリプト内で `curl` や `anthropic` / `openai` の SDK 呼び出しが **無い場合**、分析系の claim は自動で疑う。

（After）
スクリプト内で `curl` や `anthropic` / `openai` の SDK 呼び出しが **無い場合**、分析系の claim は自動で疑います。検出ゼロなら、後述の対応策 C（ハイブリッド運用として再定義）の検討に進みます。
```

**優先度: Medium**（セルフチェック手順の完結性、実運用での取りこぼし）

### 該当箇所 7
L40 （「壮大」の根拠が箇条書きではなく平文で流されている）

```markdown
SKILL.md は壮大でした。命名規則の抽出、パターン認識、アーキテクチャ判断の分析、レビューフィードバックのスタイル解析。全部できそうに書いてあります。
```

### 問題点

タイトルに「壮大な SKILL.md」と謳っている以上、**読者は「どれくらい壮大か」を期待して本文に入ってくる**。L40 は 4 機能を読点区切りの平文で列挙しているが、

1. 「壮大さ」が視覚的に伝わらない（箇条書きや引用ブロックで見せるほうが訴求が強い）
2. 直前の L23-L24 の引用ブロック（「GitHub 貢献を分析して…」）と**似た役割の文**が連続し、読者に「すでに読んだ」感を与える
3. L40 の 4 機能と L86-L89 の「動詞トレース例」4 機能（抽出する / 分析する / 判定する / 生成する）を**対応させる**と、記事全体の伏線回収になるが、現状は対応関係が平文で埋もれている

### 提案

L40 を箇条書きに展開し、後の動詞トレースに呼応させる。

```markdown
（Before）
SKILL.md は壮大でした。命名規則の抽出、パターン認識、アーキテクチャ判断の分析、レビューフィードバックのスタイル解析。全部できそうに書いてあります。

（After）
SKILL.md は壮大でした。章立てを拾うと:

- 命名規則の **抽出**
- コーディングパターンの **認識**
- アーキテクチャ判断の **分析**
- レビューフィードバックのスタイル **解析**

…と、分析系の動詞が並びます。全部できそうに書いてありました（これらの動詞は、後で実装とトレースする対象になります）。
```

こうしておくと、L80-L89 の「動詞トレース」章で「先ほどの 4 動詞を 1 つずつ見てみると」と伏線回収できる。

**優先度: Low**（ナラティブの強化、なくても記事は成立する）

### 該当箇所 8
L64-L74 （仮説 3 の記述が薄い / 対称性の欠如）

```markdown
### 仮説 1: SKILL.md は LLM の「ベストケース表現」

AI に「このスキルを説明して」と頼むと、機能の **理想像** を書きがちです。「できること」を並べる一方、「実装の制約」や「LLM なしで動く範囲」は抑えめに書かれる。

### 仮説 2: スクリプトは bash の表現力の範囲で書かれる

一方、実装を bash で書けば、LLM 呼び出しなしでは**分析的な処理**（パターン認識や文体判定など）はできません。結果として、「データ取得 + 静的テンプレ」という 2 段構成に落ち着きます。

### 仮説 3: 設計者も SKILL.md と実装が一致しているか確認していない

AI が生成した SKILL.md を読み、AI が生成したスクリプトを採用して、誰もシンクさせていない。
```

### 問題点

仮説 1 と仮説 2 は **2〜3 文** で「なぜそうなるか」のメカニズムを説明しているが、**仮説 3 は 1 文のみ**で、対称性が崩れている。また、仮説 3 は本記事の主張の核（「AI 同士の連鎖でズレが増幅する」）に最も近い仮説で、L152 の「エージェントが自分で作ったスキルを別のエージェントが使う連鎖」とも呼応する重要な論点。それにも関わらず本仮説の記述が最薄で、メカニズムの説明（なぜ確認しないのか、どこで確認機会が失われるのか）が抜けている。

また、仮説 3 の「シンク」はカタカナ技術用語（sync の日本語化）で、本記事の他の和語表現と比べると唐突。

### 提案

仮説 3 に 1-2 文追加して対称性を取り、「シンク」を日本語に置換する。

```markdown
（Before）
### 仮説 3: 設計者も SKILL.md と実装が一致しているか確認していない

AI が生成した SKILL.md を読み、AI が生成したスクリプトを採用して、誰もシンクさせていない。

（After）
### 仮説 3: 設計者も SKILL.md と実装の整合を確認していない

AI が生成した SKILL.md を読み、AI が生成したスクリプトを採用する流れでは、**人間が両者の整合を取る工程**が抜け落ちがちです。SKILL.md は「これから作るもの」、スクリプトは「作ったもの」として別タイミングで生成されるため、最終的に両者を並べてズレを潰すステップが誰にも割り当てられないまま運用に入る、というのが実際によく起きるパターンです。
```

**優先度: Low**（記述の対称性、主題への寄与が大きい仮説の肉付け）

- **物語構造が明瞭**: 「発見（L29）→ 仮説（L60）→ 検知（L76）→ 対応（L118）→ 付き合い方（L143）→ まとめ（L154）」の 6 段構成で、読者の思考の流れに沿った順接。
- **対応策 A/B/C の採否まで踏み込む**: 「採用したのは C」（L138-L141）で著者自身の意思決定を共有しており、idea 型記事として読後にアクションが取りやすい。
- **grep パターンの刷新**: 前回指摘を反映して `claude -p`・Anthropic SDK・Gemini まで含む幅広いパターンに改訂。Zenn 読者（実装者）が自スキルに流用しやすい実用性のあるスニペット。

### 改善点

- **表記の微妙な揺れ**: `LLM側`（L140）と `LLM 呼び出し`（L70 他）のスペース有無、`claim` 初出時の日本語補足が L12 のみで L58 等に再掲がない（指摘 1）。
- **スクリプト抜粋の出所が地の文で曖昧**: 「grep してみると」→ heredoc 全体が続く構造で、何を見せているブロックかの明示が弱い（指摘 2）。
- **「付き合い方」と「まとめ」の重複**: 主張 4 点が 2 ブロック並ぶ構造で、読後の持ち帰り点が 8 点と多い（指摘 3）。
- **硬い表現の残存**: 「1 次情報に戻って確認する癖」が L152 と L161 で 2 回登場、「AI 協働時代」の抽象化（指摘 4）。
- **grep の対象ディレクトリ固定と `--print` 取りこぼし**: `scripts/` 固定・短オプションのみで、実運用での漏れが残る（指摘 5）。
- **スキルの素性が本文に明示されていない**: タイトルで「AI に書かせた」と断言しているが、本文内で根拠が示されず読者が前提を組み立てにくい（指摘 6）。

### 推奨アクション

1. **Medium 優先度 5 件の対応**: 表記揺れ統一（指摘 1）、スクリプト抜粋の区分明示（指摘 2）、付き合い方とまとめの重複圧縮（指摘 3）、硬い表現の柔らかい言い換え（指摘 4）、grep の走査範囲と `--print` 対応（指摘 5）、スキル素性の 1 行補足（指摘 6）。いずれも本文の品質を一段上げる修正で、公開後の追補として段階的に入れるのが現実的。
2. **Low 優先度 2 件の対応**: 「壮大」の箇条書き展開（指摘 7）、仮説 3 の肉付け（指摘 8）。物語の強度を上げるナラティブ改善だが、現状でも記事は成立する。
3. **双方向リンクの追加**: 関連記事セクション（L163-L167）は追加済みなので、姉妹記事側（`engineering-process-improvement-skill.md` / `ai-legible-repository-design.md` / `design-md-guide-and-adoption-log.md`）にも本記事へのリンクを追記すると、回遊率が倍増する。

### SEO観点での改善提案

- **タイトル**: 現状「AIに書かせた壮大なSKILL.mdを実装したら、ただのテンプレ生成器だった話」は 32 文字で、Zenn のタイトル長・釣り感・具体語のバランスが良好。「AIに書かせた」「SKILL.md」「テンプレ生成器」が検索意図を捕捉しており、変更不要。
- **topics**: `["ai駆動開発", "claudecode", "スキル設計", "ドキュメント", "ai生成"]` は記事内容と整合。`ai駆動開発` と `ai生成` がやや重複する印象はあるが、別の検索ワードとして機能するため維持可。強いて差し替え候補を挙げるなら `ai生成` → `プロンプトエンジニアリング` や `ai-sdlc` で、姉妹記事群との topic クラスタを強められる。
- **内部リンク**: 関連記事リンク 3 件（L163-L167）は導線として十分機能。双方向化（姉妹記事側にも本記事を追記）で回遊率を底上げ可能。
- **h2/h3 階層**: 「発見 → 仮説 → 検知 → 対応策 → 付き合い方 → まとめ」の h2 構造は目次として機能しており、流し読み読者が射程を把握しやすい。
- **導入フック**: L21「胸が熱くなりました」→ L25「実装は静的テンプレ生成器だった」のギャップ提示は idea 型記事のフックとして強く、ファーストビューの離脱を抑えている。`:::message` → 引用ブロック（L23）→ 落差 → 本題の導入順も整っている。
- **キーワード密度**: 本文中に `SKILL.md` が 14 回、`claim` が 4 回、`LLM` が 6 回、`ハイブリッド運用` が 4 回登場しており、記事の主題キーワードが十分に散りばめられている。冒頭 `:::message` でも全キーワードが登場しており、SEO 的に良好。
