### 該当箇所 5
L108 （LLM 呼び出しの有無を grep するパターン）

```bash
grep -rE "curl.*anthropic|curl.*openai|Anthropic\(\)|openai\." scripts/
```

### 問題点
この regex は「LLM 呼び出しの有無」を確認するセルフチェック手段として提示されているが、以下 3 つの実運用上の漏れがある。

1. **Claude Code 固有の呼び出し方が漏れている**。Claude Code スキルの文脈で最もメジャーな LLM 呼び出しは `claude` CLI（例: `claude -p "..."`）だが、パターンに含まれていない。Anthropic 公式 SDK 以外で Claude を叩く経路を取りこぼす。
2. **`@anthropic-ai/sdk` や `google-generativeai` など他 SDK の取りこぼし**。Node/TS の `import Anthropic from '@anthropic-ai/sdk'` や Python の `from google import generativeai` などは検知できない。
3. **`curl.*anthropic` は大文字小文字を区別する**。`grep -rE` に `-i` が無いため、`CURL` や `ANTHROPIC` の混在するスニペットを取りこぼす。本記事の冒頭が `bash` スクリプトの調査を主題にしている以上、shell の慣例として小文字統一が多く影響は小さいが、SDK 定数名（`ANTHROPIC_API_KEY` 等）での言及は取りこぼす。

記事の主題が「AI 生成スキルの claim vs 実装」である以上、「LLM 呼び出しが無い」という結論を出す grep パターンの精度は、記事全体の説得力に直結する。

### 提案
検出対象を広げ、ケース非依存にし、コメント行に意図を残す書き換え案。

```bash
# LLM 呼び出しの痕跡を幅広く探す（Claude CLI / Anthropic SDK / OpenAI SDK / Gemini）
grep -rEi \
  "claude +-p|curl.*(anthropic|openai|generativelanguage)|anthropic-ai/sdk|import +Anthropic|from +anthropic|import +openai|from +openai|google-generativeai" \
  scripts/
```

さらに、**検出ゼロなら即 claim を疑う**というルールに加え、**検出があった場合は呼び出し箇所の入力・出力がスクリプトの最終生成物に反映されているか**までトレースする、という一段深いチェックを併記すると、記事の「動詞トレース」手法との一貫性が出る。

```markdown
検出ゼロなら、分析系の claim は自動で疑います。
検出があった場合は、その呼び出しの **出力が最終生成物（coding_style/*.md など）に埋め込まれているか** までトレースします。呼び出しはあるが結果が捨てられている、というパターンもあります。
```

**優先度: High**（記事の主題である「検知手法」の精度に直結、実害のあるコードブロック）

## 総合評価

### 良い点

- **主題と構成の整合性**: 「AI に書かせた SKILL.md の claim が実装と乖離していた」という発見から、「検知する 3 手法」「対応する 3 策」へと順接で展開しており、読者が追いやすい。
- **`:::message` 導入パターンの踏襲**: 著者既出記事と同じ「この記事で得られること」3 点セットで、冒頭の価値提示が整っている。
- **仮説の提示が誠実**: 「なぜこのズレが起きるか」セクションで 3 仮説を立てて、断定を避けているのが読者の信頼を得やすい。
- **対応策 A/B/C の明示**: 「採用したのは C」まで踏み込んで実運用の意思決定を共有しており、idea 型記事として具体性がある。
- **「エージェントが自分で作ったスキルを別のエージェントが使う連鎖」**（L145）: 本記事の提起する最大の論点で、短文ながら読者の印象に残る一文になっている。

### 改善点

- **語彙の統一**: 冒頭 `:::message` の「クレーム」と本文の `claim` が不揃い、`analytical` / `Hybrid 運用` の日本語副題不在など、英語ラベルの初出整備が必要（指摘 2, 4）。
- **セルフチェック grep パターンの精度**: 記事の主題である検知手法のコア grep が、Claude CLI や他 SDK を取りこぼすパターンになっている（指摘 5）。
- **全体像 → 詳細の順序**: 「ズレを検知する方法」章が詳細 3 節を冒頭で俯瞰できない構成（指摘 3）。
- **関連記事リンクの欠落**: 著者既出記事との連結・回遊導線が途切れている（指摘 9）。
- **まとめと直前章の重複**: 「付き合い方」と「まとめ」で主張が 2 重に再掲されている（指摘 7）。

### 推奨アクション

1. **High 優先度 3 件の修正**: 冒頭 `:::message` の語彙統一（指摘 2）、英語ラベル日本語副題（指摘 4）、grep パターン精度向上（指摘 5）、関連記事リンク追加（指摘 9）。このうち指摘 5 は主題の信頼性に直結するため最優先。
2. **Medium 優先度 4 件の修正**: 全体像先出し（指摘 3）、grep 出力ブロックの区分明示（指摘 6）、まとめ重複の圧縮（指摘 7）、硬い表現の柔らかい言い換え（指摘 8）、スキル出典の 1 行補足（指摘 10）。公開後の追補でも差し支えないが、読後印象に関わるため可能な範囲で公開前対応推奨。
3. **Low 優先度 1 件**: 公開ステータスの切り替えフロー（指摘 1）。レビュー反映 PR のコミット段階で対応。

### 公開可否と条件

**条件付き公開可**。High 優先度 4 件（指摘 2・4・5・9）を反映すれば Zenn 公開可能な水準。特に指摘 5（grep パターン）は、この手順を真似て「検出ゼロなのに実は Claude CLI が呼ばれている」スクリプトを見逃す読者が出る可能性があるため、公開前必須。指摘 9（関連記事リンク）は依頼背景の「関連記事リンク配置」要件に直接該当するため、こちらも公開前対応が望ましい。

### SEO観点での改善提案

- **タイトル**: 現状「AIに書かせた壮大なSKILL.mdを実装したら、ただのテンプレ生成器だった話」は 32 文字で Zenn のタイトル長として適切。「AIに書かせた」「SKILL.md」「テンプレ生成器」という具体語で検索意図を捕捉できており、釣り感のない訴求になっている。
- **topics**: `["ai駆動開発", "claudecode", "スキル設計", "ドキュメント", "ai生成"]` は記事内容と整合。「スキル設計」は `engineer-skill-creator` 系の記事とも共通するキーワードなので、著者内での相互流入が見込める。強いて言えば「ai生成」はより具体的な「AI駆動ドキュメント」「プロンプトエンジニアリング」などに差し替える余地がある（ただし現状でも十分）。
- **内部リンク**: 指摘 9 で提案した 3 件の関連記事リンクで、著者既出記事との回遊率が向上。特に `engineering-process-improvement-skill.md` との双方向リンク（あちらの「関連記事」にも本記事を追記）を入れると効果が倍増する。
- **h2/h3 階層**: 現状の階層は適切。「発見 → 仮説 → 検知 → 対応策 → 付き合い方 → まとめ」という物語構造で、目次から記事の射程を把握しやすい。
- **導入文のフック**: L19-L27「胸が熱くなりました」→「実装は静的テンプレ生成器だった」のギャップ提示は、idea 型記事のフックとして強い。`:::message` の後に続く物語導入として機能している。
