## 指摘コメント

### 該当箇所 1
L141 （MCP のトークン消費の数値）

```
| トークン消費（目安） | 約 13,700 | 約 18,000 |
```

### 問題点
「約 13,700 / 約 18,000」という具体値が、L149 に挙げた参考リンク（Steve Kinney / Test-Lab.ai）本文から確認できませんでした（WebFetch による要約では両ソースとも該当数値が surface しませんでした。**ソース側の記述欠落とは断定できず「未検証」**）。MCP のトークン消費は対象ページ・操作・バージョンで大きく変動する性質の数値であり、出典・測定条件（どのツール版で、何の操作 1 回あたりか）が無いまま具体値を載せると、E-E-A-T の「正確性」評価で減点されやすく、数字だけ独り歩きするリスクがあります。

### 提案
測定条件と出典を明示するか、断定値を避けて相対表現に弱めるのが安全です。

```
# before
| トークン消費（目安） | 約 13,700 | 約 18,000 |

# after（出典・条件を脚注で明示する場合）
| トークン消費（目安）※1 | やや少なめ | やや多め（スクショ添付のぶん嵩みやすい） |

※1 数値は <出典・測定日・MCP バージョン・対象操作> での実測。環境により変動する。
```

実測値を残すなら「いつ・どのバージョン・どの操作で測ったか」を 1 行添えると、むしろ一次情報性（E-E-A-T）が強化されます。


### 問題点

「Chromium を駆動するときは」と条件を付けるだけで正確になります。

```
# after
ポイントは最下段の **CDP（Chrome DevTools Protocol）**。Playwright も **Chromium を動かすときは** このプロトコル（に相当する経路）を使うので、
```

L75 も「Playwright も Puppeteer もこの上に乗る」→「Puppeteer や Playwright（Chromium 駆動時）がこの上に乗る」へ調整を推奨します。

### 該当箇所 8
L143, L169, L22, L36 （"Driving vs Debugging" の独自性・出典明示）

```
L143: ざっくり言えば、**「何が起きたか（ユーザー視点）= Playwright MCP」「なぜ起きたか（ブラウザ視点）= Chrome DevTools MCP」**。
```

### 問題点
「Driving vs Debugging」は既存言説（記事末尾に挙げた Steve Kinney 系の比較でも使われるフレーミング）であり、本記事のオリジナル概念ではありません。現状この対比をオリジナル主張のように提示している箇所（L22, L36, L134, L143, L169）が複数あり、独自性の観点では「既存フレームの借用」と読まれかねません。一方で本記事の真の独自性は「**OAuth 管理画面で既存セッションを流用する**」という一次体験にあります。借用フレームと自分の体験の境界を明示すると、E-E-A-T の信頼性が上がります。

### 該当箇所 9
L107, L44, L188 （OAuth 自動化の「壊れやすさ」を一般化している箇所）

```
L107: **外部認証（Google OAuth）と相性が悪い**。... ログインの自動化は外部 IdP リダイレクト・bot 検知でコストの割に壊れやすい。
```

### 問題点
主張の方向性は妥当ですが、「OAuth = Playwright で自動化すると壊れる」とやや断定的に一般化しています。実際には Google OAuth は**テスト用サービスアカウント / 専用テストユーザー / `storageState` の事前生成**で安定自動化している現場も多く、技術読者から「それは認証戦略の問題で、Playwright の弱点ではない」と反論されやすい箇所です。記事の説得力を守るには「自分の現場では割に合わなかった」という**体験ベースの限定**に寄せるのが安全です。

### 提案
```
# before
**外部認証（Google OAuth）と相性が悪い**。... コストの割に壊れやすい。

# after
**外部認証（Google OAuth）の自動化は、私の現場では割に合わなかった**。
専用テストユーザーや storageState 事前生成で安定させる手もあるが、UI 変更や bot 検知への追従コストが、
得られる安心に見合わなかった——という実測判断。
```

「一般論」ではなく「自分の実測判断」と明示することで、技術的反論を避けつつ E-E-A-T（一次情報性）を強化できます。

## 総合評価

### 良い点
- **構成が完成度高い**: 結論先出し（`:::message`）→ レイヤー整理 → 比較表 → 理由 → ワークフロー → チェックリスト → まとめ → FAQ と、読者が価値に最短到達できる順序。Zenn 読みやすさ観点 1〜8 をほぼ満たす。
- **一次体験の具体性（E-E-A-T）**: 「OAuth 管理画面で既存ログイン済みセッションを流用」という、検証可能で再現性のある具体文脈があり、量産記事との差別化軸が明確。
- **比較が表中心で一覧性が高い**: 2 章の差分早見表・5 章の判断チェックリストは、実装者が運用判断に直接使える実用性。
- **MCP への接続が自然**: AI エージェント時代の文脈（4-3）を派生論点として本筋の後ろに置き、同じ "Driving/Debugging" 軸で一貫させている。

### 改善点
- **数値の出典・測定条件**（指摘 1）: MCP トークン消費の具体値に裏取りが必要。最優先。
- **技術的正確性の精緻化**（指摘 2・6・9）: 「Playwright = CDP の上」「OAuth = 壊れる」の過度な一般化を条件付き/体験ベースに弱める。
- **独自性の明示**（指摘 8）: 借用フレーム（Driving/Debugging）と自分の貢献（セッション流用）の境界を立てる。
- **タイトル・topics の SEO**（指摘 5・10）: 最大の差別化キーワード OAuth を露出させる。
- **コア主張の反復**（指摘 4）: 「既存セッション流用」の同一フレーズ反復を段階深掘りに整理。
