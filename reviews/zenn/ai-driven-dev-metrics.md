### 該当箇所 1
L24-L29（TL;DR と `:::message` の役割重複）

```
:::message
**この記事で得られること**

- AI導入の効果を、実装速度だけで評価しない理由
- 最初に追うべき `手戻り率 / 差し戻し率 / 再オープン率` の考え方
- 監視ではなく改善につながるメトリクス設計の始め方
:::

## はじめに
...

## TL;DR

- 実装速度だけでは、AI導入の成否は判断しにくい
- まず見るべきなのは、後工程に流れた問題の量
- `手戻り率 / レビュー差し戻し率 / 再オープン率` から始めるとよい
- メトリクスは評価より改善に使う
```

### 問題点
`:::message` の「この記事で得られること」と TL;DR が実質同じ内容を繰り返している。短尺記事で冒頭に2つの要約ブロックが続くと、読者が「どちらを読めばよいか」迷うとともに、冗長感が生じる。

### 提案
TL;DR を削除し、`:::message` 内に統合する。または `:::message` を「この記事のポイント」（より概念的な入口）として残し、TL;DR を各章の小見出しで代替する。

```markdown
:::message
**この記事のポイント**

- 実装速度だけではAI導入の成否は判断しにくい
- まず見るべき指標: 手戻り率 / レビュー差し戻し率 / 再オープン率
- 監視ではなく改善につながるメトリクス設計の始め方
:::
```

### 該当箇所 2
L51-L63（手戻り率・レビュー差し戻し率・再オープン率の定義が曖昧）

```
### 手戻り率

一度「終わった」と見なした作業が、どれだけ再作業に戻ったかを見る指標です。

### レビュー差し戻し率

レビュー段階で major な修正が必要になった割合です。
実装そのものより、前提の揃え方や設計の粗さが見えます。

### 再オープン率

close 済み Issue や完了扱いのタスクが、どれだけ再度開かれたかを見ます。
```

### 問題点
3指標の定義はあるが、分母・分子が示されていない。たとえば「手戻り率 = 再作業に戻ったタスク数 / 完了扱いにしたタスク数」という計算式がなく、実際に計測しようとしたときに迷う。「major な修正」の定義も読者によって解釈が分かれる。

### 提案
各指標に計算式または計測単位を1行添える形式に変更する。

```markdown
### 手戻り率

一度「終わった」と見なした作業が、どれだけ再作業に戻ったかを見る指標です。

> 計測例: 再作業になったタスク数 ÷ 完了タスク総数

### レビュー差し戻し率

レビュー段階で major な修正（設計変更・要件再確認など）が必要になった PR の割合です。

> 計測例: 差し戻しが発生した PR 数 ÷ レビュー完了 PR 総数

### 再オープン率

close 済み Issue や完了扱いのタスクが、どれだけ再度開かれたかを見ます。

> 計測例: 再オープンされた Issue 数 ÷ クローズ済み Issue 総数
```

### 該当箇所 3
L108-L115（「やってはいけない測り方」の根拠が欠けている）

```
## 6. やってはいけない測り方

- コミット数
- 生成コード行数
- 会話ターン数
- AIが提案した件数そのもの

これらは活動量には見えても、成果や手戻り削減とは直結しません。
```

### 問題点
4つの指標が「ダメな理由」を1文で括ってしまっており、理由が薄い。コミット数と生成コード行数はアンチパターンとして広く知られているが、「会話ターン数」「AIが提案した件数」はAI駆動開発特有の指標であり、なぜ成果に直結しないかの説明が求められる読者層（Claude Code・Codex を使っているエンジニア）に向けて、根拠を添えるべきである。

### 提案
各項目に1行ずつ理由を添える。または2〜3行の説明を加えた形に圧縮する。

```markdown
- **コミット数**: 小さく commit する習慣の人が不利になる
- **生成コード行数**: 冗長なコードを多く生成しても価値は上がらない
- **会話ターン数**: 試行錯誤の多さは学習コストの高さを示す場合もある
- **AIが提案した件数**: 採用されなかった提案はカウントから除外する必要があり、分母の設計が難しい
```

### 該当箇所 5
L77-L91（「小さく始める計測設計」の箇条書きが table 化候補）

```
## 4. 小さく始める計測設計

### 最小セットで十分

最初から厳密な生産性ダッシュボードを作る必要はありません。
Issue、PR、レビュー履歴から拾える最小セットで足ります。

- 差し戻しが発生した PR 数
- 再オープンした Issue 数
- 完了後に追加修正が入った件数
```

### 問題点
3つの計測項目について「何を数えるか」「どのデータソースから取るか」「何の指標に対応するか」が記述されていない。箇条書きのままでは情報が並列的に並ぶだけで、読者が自分のプロジェクトで適用するときにどのシステムから数えればよいかわからない。Zenn 記法活用観点 11「並列・対称な情報は table にする」に該当する。

### 提案
table で計測項目・データソース・対応指標を対応させる。

```markdown
| 計測項目 | データソース | 対応指標 |
| -------- | ------------ | -------- |
| 差し戻しが発生した PR 数 | GitHub PR ラベル / レビューコメント | レビュー差し戻し率 |
| 再オープンした Issue 数 | GitHub Issue ログ | 再オープン率 |
| 完了後に追加修正が入った件数 | PR / Issue のクローズ後更新 | 手戻り率 |
```

### 該当箇所 7
L117-L122（まとめが既出の主張と完全に重複している）

```
## まとめ

AI導入の効果を見るなら、まずは「どれだけ速く書けたか」ではなく、「どれだけやり直しを減らせたか」を見たほうが実務に効きます。

速度は見えやすい指標ですが、改善に効くのは後工程で漏れた問題の量です。
小さなメトリクスから始めて、計画・レビュー・検証のどこに課題があるかを見つけるのが現実的です。
```

### 問題点
まとめの内容が `:::message` と TL;DR の内容をほぼそのまま繰り返している。短尺記事（123行）においては、3箇所で同じ主張を繰り返すと冗長感が強くなる。Zenn 読みやすさ観点での「冗長な繰り返し（まとめ・導入・章頭で同じ主張が3回以上出てくる）」に該当する。

### 提案
まとめでは既出の主張を再掲するのではなく、「次のアクション」または「記事を読んだあとに1つやること」を示す形に切り替える。

```markdown
## まとめ

この記事が示したのは「速度より手戻り削減を先に見る」という優先順位の話です。

もし今日から始めるなら、直近1ヶ月の PR から「差し戻しが発生したもの」を数えるだけでも十分です。
その数が変化するかどうかを追うことが、AI導入効果の最初の計測になります。
```

## 総合評価

### 良い点
- `:::message` を冒頭に置き、記事の価値を先出しする構成は適切
- 「速度ではなく手戻り」という主張が全章を通じて一貫している
- 「人を評価しない」という運用上の前提（L103-L106）が明記されており、現場導入時の誤用を防ぐ記述として有効
- 短尺に絞り込んだことで、概念の核心が読み取りやすい

### 改善点
- `:::message` と TL;DR の内容重複（指摘1・指摘7）により、短尺記事にもかかわらず冗長感が生じている
- 3指標の定義が概念説明にとどまり、計測式・計測単位がない（指摘2）
- 「やってはいけない測り方」の根拠が1文で括られており、AI駆動開発特有の指標（会話ターン数・提案件数）の説明が不足（指摘3）
- 章の入口に全体像がなく、各 `###` を読んで初めて構造がわかる（指摘4）

### 推奨アクション
1. `:::message` と TL;DR のいずれかを削除し、冒頭の要約を1箇所に統合する
2. 各指標（手戻り率・差し戻し率・再オープン率）に計測式または計測単位を1行添える
3. 「やってはいけない測り方」の4項目に、それぞれ1行ずつ理由を付加する
4. 「最初に見るべき指標」章の冒頭に指標一覧（3つ）を先出しする
5. 「小さく始める計測設計」の箇条書きを table（計測項目・データソース・対応指標）に変換する

### SEO観点での改善提案
- タイトル「速度ではなく手戻り率を見る」は検索意図に合っているが、「AI駆動開発」「開発生産性」など topics に登録済みのキーワードをタイトル本文に含めると検索流入が増加しやすい
- topics に `metrics` と `開発生産性` が両立しているが、記事本文に「開発生産性」という語が登場しない。本文中に1〜2回取り込むと一貫性が上がる
- 現状 `published: false` のため、公開前に指摘事項の反映と最終確認を行う
