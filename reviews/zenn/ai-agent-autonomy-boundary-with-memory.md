## 指摘コメント

### 該当箇所 1
L17-L28 （`## はじめに` の導入が「エージェント/ユーザー」の対話ログとして読み取りにくい）

```markdown
AI エージェント（Claude Code / Codex など）と協働していて、同じ確認ループに **5 回連続で入った** ことがあります。

ユーザー: 「進めて」  
エージェント: 「A / B / C どれにしますか？」  
ユーザー: 「進めて」  
エージェント: 「A / B どれにしますか？」  
```

### 問題点

本記事の主張の起点となる「5 回連続で同じループに入った」という体験が、対話ログ L21-L24 では 2 往復分しか提示されておらず、読者が「本当に 5 回入ったのか」を実感しにくい。また、ログが「コードブロックではなく、末尾に半角スペース 2 つ入りの改行」で書かれているため、Zenn のレンダリング上は行の区切りがつくが、Markdown エディタで閲覧する段階ではエージェントとユーザーの発言が視覚的に区別されにくい。idea 記事は主張より前に「共感できる具体性」を置くことで読者の納得度が決まるため、この導入の弱さは記事全体の訴求力に直結する。

### 提案

対話ログをコードブロック化し、3 往復目以降の「選択肢が減っていく様子」まで見せて、5 回目の気づきを読者も追体験できるようにする。

```markdown
AI エージェント（Claude Code / Codex など）と協働していて、同じ確認ループに **5 回連続で入った** ことがあります。

\`\`\`text
ユーザー: 進めて
エージェント: A / B / C どれから着手しますか？

ユーザー: 進めて
エージェント: A か B のどちらにしますか？

ユーザー: 進めて
エージェント: A を進める前に、対象ファイルを確認しますか？

ユーザー: 進めて
エージェント: 対象ファイルは X / Y / Z です。どれから編集しますか？

ユーザー: 進めて
エージェント: ……（ここで 5 回目。ようやく「自走合図だった」と気づく）
\`\`\`
```

**優先度: High**（idea 記事は体験談の具体性で主張の説得力が決まる。ここを強化するだけで記事全体の読後感が大きく変わる）

### 該当箇所 5
L47-L72 （自走 OK / 人間確認必須 / グレーゾーンの 3 分類が、CLAUDE.md の記述と細部で食い違う）

```markdown
### 自走 OK

- ファイル編集 / 作成 / 削除（ローカル、かつ reversible）
- `git commit` / `git push`（feature branch へのみ）
- テスト実行 / lint / typecheck
- PR の作成（本文生成も含めて）
- ドキュメント更新 / コメント追加

### 人間確認必須

- `git push --force` / `git reset --hard` 等の破壊操作
- `git push main`（保護ブランチへの直接 push）
- PR のマージ（自動マージ禁止の規約下）
- 外部サービス（SaaS、CMS、本番環境）への操作
- セキュリティ / 認証情報に関わる変更
- 新しい依存関係の導入
```

### 問題点

本記事は `published: false` だが、著者の運用規約である `CLAUDE.md`（本リポジトリ L96-L99 の Permissions 指針）では **「自動許可してよい」は Read / Glob / Grep / 記事ディレクトリへの Write / `npm run preview|check`**、**「確認を挟む」は `git push`, `gh pr create`, `gh pr merge`, 大規模削除** とされている。つまり `git push`（feature branch 含む）と `gh pr create` はどちらも「確認を挟む」側に分類されている。

一方、本記事 L54 では **`git commit` / `git push`（feature branch へのみ）を自走 OK** としており、さらに L56 で **PR の作成（本文生成も含めて）を自走 OK** としている。これは著者自身のリポジトリ運用規約と矛盾しており、読者が本記事をそのまま真似すると「著者のリポジトリでは確認されるはずの操作が、著者の記事では自走推奨になっている」状態になる。

idea 記事として主張の一貫性は重要な評価軸で、ここで自己矛盾があると「この 3 分類は本当に実運用されているのか」という読者の信頼を損なう。

### 提案

どちらか一方に揃える。以下の 3 択のいずれか。

**(A) 記事の分類を CLAUDE.md に合わせる（最も安全）**

```markdown
### 自走 OK

- ファイル編集 / 作成 / 削除（ローカル、かつ reversible）
- `git commit`（feature branch へのみ）
- テスト実行 / lint / typecheck
- ドキュメント更新 / コメント追加

### 人間確認必須

- `git push`（feature branch でも、初回 push は確認）
- `git push --force` / `git reset --hard` 等の破壊操作
- `git push main`（保護ブランチへの直接 push）
- PR の作成（`gh pr create`）と マージ
- 外部サービス（SaaS、CMS、本番環境）への操作
- セキュリティ / 認証情報に関わる変更
- 新しい依存関係の導入
```

**(B) 「著者の現在地」と「一般的な推奨」を分けて書く**

```markdown
以下は私の現時点の運用です。プロジェクト初期はより厳格（`git push` / PR 作成も確認）で、信頼が積み上がるにつれて自走範囲を広げてきました。

### 自走 OK（信頼が積み上がったあと）
...

### 人間確認必須（どのプロジェクトでも固定）
...
```

**(C) この食い違い自体を記事の論点にする**

「自走 OK の境界は、プロジェクトの成熟度とエージェントとの信頼残高で動く」という追加節を設け、`CLAUDE.md` の現時点のルールと本記事で提示する理想像の差を意図的に見せる。

**優先度: High**（idea 記事の主張の一貫性と、著者の運用規約との整合性は、シリーズ読者の信頼性評価に直結する）

### 該当箇所 6
L97-L117 （`memory` ファイルのサンプルで、front matter の `name` が英語文で書かれている）

```markdown
\`\`\`markdown
---
name: User prefers 進めて/続き as autonomous proceed signal
type: feedback
---

ユーザーは「進めて」「続き」を、直前に提示した計画をそのまま承認して
自走する合図として使う。
```

### 問題点

`name: User prefers 進めて/続き as autonomous proceed signal` は、本文（日本語）と front matter（英語の自然文）が混在しており、読者の再利用時に「`name` は英語で書くのが決まりなのか、日本語でも良いのか」が判断できない。また、`type: feedback` がどういう値を取り得るか（例: `rule` / `preference` / `learning` など他の候補があるのか）が本文で説明されない。

Claude Code の memory 仕様に詳しい読者は推測できるが、記事の `:::message` で「memory 機能で feedback を永続化する書き方」を価値として提示している以上、「書き方」の仕様を読者が再現できる形に整えるべき。

### 提案

front matter の直前に、値の取り方の 1 行補足を添える。

```markdown
以下が実際の memory エントリです。`name` は検索しやすい短い要約、`type` はそのエントリの種別（`feedback` = セッションの学び、`rule` = 守るべき規約、など）を入れます。

\`\`\`markdown
---
name: 「進めて」を自走合図として扱う
type: feedback
---

ユーザーは「進めて」「続き」を、直前に提示した計画をそのまま承認して
自走する合図として使う。
...
```

あるいは `name` 英語のままにするなら、**その理由**（例: 「英語のほうがエージェント側の検索インデックスに乗りやすい」など）を 1 行添える。理由が明確でなければ日本語に揃えるほうが再現性が高い。

**優先度: Medium**（memory の書き方を提示する記事なので、サンプルの一貫性は読者の再利用時の摩擦に直結する）

L68-L72 （「グレーゾーン」の 3 項目に対して、本記事の主張（memory 永続化）がどう効くかの接続がない）

```markdown
### グレーゾーン（状況次第）

- ライブラリ更新 → 破壊的変更の有無で判断

```

### 問題点

L51-L72 で「自走 OK / 人間確認必須 / グレーゾーン」の 3 分類を提示しているが、直後の `## 「進めて」を自走合図として扱う記法`（L74）で扱われるのは「自走 OK」と「破壊操作（= 人間確認必須）」の 2 分類のみで、**グレーゾーンへの言及がそこから消える**。L91 に「一方、グレーな破壊操作が途中で出てきた場合だけ確認を挟みます」と一度出てくるが、L68 で定義した「大規模リファクタ / ライブラリ更新 / CI 設定変更」とは粒度が噛み合っていない（こちらは「破壊操作」ではなく「影響範囲判断」が必要な系統）。

読者は「グレーゾーンは結局 memory にどう書くのか」「『進めて』合図で自走して良いのか」を知りたいはずだが、記事は答えを出さずに memory 章へ進む。idea 記事として「線引きを memory で永続化する」を主張する以上、グレーの扱いこそが最も永続化の価値があるはずで、ここが空白なのは主張の弱点になる。

### 提案

グレーゾーン節の末尾か、`## 「進めて」を自走合図として扱う記法` の末尾に、「グレー項目はどう扱うか」を 1 段落追加する。

```markdown
### グレーゾーン（状況次第）

- 大規模リファクタ → 影響範囲を見て判断
- ライブラリ更新 → 破壊的変更の有無で判断
- CI 設定変更 → 既存ジョブへの影響で判断

**グレーゾーンは「進めて」合図では自走せず、影響範囲の説明と選択肢提示をワンクッション挟む**、というのが私の現在のルールです。これもあとで memory に書きます。
```

**優先度: Medium**（主張の主軸は自走 OK / 必須の 2 軸でも通るが、グレー扱いは idea 記事の独自性を支える部分なので、触れておくと記事の厚みが増す）

### 該当箇所 9
L2 （タイトルと topics の SEO キーワード密度）

```markdown
title: "AIエージェントの『進めて』問題：自走と確認の境界をmemoryで永続化する"
topics: ["ai駆動開発", "claudecode", "codex", "対話設計", "エージェント設計"]
```

### 問題点

タイトルは記事内容を的確に表現しており、「進めて」問題というキャッチーな題材で目を引く構造は悪くない。ただし Zenn の検索流入を意識すると、以下の点が弱い。

1. **`memory` が英小文字のまま**: Zenn 検索は表記ゆれに弱く、`memory` / `メモリ` / `記憶` のどれでも拾えるようにしたほうが流入が広がる。タイトルに「永続化する」は入っているが、「memory 機能」の全体像を検索する読者はカタカナで探すケースが多い。
2. **「自走」という語のヒット可能性**: `自走` は AI エージェント文脈では定番語になりつつあるが、初見読者は「自律」「オートノミー」で検索する可能性が高い。topics に `agent-autonomy` 相当の語がない。
3. **topics に `AGENTS.md` / `memory` に関連する語がない**: 本記事のコア概念は topics に反映されておらず、タイトルと topics の齟齬がある。

タイトル案を複数並べる。

```text

案 C（主張寄せ）:
「AIエージェントに『待つ』か『走る』かを覚えさせる：memory で境界を永続化する」
```

topics は以下を検討。

```yaml
topics: ["ai駆動開発", "claudecode", "codex", "エージェント設計", "memory"]
# または
topics: ["ai駆動開発", "claudecode", "codex", "autonomy", "memory"]
```

（`対話設計` は本記事の主題からやや外れるため、`memory` と差し替える案）

**優先度: Low**（現タイトルでも意味は通るため、公開前の最終調整段階で A/B 判断可）

### 該当箇所 10
L43, L76, L99 （「進めて」「続き」の括弧・引用符の表記ゆれ）

```markdown
# L42-L43（はじめに節）
2. **承認合図**: 直前に提示された計画を承認し、実行に移ってほしい

# L76
私が今採用しているルールは「**直前に番号付きプランを提示していて、ユーザーが『進めて』と返したら、リストの上から実行する**」です。

# L99 (memoryブロック内)
name: User prefers 進めて/続き as autonomous proceed signal

# L103
ユーザーは「進めて」「続き」を、直前に提示した計画をそのまま承認して
```

### 問題点

「進めて」「続き」という 2 つのキーワードの表現にゆれがある。

- L19-L24: 対話ログでは `「進めて」` のみ（`続き` は登場しない）
- L28: 「5 回目でようやく『これは自走合図だったのか』と気づいて」で `「」` と `『』` が混在
- L40: `ユーザー側の「進めて」には 2 つのモードがあります` → `「進めて」` のみ言及（`続き` はここでもなし）
- L76: `「**直前に番号付きプランを提示していて、ユーザーが『進めて』と返したら…**」` で太字の `「」` 内に `『』` の入れ子
- L99: `User prefers 進めて/続き` で `続き` が登場
- L103: `ユーザーは「進めて」「続き」を、…` で `続き` が登場

記事の主張として扱うキーワードは「進めて」なのか「進めて / 続き」の 2 つなのかが、**読む位置で揺れる**。memory ブロック（L99-L117）で `続き` が追加されたあとも、本文中で `続き` が果たす役割は説明されない。

### 提案

「進めて / 続き」の 2 つを扱うならば、`## はじめに` で両方提示する。`続き` の用例も 1 つ添える。

```markdown
## はじめに

AI エージェント（Claude Code / Codex など）と協働していて、「進めて」や「続き」で自走させたいのに、毎回確認ループに入ってしまう同じパターンに **5 回連続で遭遇** したことがあります。
```

引用符は記事全体で `「」` に統一し、入れ子のときだけ `『』` を使う。

**優先度: Low**（表記ゆれは読者の違和感として蓄積するが、記事の主張自体は影響を受けない）
