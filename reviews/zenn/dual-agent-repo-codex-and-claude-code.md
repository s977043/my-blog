### 該当箇所 1
L28（「規約レイヤリング」の定義が冒頭で提示されず読者が概念を掴みにくい）

```
この記事は、マルチエージェント時代の **規約レイヤリング** について整理したものです。
```

### 問題点
**優先度: Medium**

「規約レイヤリング」は本記事の核キーワードだが、初登場（L28）時点では定義がなく、読者は次のセクション（L30「## 2 層規約の役割」）まで読んで初めて「2 層に分けること」だと理解できる。冒頭 `:::message`（L13）でも「2 層設計」という別表現が使われており、**同じ概念を指す表現が揺れている**（「規約レイヤリング」と「2 層規約」「2 層設計」）。Web 編集者視点での表現統一の観点でも改善余地がある。

`plangate-ai-coding-workflow.md` や `engineering-process-improvement-skill.md` では、核キーワードを初登場時に 1 行定義するパターンが定着している。本記事はタイトルで「2 層規約」を押し出しているので、そちらに表現を寄せるほうが記事全体の一貫性も高まる。

### 提案

```markdown
この記事は、マルチエージェント時代の **2 層規約**（共通の規約は `AGENTS.md`、エージェント固有のツール索引は `CLAUDE.md` 等に分ける設計）について整理したものです。
```

または「規約レイヤリング」を残すなら、初登場時にカッコ書きで定義を添える:

```markdown
この記事は、マルチエージェント時代の **規約レイヤリング**（規約とツール索引をレイヤ分けする設計）について整理したものです。以降、本記事ではこの設計を **2 層規約** と呼びます。
```
