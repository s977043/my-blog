## 指摘コメント

### 該当箇所 1
L57 （「イメージ」で済ませている比較章の根拠不足）

```
同じソースを 3 箇所で表示させると差が出ます。以下はイメージです。
```

### 問題点
本記事の核心である「プラットフォーム間で表示が違う」を証明する章が「イメージ」という断り書きだけで、実機スクリーンショットも、レンダリング結果の HTML / CSS 差分の提示もありません。L21 で「note で投稿した実記事が視覚的に間延びした」と個人の実体験を強調しておきながら、比較章ではエビデンスが抜けており、読者が追検証できません。技術記事の信頼性を損なうため、少なくとも 1 点は実測データ（スクリーンショット / CSS 検証結果 / 具体的な余白 px 値など）を示すべきです。**改善優先度: High**。

### 提案
いずれかの形で実測情報を追加することを推奨します。

- 3 プラットフォームに同一 Markdown を投稿し、スマホ幅での実表示を並べたスクリーンショット
- note / Zenn / Qiita 各々の `<br>` に対する CSS（margin / line-height）を DevTools で抽出し、下記のような表で提示

```markdown
| プラットフォーム | `<br />` 直後の視覚的空きサイズ | 段落 `<p>` のマージン |
| ------------- | --------------------------- | ------------------- |
| Zenn          | 1 行分（約 1.5em）             | 1.5em               |
| Qiita         | 1 行分（約 1.2em）             | 1.2em               |
| note          | 約 2 行分（段落級）             | 1em                 |
```

スクリーンショット撮影が難しい場合でも「実測値が取れていない」「観察ベースの比較である」と明示する一文を足すだけで信頼度が上がります。

### 該当箇所 2
L88-L92, L131-L134, L144-L149 （スペース除去正規表現の不整合と移植性）

```python
# L88-L92: Python
import re
fixed = re.sub(r" +$", "", source, flags=re.MULTILINE)
```

```bash
# L131-L134: sed
sed -i '' 's/  *$//' articles_note/new/<slug>.md
```

```python
# L144-L149: 簡易チェック
two_space_breaks = sum(1 for line in text.split("\n") if line.endswith("  "))
if two_space_breaks > 5:
    print(f"[WARN] 2 スペース改行 {two_space_breaks} 箇所。note で余白過多の可能性")
```

### 問題点
同じ目的（行末 2 スペース改行の除去/検出）に対して、3 つの例でマッチ条件が揃っていません。

1. **L91 Python**: `r" +$"` は **1 個以上** の末尾スペースを除去します。これは副作用として、インデント揃えや単なる打ち間違いの 1 スペースも除去してしまうため、「2 スペース改行の除去」という本文の説明と挙動が一致しません。
2. **L133 sed**: `'s/  *$//'` は「スペース 1 個 + スペース 0 個以上」= **1 個以上**。Python 例と同じく、記事の主題である「2 個以上」ではありません。
3. **L146 Python**: `line.endswith("  ")` は **ちょうど末尾 2 文字がスペース** を満たすか。CommonMark 仕様で hard break は「2 個以上のスペース」のため、3 個・4 個のケースもカウントされますが、これは一致します。ただし 1 と 2 の正規表現が「2 個以上」を対象にしていないため、仕様記述（L27「2 つ以上のスペース」）との一貫性が崩れています。

加えて **L133 の `sed -i ''` は macOS (BSD sed) 専用**、Linux (GNU sed) では `sed -i 's/...'` と書く必要があり、記事の対象読者がマルチ OS の可能性を考えると注釈が必要です。**改善優先度: High**。

### 提案
仕様（2 個以上）と例を揃え、OS 差も明記します。

```python
# 行末 2 個以上のスペース（= CommonMark hard break）を除去
import re
fixed = re.sub(r" {2,}$", "", source, flags=re.MULTILINE)
```

```bash
# macOS (BSD sed)
sed -i '' -E 's/ {2,}$//' articles_note/new/your-slug.md

# Linux (GNU sed)
sed -i -E 's/ {2,}$//' articles_note/new/your-slug.md
```

```python
# 2 個以上の末尾スペースを検出
two_space_breaks = sum(
    1 for line in text.split("\n") if line.rstrip(" ") != line and len(line) - len(line.rstrip(" ")) >= 2
)
```

### 該当箇所 4
L53 （主要 Markdown 実装の挙動に関する断定）

```
これは仕様として正しい動きで、Python-Markdown / marked / remark などの主要実装も同じ挙動です。
```

### 問題点
「主要実装も同じ挙動」は厳密には正しくありません。たとえば **Python-Markdown は標準では CommonMark 完全準拠ではなく**、hard line break の扱いは `nl2br` 拡張や実装バージョンで変わります。また **marked** は CommonMark / GFM 設定で挙動が変わり、**remark** は remark-gfm / remark-breaks 等のプラグイン構成に依存します。記事の主題（同じ Markdown でも表示差が出る）とむしろ矛盾する断定になりかねないため、条件付きの書き方が望ましいです。**改善優先度: Medium**。

### 提案
```markdown
これは仕様として正しい動きで、CommonMark 準拠の実装（cmark, markdown-it など）では同じ挙動になります。Python-Markdown / marked / remark 等の実装では、標準設定か拡張（`nl2br`, GFM, remark-breaks）の有無で挙動が変わる点に注意してください。
```

### 該当箇所 5
L34-L38 （仕様説明で「段落内の空白改行」という表現が不正確）

```
↓ `<p>` で包まれて段落内の空白改行

```html
<p>この行は このように改行されます</p>
```

### 問題点
「段落内の空白改行」という表現が曖昧です。CommonMark では、hard break がない状態で改行された 2 行は、レンダリング後は **単一の半角スペースでつながった 1 行**（soft line break）になります。出力例 `<p>この行は このように改行されます</p>` はまさに半角スペースでつながった形ですが、本文の説明文「空白改行」では読者が「改行が残るのかスペースになるのか」を判断できません。**改善優先度: Medium**（誤字脱字というより誤解を招く表現）。

### 提案
```markdown
↓ 改行は半角スペースに置き換えられ、1 行の段落として描画されます

```html
<p>この行は このように改行されます</p>
```
```

### 該当箇所 8
L146-L149 （コメントとログ文の「余白過多の可能性」で推測調が残る）

```python
two_space_breaks = sum(1 for line in text.split("\n") if line.endswith("  "))
if two_space_breaks > 5:
    print(f"[WARN] 2 スペース改行 {two_space_breaks} 箇所。note で余白過多の可能性")
```

### 問題点
記事本文（L77, L86, L155）では「note では `<br />` が段落級の余白になる」と断定している一方で、サンプルコードのログメッセージは「可能性」と推測調になっています。記事主張とコードのトーンに乖離があり、読者に「結局断定できないのか」と感じさせます。また閾値 `5` の根拠説明もありません。**改善優先度: Low**。

### 提案
```python
two_space_breaks = sum(1 for line in text.split("\n") if line.endswith("  "))
# 経験則: 5 箇所を超えるとスマホで余白過多が目立ちやすい
if two_space_breaks > 5:
    print(f"[WARN] 2 スペース改行 {two_space_breaks} 箇所。note では段落級の余白が発生します")
```

```yaml

## 総合評価

### 良い点
- **冒頭の `:::message` で記事の価値を先出し**：3 つの持ち帰り事項が冒頭で明確になっており、Zenn 読みやすさ観点 1 を満たしている（L9-L15）
- **一次体験に基づく説得力**：「note に投稿したら実記事が間延びした」という実体験ベースの問題提起が、技術記事として読み手を惹きつける（L19-L23）
- **構成が素直**：仕様 → 実例 → 運用ルール → AI 生成時の注意 → まとめ、という流れが読者の思考順序と合っている
- **見出し階層の一貫性**：h2/h3 の使い分けが整理されていて、目次生成時の可読性が高い
- **まとめ前に主張を再掲**：L151-L160 のまとめが記事の核を圧縮して反復しており、読後の持ち帰りが明確

### 改善点（優先度順）
- **High**: 「イメージ」で済ませている比較章に実測データを最低 1 つ追加（指摘 1）
- **High**: スペース除去サンプルコード 3 種の仕様不一致と OS 差を解消（指摘 2）
- **High**: 実例比較章の冒頭に 3 プラットフォーム比較表を追加（指摘 3）
- **Medium**: 主要 Markdown 実装の挙動記述を条件付きに修正（指摘 4）
- **Medium**: 「段落内の空白改行」の曖昧表現を「半角スペースに置き換え」等に修正（指摘 5）
- **Medium**: PR #67 / #71 へのリンク付与（指摘 6）
- **Medium**: note CSS 設計意図への言及を観察ベースに修正（指摘 9）
- **Low**: 「全角 5 行以上」の表現統一（指摘 7）
- **Low**: WARN ログと本文主張のトーン揃え（指摘 8）
- **Low**: タイトル末尾の英語とトピックタグの再検討（指摘 10）

### 推奨アクション
1. **High 指摘 3 件を優先反映**：比較表 + スペース除去コード修正 + 実測データ補強。これだけで技術記事としての信頼度が一段上がる
2. **Medium 指摘 4 件を公開前に反映**：仕様説明の正確性と実体験のトレーサビリティを担保
3. **Low 指摘 3 件**：反映推奨だが公開ブロッカーではない
4. **公開可否**: High 指摘 3 件を反映した上で `published: true` へ切り替え。現状（published: false）での公開判断は見送り推奨

### SEO観点での改善提案
- **タイトル日本語化**: 「semantics」を「改行の扱い」に置き換えると、日本語検索流入が増やせる（指摘 10）
- **トピックタグの精度向上**: `python` を `commonmark` に差し替える、もしくは `markdown-lint` 等のツール系タグを足す
- **関連記事への内部リンク**: 同リポジトリ内の note 運用系記事（記事レビュー関連や `note-export-import` スキル紹介記事等）への相互リンクを追加することで回遊性と SEO 双方に寄与
- **メタ記述（Zenn の `:::message` を代替）**: すでに冒頭で価値提示ができているため、現状維持で可。ただし検索結果のスニペットとして表示されることを意識した 120 文字程度の要約を L17-L23 のはじめにで整えると、CTR 改善に寄与する
- **キーワードの早期出現**: タイトルにある `2 スペース改行` が本文冒頭（L19）で `  \n` と記号表記されているのみ。早い段階で日本語キーワード「2 スペース改行」を明示的に登場させると検索エンジン側の主題判定が安定する
