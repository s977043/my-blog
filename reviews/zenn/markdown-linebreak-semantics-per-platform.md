## 指摘コメント

### 該当箇所 1
L53 （主要 Markdown 実装の挙動に関する断定）

```
これは仕様として正しい動きで、Python-Markdown / marked / remark などの主要実装も同じ挙動です。
```

### 問題点
「主要実装も同じ挙動」は厳密には正しくありません。Python-Markdown は標準では CommonMark 完全準拠ではなく hard line break の扱いは `nl2br` 拡張やバージョンに依存します。marked は CommonMark / GFM 設定で挙動が変わり、remark は remark-gfm / remark-breaks 等のプラグイン構成に依存します。記事の主題（同じ Markdown でも表示差が出る）と矛盾する断定になりかねないため、条件付きの表現が望ましいです。**改善優先度: Medium**。

### 提案
```markdown
これは仕様として正しい動きで、CommonMark 準拠の実装（cmark, markdown-it など）では同じ挙動になります。Python-Markdown / marked / remark 等の実装では、標準設定か拡張（`nl2br`, GFM, remark-breaks）の有無で hard line break の扱いが変わる点に注意してください。
```

### 該当箇所 2
L34-L38 （仕様説明の「段落内の空白改行」が曖昧）

```
↓ `<p>` で包まれて段落内の空白改行

```html
<p>この行は このように改行されます</p>
```
```

### 問題点
「段落内の空白改行」という表現が曖昧で、読者が「改行タグが残るのか／半角スペースに置換されるのか」を判断できません。CommonMark では hard break がない場合、2 行は **半角スペースでつながった 1 行** としてレンダリングされます（soft line break）。出力例の HTML もまさに半角スペース区切りの 1 行ですが、本文の説明語と HTML 例が噛み合っていません。**改善優先度: Medium**（誤字脱字ではなく、仕様紹介章の理解負荷）。

### 提案
```markdown
↓ 改行は半角スペースに置き換えられ、1 行の段落として描画されます（soft line break）

```html
<p>この行は このように改行されます</p>
```
```

### 該当箇所 3
L85 （note の CSS 設計意図への言及が推測ベース）

```
note の CSS 設計は、読み物の「余韻」を優先する方向にチューニングされていて、`<br />` も段落級の余白として扱われます。これは note の UX デザインとしては妥当ですが、Zenn / Qiita と同じ感覚で書くと事故ります。
```

### 問題点
「読み物の『余韻』を優先する方向にチューニング」は note 運営の公式アナウンスや CSS 実装を根拠に示しているわけではなく、筆者の推測です。L65 では比較章に「観察ベース」と明示できているのに対し、本段落は運営の設計意図を当てて断定する語り口になっており、レビュー観点「推測・曖昧な記述を避ける」に該当します。記事の信頼性を落とさないためには、観察された CSS 挙動の提示にとどめるか「筆者の推測だが」と明示するのが誠実です。**改善優先度: Medium**。

### 提案
```markdown
note の HTML/CSS を DevTools で確認すると、`<br />` の前後に段落級のマージンが付与されているのが観察できます（筆者環境での観測）。読み物 UX として妥当な設計と推測できますが、Zenn / Qiita と同じ感覚で書くと間延びした表示になります。
```

### 該当箇所 4
L87-L94 （「実測」見出しと内容のトーンにギャップ）

```
## 実測: 11 箇所の 2 スペース改行を入れた記事の末路

私が書いた記事（約 5,500 字）に、レビュー指摘に従って 11 箇所の 2 スペース改行を導入したところ:

- Zenn プレビュー: 問題なし。文のリズムが効く
- note 下書きプレビュー: スマホで間延び、段落境界が曖昧化
```

### 問題点
見出しが「実測」となっていますが、本文で示されるのは定性的な感想（「リズムが効く」「間延び」「境界が曖昧化」）で、余白の px 値やスクリーンショット等の定量的な「実測」ではありません。L65 で「実機スクリーンショットや DevTools による詳細な CSS 値の測定は本記事範囲外」と明示している方針と、この「実測」見出しが矛盾しています。また「約 5,500 字」「11 箇所」という数値も出典（PR 等）が本文に示されていないため追検証できません。**改善優先度: Medium**。

### 提案
見出し側を定性的な観察に合わせるか、本文側に最低 1 つの定量情報（スクショ or CSS 値 or 根拠 PR へのリンク）を足します。

```markdown
## 観察: 11 箇所の 2 スペース改行を入れた記事の末路

私が書いた記事（約 5,500 字、note 公開版）に 11 箇所の 2 スペース改行を導入したところ、各プラットフォームのプレビューで以下の差が出ました。
```

定量化できる余地があれば、note 側の `<br />` 前後マージンを DevTools で 1 回計測して表に足すと、タイトルの「実測」と本文が合致します。

### 該当箇所 6
L150-L153 （「全角 5 行以上連続」の語が不自然）

```
- 本文中の `  $`（行末スペース）をカウント
- 全角 5 行以上連続で `<br />` が入るなら警告
- 空行による段落分けで代替可能かを確認
```

### 問題点
「全角 5 行」という言い方が日本語として不自然です。「全角」は通常文字の幅（全角／半角）を指す語で、行数を修飾する用法は一般的ではありません。また `  $` という正規表現は「末尾の 2 個のスペース」のつもりと思われますが、`$` 単体の前に何文字マッチさせるかの表記が曖昧で、L99 の `r" {2,}$"` と表記が揃っていません。**改善優先度: Low**（編集者視点の表現統一）。

### 提案
```markdown
- 本文中の行末 2 個以上のスペース（正規表現 ` {2,}$`）をカウント
- 5 行以上連続で `<br />` が入るなら警告
- 空行による段落分けで代替可能かを確認
```

### 該当箇所 7
L155-L160 （ログ文「可能性」が記事主張のトーンと乖離）

```python
# 簡易チェック
two_space_breaks = sum(1 for line in text.split("\n") if line.endswith("  "))
if two_space_breaks > 5:
    print(f"[WARN] 2 スペース改行 {two_space_breaks} 箇所。note で余白過多の可能性")
```

### 問題点
記事本文（L83, L165）では「note では `<br />` が段落級の余白になる」と断定している一方、サンプルコードのログ文は「可能性」と推測調です。記事主張とコードのトーンに乖離があり、読者に「結局断定できないのか」と受け取られます。また閾値 `5` の根拠が本文・コード双方で説明されていません。加えて `line.endswith("  ")` は末尾 2 文字のみチェックするため、行末がスペース 3 個で途中にタブ等がある極端なケースでは誤判定しうる点も、指摘 6 の正規表現統一と合わせて揃えるべきです。**改善優先度: Low**。

### 提案
```python
# 簡易チェック: 行末 2 個以上のスペース（= hard break）を検出
import re
two_space_breaks = sum(1 for line in text.split("\n") if re.search(r" {2,}$", line))
# 経験則: 5 箇所を超えるとスマホで余白過多が目立ちやすい
if two_space_breaks > 5:
    print(f"[WARN] 2 スペース改行 {two_space_breaks} 箇所。note では段落級の余白が発生します")
```

### 該当箇所 10
L141-L145 （sed の slug プレースホルダ `<slug>` がそのままでは動かない）

```bash
# macOS (BSD sed)
sed -i '' -E 's/ {2,}$//' articles_note/new/<slug>.md

# Linux (GNU sed)
sed -i -E 's/ {2,}$//' articles_note/new/<slug>.md
```

### 問題点
`<slug>` はプレースホルダ意図と読めますが、シェルで `<` はリダイレクト記号として解釈され、そのままコピーペーストすると構文エラーになります。読者がコードブロックをそのまま貼った際に動作しない可能性が高いため、表記を変えるか注釈を足すのが親切です。**改善優先度: Low**（読者の実行体験）。

### 提案
プレースホルダを山カッコではなく変数風 or 具体例に変えます。

```bash
# macOS (BSD sed) — SLUG=your-slug に置換して実行
sed -i '' -E 's/ {2,}$//' articles_note/new/your-slug.md

# Linux (GNU sed)
sed -i -E 's/ {2,}$//' articles_note/new/your-slug.md
```

もしくは直前に「`<slug>` は実際の slug に置き換えてください」と 1 行注記を足します。

## 総合評価

### 良い点
- **冒頭の `:::message` で記事の価値を先出し**（L9-L15）：3 つの持ち帰り事項が明確で、Zenn 読みやすさ観点 1 を満たす
- **比較表の冒頭提示が反映済み**（L59-L63）：前回指摘の「全体像を先に」が解消され、読者が比較結果に 1 画面で到達できる
- **観察ベースの断り書きが追加**（L65）：「本記事範囲外」「読者が自分の記事で再現確認するための指針」と明示され、推測と観察が分離されている
- **サンプルコードの仕様一致と OS 差明記が反映済み**（L97-L100, L141-L145）：`r" {2,}$"` / `sed -E 's/ {2,}$//'` で仕様と例が揃い、macOS/Linux 両記法が併記されている
- **一次体験に基づく説得力**（L19-L23）：実体験ベースの問題提起がフックとして機能
- **まとめ前に主張を再掲**（L162-L171）：読後の持ち帰りが明確

### 改善点（優先度順）
- **Medium**: 主要 Markdown 実装の挙動記述を条件付きに修正（指摘 1）
- **Medium**: 「段落内の空白改行」の曖昧表現を soft line break の説明に修正（指摘 2）
- **Medium**: note の CSS 設計意図への言及を観察ベースに修正（指摘 3）
- **Medium**: 「実測」見出しと本文トーンのギャップ解消 or 根拠 PR リンク追加（指摘 4, 5）
- **Low**: 「全角 5 行以上」の表現統一と正規表現の表記揃え（指摘 6）
- **Low**: サンプルコードのログ文「可能性」と本文断定トーンの統一（指摘 7）
- **Low**: タイトル末尾の英語 `semantics` とトピックタグ `python` の見直し（指摘 8）
- **Low**: 「事故／事故る」の口語重複解消（指摘 9）
- **Low**: sed 例の `<slug>` プレースホルダが実行時エラーになる問題を回避（指摘 10）

### 推奨アクション
1. **Medium 指摘 4 件を公開後の修正候補として反映**：すでに `published: true` のため、仕様記述の正確性（指摘 1, 2）と推測表現の回避（指摘 3）は信頼性担保の観点で優先
2. **「実測」見出し問題（指摘 4）は見出し変更 or 根拠リンク追加（指摘 5）の片方で解消可能**：編集コストが最も低い
3. **Low 指摘 5 件**：次回まとめて反映でよい（公開ブロッカーではない）
4. **公開判定**: High 指摘が残っていないため公開継続で問題なし。Medium 指摘は次の小改版タイミングで反映推奨

### SEO観点での改善提案
- **タイトル日本語化**: 末尾 `semantics` を「改行の扱い」に置き換えると、日本語検索での主題判定が安定する（指摘 8）
- **トピックタグの精度向上**: `python` を `commonmark` に差し替えると、仕様準拠で検索する読者に届きやすい
- **キーワードの早期出現**: タイトルにある「2 スペース改行」が本文冒頭（L19）で `  \n` と記号表記のみ。L19 直後の段落で「2 スペース改行」という日本語句を明示的に登場させると検索エンジン側の主題判定が安定する
- **関連記事への内部リンク**: 同リポジトリ内の note 運用系記事（note-export-import スキル紹介、記事レビュー関連）への相互リンクを追加すると、回遊性と SEO 双方に寄与
- **見出しの H2 数**: 現状 8 つで分量（約 160 行）に対して適量。`:::details` 折り畳みは不要
