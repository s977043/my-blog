## 指摘コメント

### 該当箇所 1
L2 / L32 / L117-L126 （タイトルの「4つの置き場所」と章立て・まとめの対応）

```
title: "AIが迷わないリポジトリ設計：長いプロンプトより先に整える4つの置き場所"
...
## 1. 入口ファイルは短くする
## 2. 危険領域には局所ルールを置く
## 3. 強制したいことは仕組みに寄せる
## 4. 変わる情報をMarkdownに抱え込まない
...
- 入口ファイル
- 局所ルール
- 強制する仕組み
- 正本と外部参照
```

### 問題点
タイトルは「4つの**置き場所**」と名詞でまとめているが、本文4章の見出しは「短くする」「置く」「寄せる」「抱え込まない」と**動詞ベースの原則/アクション**になっている。一方で「まとめ」章の箇条書き（L121-124）は「入口ファイル / 局所ルール / 強制する仕組み / 正本と外部参照」と**名詞**に戻っているため、タイトル・本文章タイトル・まとめの間で表現粒度がねじれ、読者がタイトル予告と章構成を対応づけにくい。特に `3. 強制したいことは仕組みに寄せる` は「置き場所」ではなく「どこに寄せるか」の方針で、タイトル語と噛み合いが弱い。

### 提案
章タイトルを「まとめ」側の名詞表現と揃えるか、もしくはまとめ側を動詞表現に統一する。例えば章タイトルを次のように調整すると、タイトル→目次→まとめが直線的に対応する。

```
## 1. 入口ファイル（AGENTS.md / CLAUDE.md）
## 2. 局所ルール（危険領域ごとの AGENTS.md）
## 3. 強制する仕組み（hooks / CI / ブランチ保護）
## 4. 正本と外部参照（変わる情報はリポジトリに固定しない）
```

これに合わせて TL;DR（L27-30）も「入口ファイル / 局所ルール / 強制する仕組み / 正本と外部参照」の 4 ラベル先出しに揃えると、タイトル→TL;DR→本文→まとめの 4 箇所で同じ語が繰り返されて認識が定着しやすくなる。

**優先度: High**（タイトルが記事の骨格を約束しているため）

### 該当箇所 2
L25-L30 と L103-L113 と L115-L126 （TL;DR / 最小チェックリスト / まとめの内容重複）

```
## TL;DR

- `AGENTS.md` や `CLAUDE.md` は入口に徹する
- 詳細ルールは局所ファイルや docs に逃がす
- 必ず守らせたい制約は hooks や CI に寄せる
- 変わる情報は手書きのMarkdownに固定しない
...
## 最小チェックリスト

AI向けにリポジトリを整えるなら、まずはこの7点を見ます。

- 入口ファイルが長くなりすぎていないか
- 目的、地図、検証コマンド、正本が分かるか
- 危険領域の局所ルールがあるか
- 必ず守る制約がCIやhooksに寄っているか
- 仕様、設計、タスクの置き場所が分かれているか
- 変わる情報をMarkdownに固定していないか
- AIが迷ったとき、どこを読めばよいか分かるか
...
## まとめ
...
- 入口ファイル
- 局所ルール
- 強制する仕組み
- 正本と外部参照
```

### 問題点
「TL;DR」「最小チェックリスト」「まとめ」の 3 箇所で、同じ 4 原則がほぼ同じ粒度で 3 回繰り返されている。137 行の短尺記事の中で 3 回の反復は密度が高く、特に「最小チェックリスト」は「TL;DR を疑問形に書き換えただけ」の印象を与える。役割分担が不明瞭。

### 提案
3 箇所の役割を明確に切り分ける。例えば次のように整理する。

- **TL;DR**: 本記事の結論（4 原則のラベルのみ）
- **最小チェックリスト**: 自分のリポジトリに当てはめて自己診断できる**疑問形**リスト（現状を維持）
- **まとめ**: TL;DR の再掲ではなく、「4 つを分けると何が変わるか」の**効果**に寄せて記述

具体的には、まとめ章 L121-124 の箇条書きを削除し、以下のように効果だけを述べて締める。

```markdown
## まとめ

AIが迷わないリポジトリを作るうえで、長いプロンプトは最初の答えではありません。

入口・局所・強制・正本の4層を分けておくだけで、AIは推測で進みにくくなり、セッションをまたいでも判断の再現性が保てます。

AIに何を言うかより先に、AIがどこを見ればよいかを設計する。
ここから始めると、AIコーディングの安定性はかなり変わります。
```

**優先度: High**（短尺記事での 3 回反復は読了体験を損なう）

### 該当箇所 4
L94 （MCP の初登場が未説明）

```
変わる情報は、GitHub、Notion、MCP、APIなど、その時点の正本から取りに行くほうが安全です。
```

### 問題点
`MCP`（Model Context Protocol）が本記事で初めて、かつ唯一登場する。`GitHub`、`Notion`、`API` と並列に列挙されているが、前 3 つは一般名詞で MCP は固有プロトコルであり、粒度が揃わない。また、トピックス（`ai`, `claudecode`, `codex`, `documentation`, `開発環境`）に MCP が含まれていないため、検索流入読者は MCP を前提にしていない可能性が高い。

### 提案
1 回だけの登場であれば、初登場時に1語だけ副題を添えるか、あるいは削除する。

```markdown
変わる情報は、GitHub、Notion、MCP（外部情報を AI に渡す標準プロトコル）、API など、その時点の正本から取りに行くほうが安全です。
```

あるいは、列挙から落として汎用化する。

```markdown
変わる情報は、GitHub や Notion、外部 API など、その時点の正本から取りに行くほうが安全です。
```

**優先度: Medium**（読者層によっては引っかかるが、本筋は伝わる）

### 該当箇所 7
L73 （章タイトルの硬さ）

```
## 3. 強制したいことは仕組みに寄せる
```

### 問題点
「強制したい」は書き言葉として硬く、他 3 章のタイトル（「入口ファイルは短くする」「危険領域には局所ルールを置く」「変わる情報をMarkdownに抱え込まない」）と比べて語感が浮く。また、章冒頭（L74-75）は「自然言語のルールは、あくまで文脈です。AIは読んで従おうとしますが、必ず守る保証にはなりません。」という narrative なので、章タイトルも「ルールではなく仕組みで守る」の方向に合わせたほうが、本文との接続が滑らかになる。

## 3. 守りたいルールは仕組みに寄せる

### 該当箇所 10
L77 / L79-L83 （箇条書きが体言止めと動詞止めの混在）

```
だから、必ず守らせたいことは hooks、CI、テスト、ブランチ保護などに寄せます。

- format を必ず走らせる
- 特定パス変更時にテストを走らせる
- secret を検出したら止める
- main への直接 push を防ぐ
- migration 変更時にレビューを必須にする
```

### 問題点
最後の「migration 変更時にレビューを必須にする」だけが「〜必須にする」という異なる終わり方で、他 4 つは「走らせる / 止める / 防ぐ」と自動アクションの動詞終止形で揃っている。また「必須にする」は運用設定であって、他項目のような「仕組みが走る」イメージと粒度が違う。

### 提案
自動アクションの動詞終止形に揃える。

```markdown
- format を必ず走らせる
- 特定パス変更時にテストを走らせる
- secret を検出したら止める
- main への直接 push を防ぐ
- migration 変更時にレビュー必須のラベルを付ける
```

あるいは

```markdown
- migration 変更時に人間のレビューで止める
```

**優先度: Low**（意味は通るが、箇条書きの終止形統一はリズムを整える）
