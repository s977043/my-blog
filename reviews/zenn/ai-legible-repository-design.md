## 指摘コメント

### 該当箇所 1
L2 / L32 / L46 / L74 / L89 / L121-L126 （タイトル「4つの置き場所」と章タイトル・まとめの表現粒度のねじれ）

```
title: "AIが迷わないリポジトリ設計：長いプロンプトより先に整える4つの置き場所"
...
## 1. 入口ファイルは短くする
## 2. 危険領域には局所ルールを置く
## 3. 強制したいことは仕組みに寄せる
## 4. 変わる情報をMarkdownに抱え込まない
...
- 入口ファイル
- 局所ルール
- 強制する仕組み
- 正本と外部参照
```

### 問題点
タイトルは「4つの**置き場所**」と名詞でまとめているが、本文 4 章の H2 は「短くする / 置く / 寄せる / 抱え込まない」の動詞ベースになっている。一方で「まとめ」章 L121-L126 の箇条書きは「入口ファイル / 局所ルール / 強制する仕組み / 正本と外部参照」と再び名詞に戻るため、タイトル → 章 H2 → まとめで表現粒度が 2 回反転する。特に `3. 強制したいことは仕組みに寄せる` は「どこに置くか」より「どう強制するか」の方針語で、タイトル語「置き場所」との呼応が弱い。前回レビュー（指摘 1）で同指摘をしているが本文には未反映で、タイトル語を本文 H2 で拾えていない状態が継続している。

### 提案
章 H2 を「まとめ」側の名詞ラベルに寄せる方向で統一する。タイトル → TL;DR → H2 → まとめの 4 箇所で同じ 4 ラベルが並ぶと、目次カードと本文の対応が直線的になる。

```
## 1. 入口ファイル（AGENTS.md / CLAUDE.md）
## 2. 局所ルール（危険領域ごとの AGENTS.md）
## 3. 強制する仕組み（hooks / CI / ブランチ保護）
## 4. 正本と外部参照（変わる情報はリポジトリに固定しない）
```

TL;DR（L27-L30）も対応を揃えると、Zenn の目次生成で「入口ファイル / 局所ルール / 強制する仕組み / 正本と外部参照」の 4 語が拾われ、SEO とクリック後の予測可能性が両方上がる。

**優先度: High**（タイトルが記事の骨格を約束しているため、ここが最優先）

```
## TL;DR

- 詳細ルールは局所ファイルや docs に逃がす

...

...
- 入口ファイル
- 局所ルール
- 強制する仕組み
- 正本と外部参照
```

### 問題点
139 行の短尺記事で、同じ 4 原則が「TL;DR（結論の先出し）」「最小チェックリスト（疑問形の自己診断）」「まとめ（名詞ラベル）」の 3 箇所でほぼ同じ粒度で繰り返されている。最小チェックリストは「TL;DR を疑問形に書き換えただけ」の印象を与え、まとめの箇条書きは TL;DR をラベル化しただけ。読者が 3 回同じ主張を読むことになり、短尺記事の密度を削いでいる。前回レビュー（指摘 2）で同指摘をしているが未反映。

### 提案
3 箇所の役割を切り分ける。具体的にはまとめ章の箇条書き（L121-L126）を削除し、効果だけを述べて締める構成にする。

```markdown
## まとめ

AIが迷わないリポジトリを作るうえで、長いプロンプトは最初の答えではありません。

入口・局所・強制・正本の4層を分けておくだけで、AIは推測で進みにくくなり、セッションをまたいでも判断の再現性が保てます。

AIに何を言うかより先に、AIがどこを見ればよいかを設計する。
ここから始めると、AIコーディングの安定性はかなり変わります。
```

この形にすると、TL;DR = 結論の先出し / 最小チェックリスト = 自己診断 / まとめ = 効果の再確認、と役割が切り分けられる。

**優先度: High**（短尺記事での 3 回反復は読了体験を直接損なう）

### 問題点
前回レビュー（指摘 3）を受けて `spec/` の補足が追加された点は改善だが、新たに `PBI`（Product Backlog Item）という略語が無説明で登場している。記事のトピックス（`ai`, `claudecode`, `codex`, `documentation`, `開発環境`）にはスクラム文脈は含まれておらず、AI コーディング実装者の多くが PBI を当然の語彙として扱うとは限らない。また、`docs/`（長く効く設計知識）という括弧書きは簡潔で良いが、`spec/` 側の括弧書きが無く粒度が揃っていない。

### 提案
PBI を避けて「現在進行中の作業単位」に言い換えるか、カッコ補足を双方に足して粒度を揃える。

```markdown
`spec/` は、`docs/`（長く効く設計知識）と分けて、`spec/`（現在進行中の作業単位ごとの要件・設計・タスク）を置く場所として切り出しています。
```

あるいはスクラム文脈を残したいなら副題で補う。

```markdown
`spec/` は、`docs/`（長く効く設計知識）と分けて、「今動いている PBI（Product Backlog Item：開発単位の作業チケット）の要件・設計・タスク」を置く場所として切り出しています。
```

**優先度: Medium**（前回指摘の修正過程で発生した新規指摘）

### 該当箇所 7
L91-L93 （「PRの状態、Issueの担当者、外部サービスの現在値」の列挙に具体性のバラつき）

```
PRの状態、Issueの担当者、外部サービスの現在値、APIのレスポンスなどは変わります。

こうした情報を手書きのMarkdownに固定すると、すぐ古くなります。
```

### 問題点
列挙 4 項目のうち、「PR の状態」「Issue の担当者」「API のレスポンス」は具体的な粒度で並ぶが、「外部サービスの現在値」だけが抽象度が高く、読者が何を指すか推測しづらい（価格 API / 残高 API / フラグ系? など）。L100-L101 の code block 行「変わるもの: PR状態 / Issue状態 / 外部APIの現在値 / 本番データ」とも「外部APIの現在値」で表現が少しズレる（こちらは API 限定）。

### 提案
列挙を API 側に統合するか、具体例で補強する。

```markdown
PRの状態、Issueの担当者、外部APIのレスポンス、本番データなどは変わります。
```

もしくは

```markdown
PRの状態、Issueの担当者、外部サービスの現在値（料金・在庫・フラグなど）、APIのレスポンスなどは変わります。
```

L100-L101 のコードブロック表記と語彙を揃えると、読者が 2 箇所を往復しても同じ項目として認識できる。

**優先度: Low**（意味は通るが、narrative の精度を上げる）

### 該当箇所 8
L35-L40 / L42-L43 （「入口ファイルに置きたいもの」箇条書きと「逆に」以降の非対称）

```
- このリポジトリは何をするものか
- 主要ディレクトリの責務は何か
- どのコマンドで検証するか
- どのファイルが正本か
- 触ってはいけない領域はどこか

逆に、たまにしか使わない手順や、特定領域だけの細かい注意点まで入れると、入口が重くなります。
```

### 問題点
「入口に置きたいもの」は疑問形 5 項目で具体化されているが、「逆に（置かない方がよいもの）」は「たまにしか使わない手順や、特定領域だけの細かい注意点」の一文に圧縮されている。前者が疑問形・後者が名詞句という非対称があり、「入口ファイルは地図」という比喩（L44）の具体性に差が出る。また、読者は「何を置くか」よりむしろ「何を置かないか」で迷うことが多いため、置かない側の例示が薄いと自己判断に接続しにくい。

### 提案
「逆に」側を箇条書きに揃え、例示を加える。

```markdown
逆に、次のようなものは入口に置かず、局所ファイルや docs に逃がします。

- 特定領域だけのデプロイ/ロールバック手順
- 外部サービスごとの API キー取得フロー
- 月に一度しか触らない運用タスクの詳細
- エラーコード一覧のような参照用データ

これらまで入れると、入口が重くなります。
```

もしくは現行の 1 文のまま、1 つだけ具体例を添える。

```markdown
逆に、たまにしか使わない手順（例: 四半期ごとのキーローテーション）や、特定領域だけの細かい注意点まで入れると、入口が重くなります。
```

**優先度: Low**（対称性の問題で、本筋の理解を妨げるものではない）
