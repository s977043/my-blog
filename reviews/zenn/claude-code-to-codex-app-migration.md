# claude-code-to-codex-app-migration レビュー記録

対象: `articles/claude-code-to-codex-app-migration.md`  
目的: Zenn向け記事として、読者訴求を落とさずに実務性・正確性・読みやすさを強化する。

## Loop 1: セルフレビュー

### 観点

- 冒頭で読者の課題が明確か
- 記事を読むメリットが早い段階で伝わるか
- 移行対象の全体像が分かるか
- 単なる感想ではなく、実務手順になっているか

### 指摘

1. 冒頭の訴求が弱く、「この記事で何が得られるか」が曖昧だった。
2. 移行対象の分類が後半まで出てこず、読者が全体像を掴みにくかった。
3. 自動インポート後に何をレビューするかがやや抽象的だった。
4. Zenn記事として、最初に `:::message` で得られることを出した方が読み進めやすい。

### 反映

- 冒頭に「この記事で得られること」を追加。
- TL;DRを強化。
- 「ほぼ移せる / 確認して移す / 設計し直す」の3分類を追加。
- 移行対応マップを早い位置に移動。
- 「完全互換ではない」注意書きを追加。

## Loop 2: 複数視点レビュー

### A. Webディレクター視点

#### 指摘

- 想定読者は明確だが、読者がすぐ行動できる導線がもう少し欲しい。
- 「何から試せばよいか」が最後のチェックリストまで行かないと見えにくい。
- タイトルは十分強いが、本文内の導入で「半日で試せる」粒度があると訴求が上がる。

#### 反映

- 「半日で試す移行手順」を追加。
- 小さいrepo、自動インポート、AGENTS.md、Skill、MCP、Worktreeの順で実行導線を明確化。
- 本格移行ではなく、小さい検証から始める提案を強化。

### B. Web編集者視点

#### 指摘

- 同じ意味の文章が一部重複していた。
- 「安全」「現実的」が多く、少し抽象的に見える箇所があった。
- Markdown内の入れ子コードフェンスが壊れる可能性があった。

#### 反映

- 重複表現を整理。
- 抽象語の後に具体例や表を置く構成へ変更。
- `SKILL.md` のサンプルやSkill化例は4連バッククォートで囲み、Zenn上でコードブロックが崩れないよう修正。

### C. Webエンジニア読者視点

#### 指摘

- MCPのトークン例は、ハードコードを助長しない注意が必要。
- SubagentsのTOML例は、バージョン差分で仕様が変わる可能性があるため、断定しすぎない方がよい。
- Hooksはイベント名の対応表だけでなく、sandbox / approvalとの組み合わせを強調した方が実務的。

#### 反映

- MCPトークン例に「構造例」「実運用では直書きしない」注意を追加。
- Custom agents例を「概念例」として扱い、実際のキーは手元のCodex環境で確認する注記を追加。
- Hooks章で、Hook単体ではなくsandbox / approval / permissionsとの組み合わせで安全性を設計する流れを強化。

## Loop 3: 最終レビュー

### 観点

- 読者訴求は落ちていないか
- 仕様変更に弱い断定が残っていないか
- Zenn Markdownとして壊れないか
- リポジトリ規約に合っているか

### 指摘

1. 「移行できる」と断定しすぎる箇所は、仕様変更時に古く見える。
2. `Custom agents` の例は便利だが、正確なスキーマの断定に見えると危険。
3. Zennのslug規約上、ファイル名は半角英小文字とハイフンで維持したい。
4. 記事の最後は、読者が次にやることを明確にした方が強い。
5. Hooksの承認イベントは実装状況の差分があり得るため、同名移植できる前提を避けるべき。

### 反映

- 「移行できる」を「移行しやすい」「確認する」に調整。
- Custom agents例に `Example only` コメントを追加。
- slugを `claude-code-to-codex-app-migration` に固定。
- まとめを5ステップの実行順に変更。
- Hooks章の承認要求まわりを `PermissionRequest系` 断定から、approval / sandbox / permissionsで再設計する表現へ修正。
- `published: false` の下書き状態でPR化。

## 最終判断

公開前ドラフトとして妥当。  
特に、以下の価値が出ている。

- Claude Codeユーザーが迷う移行対象を網羅している
- 自動インポートを入口にしているため、読者の心理的負担が低い
- Subagents / Hooks / 自動化を「再設計」として扱っており、誤解が少ない
- 半日検証手順とチェックリストがあり、実務に移しやすい
