### 該当箇所 8
L86-L92 （セルフチェック 5 項目の粒度のばらつき）

```
1. **根拠は実測か、推測か**
2. **推測なら、そう明示したか**（「〜と思われる」「〜の可能性が高い」）
3. **実測なら、再現方法を書いたか**（対象ファイル、コマンド、出力）
4. **関連する既存エントリとの矛盾はないか**（矛盾する場合は訂正として追記）
5. **リンクすべき PR / コミットを引用したか**
```

### 問題点
チェック項目 1〜3 は「根拠の種類と表現」という同一テーマで 3 つ並び、4 は「既存エントリとの整合性」、5 は「外部リンク」と、**論点の粒度が揃っていない**。読者がセルフチェックリストとして使う場合、カテゴリ分けがあると運用しやすい。例えば「根拠の明示」「既存との整合」「参照の明示」の 3 カテゴリに分けると、追記時のフロー（自問の順序）が明確になる。

### 提案
3 カテゴリにグルーピングする。

```markdown
**A. 根拠の明示**

1. 根拠は実測か、推測か
2. 推測なら、そう明示したか（「〜と思われる」「〜の可能性が高い」）
3. 実測なら、再現方法を書いたか（対象ファイル、コマンド、出力）

**B. 既存との整合**

4. 関連する既存エントリとの矛盾はないか（矛盾する場合は訂正として追記）

**C. 参照の明示**

5. リンクすべき PR / コミットを引用したか
```

**優先度: Low**（運用しやすさ、読者体験の改善）

### 該当箇所 9

```
## エントリ形式のテンプレート

実運用しているテンプレは次のとおり。

```markdown
### YYYY-MM-DD — <短い見出し> [カテゴリ]

**観察**: 何が起きたか。事実ベース。

**対策/学び**: どう対応したか、または今後どうするか。

**根拠**: 参照 PR / Issue / commit（あれば）。推測なら明示。
```
```

### 問題点
テンプレートは `観察 / 対策・学び / 根拠` の 3 フィールド構成。一方、L55-L71 の実エントリ例（打ち消し線 Markdown 例）は `観察 / 根拠` の 2 フィールドで、`対策/学び` フィールドが無い。テンプレートと実例の不一致で、読者が「テンプレートのうち `対策/学び` は省略可能なのか必須なのか」を判断できない。

### 提案
実例（L55-L71）側に `対策/学び` を補うか、テンプレート側に「`対策/学び` は訂正エントリでは省略可」と注記する。

```markdown
## エントリ形式のテンプレート

実運用しているテンプレは次のとおり。訂正エントリの場合は `対策/学び` を省略し、`観察` と `根拠` のみで残すことが多いです。

`AGENT_LEARNINGS.md` は本プロジェクト固有のファイル名で、一般の Zenn 検索／Google 検索の語としては流入が期待しにくい。`topics` に `ナレッジマネジメント` / `ai駆動開発` / `claudecode` と定義されているのに対して、タイトルが具体ファイル名に寄りすぎており、検索語とのマッチが弱い。同著者の `engineering-process-improvement-skill.md` では「開発セッション直後の振り返りを改善フローにつなげる方法」のように、`AGENT_LEARNINGS.md` という固有名を使わずに検索語寄りにしている。

### 提案
タイトル副題に一般検索語を加える。本文は変更せず front matter のみの変更。

```yaml
title: "推測で書いた学びが次の罠になった — AI エージェント向け lessons-learned（AGENT_LEARNINGS.md）の運用設計"
```

あるいは、主題を検索語寄りに入れ替え。

```yaml
title: "AI エージェント向け lessons-learned の運用設計 — 推測で書いた学びが次の罠になった話"
```
