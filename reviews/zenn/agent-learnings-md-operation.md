```markdown
## 訂正の残し方

この時、AGENT_LEARNINGS.md をどう修正するかで迷いました。

- 前のエントリを **削除** する → 履歴が残らず、なぜ書き換えたかが不明
- 前のエントリを **残したまま訂正エントリを追加** する → 2 つの記述が並んで読み手が混乱
```

### 該当箇所 9
L117-L126 （「書かないこと」の境界 — セルフチェックとの重複感）

```markdown
AGENT_LEARNINGS.md は **追記型**。古いエントリも履歴として残し、訂正が必要なら打ち消し線で明示。削除はしないのが原則です。

- 同じ罠を 2 回以上踏んだら追記
- 特定の成功パターンが再現できると確認できたら追記
- ユーザーから「次回はこうして」と明示的な指示があったら追記

逆に、**書かないこと** は:

- コードを読めば分かる事実（ファイルパスや関数名の羅列）
- 一度きりのタスクの詳細（メモリや会話で完結する内容）
```

### 問題点
このセクション「エントリの育て方」が、先行する L84「追記時のセルフチェック 5 項目」と **追記の粒度ルールで一部重複** している。具体的には:
- L90「関連する既存エントリとの矛盾はないか」とこの節の「打ち消し線で訂正」は同主旨
- L92「リンクすべき PR / コミットを引用したか」とこの節の「書かないこと」は補完関係だが、記事中で章が分かれて散在

Zenn 読みやすさ観点（重複表現・構成）に該当。章の役割分担を明示するか、統合するとコンパクトになる。

### 提案
L115-L126「エントリの育て方」の冒頭に、セルフチェック章との役割分担を 1 行で示す。

```markdown
## エントリの育て方

セルフチェック 5 項目が **「書く時の粒度」** を決めるのに対し、このセクションでは **「そもそも追記すべきか／書くべきでないか」** の入口判断を扱います。
```

**優先度: Medium**（重複感を役割分担の明示で解消、Issue #11「内容重複」観点）

記事が まとめ（L136）で終了しており、著者の既出記事で定着している **関連記事セクションが無い**:
- `engineering-process-improvement-skill.md` L351-L356 は「関連記事」として 3 本へリンク
- `design-md-guide-and-adoption-log.md` L412-L414 は「関連ページ」を明示
- `ai-legible-repository-design.md` L133-L138 は「参考」で 4 本へリンク

本記事の主題（AI エージェント向けドキュメント運用）は、既出記事と密接に関連する:
- `ai-legible-repository-design.md`（AI が迷わないリポジトリ設計 — `AGENTS.md` / `CLAUDE.md` の入口整備）
- `engineering-process-improvement-skill.md`（振り返り → 改善 PR 化 — `AGENT_LEARNINGS.md` への追記運用）
- `note-wxr-import-structural-validation.md`（本記事の題材となった WXR import エラーの関連記事）

関連記事導線が無いと、Zenn 内部 PV 循環・SEO（内部リンク）・読者の次アクションいずれにもマイナス。

### 提案


## 総合評価

### 良い点
- 冒頭の `:::message`「この記事で得られること」（L9-L15）が、著者の `engineering-process-improvement-skill.md` と揃っており、読者が 10 秒で記事の価値を判断できる構成になっている
- 見出しの言い回しが柔らかい（「訂正の残し方」「エントリの育て方」など、Zenn 読みやすさ観点 #8 を満たす）
- 「推測 → 失敗 → 訂正 → 仕組み化」という narrative の流れが明快で、実体験ベースの説得力がある
- L37-L40 の比較表が短語・3 カラムで視認性が高く、記事の核心（WordPress 慣例と note 実装の逆転）を端的に示している
- まとめ（L128-L136）で 3 点に再集約しており、Zenn 読みやすさ観点 #7（まとめ前の主張再掲）が満たされている

### 改善点
- **全体像の提示不足**: はじめに直後にロードマップが無く、7 章続く構成で読者が現在位置を見失いやすい（指摘 1）
- **`type: "idea"`** のままだと `tech` 記事層に届かない。他の同テーマ記事と揃えて `tech` 推奨（指摘 2）
- **Markdown 構文の崩れ**: L97 の入れ子バッククォートは Zenn 上でレンダリング崩れの可能性が高い（指摘 4）
- **関連記事セクションの欠落**: 著者の既出記事と構成が一貫せず、SEO 内部リンクも失われている（指摘 10）
- **参照 PR がリンク化されていない**: 記事主張（実測根拠）と記事自身の実践が矛盾する（指摘 5）

### 推奨アクション
1. **L4 `type: "idea"` → `"tech"` に変更**（指摘 2）— 最も影響範囲が広く、かつ変更コスト最小
2. **L97 の入れ子バッククォートを修正**（指摘 4）— レンダリング崩れを避ける
3. **記事末尾に関連記事セクションを追加**（指摘 10）— 既出 3 記事への導線を確保
4. **L69 の PR #62 をリンク化**（指摘 5）— 記事主張との整合
5. **L23 と L25 の間にロードマップを追加**（指摘 1）— 読者の現在位置把握
6. （余裕があれば）指摘 3, 6, 7, 9 をまとめて反映

### SEO観点での改善提案
- **タイトル**: `"推測で書いた学びが次の罠になった — AGENT_LEARNINGS.md の運用設計"` は訴求力が高いが、`AGENT_LEARNINGS.md` は一般検索語ではない。`topics` の `ナレッジマネジメント` をキーワードとして拡張し、タイトル副題に「AI エージェント lessons-learned の運用」等を補うと流入幅が広がる可能性あり
- **`type: "idea"` → `"tech"`**（指摘 2 と同じ）により Zenn の `tech` トレンド面露出が得られる
- **関連記事の内部リンク**（指摘 10）により、同一著者の記事群が Zenn 内で循環し、ドメイン滞在時間が伸びる
- **参照 PR / 公式エクスポートファイルへの外部リンク**（指摘 5）により、一次情報との接続が Zenn 評価指標で加点されやすい
- `topics` 構成は妥当（`ai駆動開発` / `claudecode` / `ドキュメント` / `ナレッジマネジメント` / `チーム開発`）。ここは変更不要
