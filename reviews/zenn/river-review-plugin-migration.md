### 該当箇所 1（Round 1 #2 / #9 継続：blocker 候補）
L68-L74（Codex 章のコマンド対称性）

```
codex plugin marketplace add s977043/river-review
codex plugin install river-review@river-review-marketplace
codex plugin marketplace add s977043/river-review@v1.2.1  # バージョン pin する場合
```

### 問題点
Round 1 #2 で指摘した「Claude Code は 3 ステップ（add → install → reload）、Codex は 2 ステップ + pin 例で対称性が崩れている」が未対応。L58 で「v1.2.1 で動作確認した」というエビデンス（#9 採用）は加わったが、コマンドブロック自体は変更されていないため、L76 の「**同じ流れが Claude Code と Codex で揃っている**」という主張と本文上の見た目が乖離する状態が続いている。公開後に「Codex の 3 行目がなぜ marketplace add の再実行なのか」という読者の疑問が直接届く形なので、**公開前に解消すべき blocker** と判定。

### 提案
```bash
codex plugin marketplace add s977043/river-review
codex plugin install river-review@river-review-marketplace
codex plugin reload river-review   # Codex 側に reload 相当があれば
```

その上で pin 例は L78 の引用ブロックに統合：

```
> リリースタグでバージョンを pin する運用は、Claude Code なら `/plugin install river-review@v1.2.1`、Codex なら `codex plugin marketplace add s977043/river-review@v1.2.1` のようにタグを付けます。プロダクションでは pin が安全です。
```

Codex に reload コマンドが存在しないなら、L76 の対称性主張を「**ほぼ揃っている（Codex は reload 不要）**」に弱める。

### 問題点
Round 1 では「シナリオごとの使い分け表」を良い点として評価したが、再読すると **3 行目「スキルのマーケットプレイス + バージョニング / 両方」が他 3 行と粒度が違う**ことに気付く。他 3 行は「利用ユースケース → 向くツール」だが、3 行目は「機能 → どちらも対応」で対称性が崩れている。L95「使い分けは**配布側の設計**ではなく**利用側のユースケース**の話に分離される」という宣言と矛盾する（Webディレクター視点・整合性）。

### 提案
3 行目を表から削除し、本文の前置きで「マーケットプレイスとバージョニングは両ツール共通」と 1 行先出しする。例：

```
同じスキルパックを使えるとはいえ、**何に向くか**は違います。マーケットプレイスとバージョニングは Claude Code / Codex 共通なので、ユースケース側の差だけを表に整理します。
```
