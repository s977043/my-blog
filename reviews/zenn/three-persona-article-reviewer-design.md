### 問題点

### 該当箇所 2
L29-L33 （記事本文のペルソナ定義と、実装ソース `.claude/agents/article-reviewer.md` のペルソナ定義が一致していない）

```markdown
### Zenn 向け 3 ペルソナ

```

### 問題点
実装ソース `.claude/agents/article-reviewer.md` の「3 ペルソナ」セクションでは以下のように定義されている。

```
- a) Webディレクター視点: 構成・読みやすさ・対象読者整合性・SEO最適化
- b) Webサイト編集者視点: 誤字脱字・表現統一・文章明確性・重複表現
- c) Webエンジニア読者視点: 技術的正確性・コード検証・実装可能性
```

記事側（L31-L33）では、ディレクターに「タイトル訴求 / 目次設計」、編集者に「Zenn 表現規約（Front Matter / 言語指定 / :::message）」が追加されている。L114-L118 で実装リファレンスへのリンクが追加済みのため、読者がクリックした先でペルソナ定義のズレに直面する。仕組み化を主張する記事で「記事と実装の差異」があると、再現性の信頼が下がる。

### 提案
以下のどちらかで対応する。

- **Option A**: 記事側を実装ソースに合わせて最小定義に戻し、拡張分（タイトル訴求・Zenn 表現規約など）は「本記事ではこれを拡張した運用を取る」と 1 行で明示する。
- **Option B**: 実装ソース `.claude/agents/article-reviewer.md` を記事の定義に合わせて更新し、両者の整合を取る。

```markdown
# Option A（記事側を実装に合わせる）の例
### Zenn 向け 3 ペルソナ（実装ソース準拠）

- **Webディレクター視点**: 構成・読みやすさ・対象読者整合性・SEO最適化
- **Webサイト編集者視点**: 誤字脱字・表現統一・文章明確性・重複表現
- **Webエンジニア読者視点**: 技術的正確性・コード検証・実装可能性

本記事の運用では、ディレクター視点に「タイトル訴求・目次設計」、編集者視点に「Zenn 表現規約（Front Matter / 言語指定 / :::message）」を拡張している。
```

**優先度: High**（記事末尾の実装リファレンス導線と組み合わさり、ズレが顕在化する）

### 該当箇所 6
L19-L23 （はじめに章で、他 LLM と内製エージェントの切り分け背景が不足）

```markdown
## はじめに

技術記事を書くたびに ChatGPT や Gemini に構成レビューをもらっていましたが、**同じような指摘を毎回もらっている** ことに気づきました。

「冒頭に要約ブロックを入れたほうが良い」「詳細の前に全体像を」「英語用語に日本語副題を」――。これらは一度ルール化してしまえば、次の記事から自動適用できるはずです。

この記事は、**外部 LLM の一回性レビューを、内製の reviewer agent（記事レビュー用エージェント）に観点として移植する** アプローチを共有するものです。
```

### 問題点
「同じような指摘を毎回もらっている」という観察が起点だが、具体的に何回くらい頼んで何%が重複していたか、どんな指摘が重複していたか、の根拠提示がなく「気づき」のまま話が進む。Webディレクター視点では記事の導入部に「課題の定量感」が欲しいところ。また、「一回性レビュー」という表現は初出で、「一回性」が「1 記事につき 1 回 ChatGPT にお願いしている」の意か「毎回対話履歴がリセットされる」の意か曖昧。

### 提案
導入で「重複率の体感」と「一回性」の意味を 1 行ずつ補足する。

```markdown
技術記事を書くたびに ChatGPT や Gemini に構成レビューをもらっていましたが、**同じような指摘を毎回もらっている** ことに気づきました。ざっくり 3〜4 本レビューを通すと、8 割は冒頭要約・全体像・英語用語副題といった定型指摘で、残り 2 割だけが記事固有の深い指摘でした。

「冒頭に要約ブロックを入れたほうが良い」「詳細の前に全体像を」「英語用語に日本語副題を」――。これらは一度ルール化してしまえば、次の記事から自動適用できるはずです。

この記事は、**外部 LLM の一回性レビュー（記事ごとに対話履歴がリセットされ、過去の指摘が蓄積されない形式）を、内製の reviewer agent（記事レビュー用エージェント）に観点として移植する** アプローチを共有するものです。
```

**優先度: Low**（主張の補強、数値は体感値でも「課題感」の伝達に効く）

### 該当箇所 7
L43-L46 （「観点の構造化」章の導入が短く、4 カテゴリの由来が本文で説明されない）

```markdown
## 観点の構造化（ChatGPT レビューから移植）

ChatGPT に記事を見てもらった時の指摘を「構成 / 圧縮 / 段落・セクション / 表現」の 4 カテゴリに分類し、**Zenn 8 観点 + note 10 観点** として番号付きで整理しました。
```

### 問題点
4 カテゴリ名（構成 / 圧縮 / 段落・セクション / 表現）が唐突に登場するが、なぜこの 4 つなのか、どうやって分類したのかの説明がない。`.claude/agents/article-reviewer.md` の「Zenn 読みやすさ観点（構成と圧縮）」セクションではこれらカテゴリが使われており、実装ソース側の分類を踏襲していると思われるが、記事単体では分類の意図が掴めない。

### 提案
4 カテゴリの役割を 1 行ずつ補足する。

```markdown
## 観点の構造化（ChatGPT レビューから移植）

ChatGPT に記事を見てもらった時の指摘を、次の 4 カテゴリに分類しました。

- **構成**: 価値を前に、詳細を後ろに、派生論点は末尾に（記事の骨格）
- **圧縮**: コード断片で本文を埋めず、narrative を優先する（読み負荷の軽減）
```
