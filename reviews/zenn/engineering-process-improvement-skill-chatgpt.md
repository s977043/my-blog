## 必須対応（優先度 A）

### A-1. 想定読者・前提条件を `:::message` で明示

**追加位置**: 導入（はじめに）直後
**理由**: Claude Code / AI 支援開発の前提文脈が自然に出てくるため、未経験読者が入口で迷う

**追加文**:

```markdown
:::message
想定読者
- Claude Code など AI 支援開発を日常的に使っている人
- 開発セッション直後の振り返りを仕組みに変えたい人
- Git / GitHub CLI ベースで改善を PR として残せる人

前提
- `gh` コマンドが使える
- 改善内容を PR として残せる
- 運用ドキュメントをリポジトリに置ける
:::
```

### A-2. コアメッセージを `:::message` で先出し

**追加位置**: 「この記事で得られること」の直後、8 ステップ全体像の直前
**理由**: 読者が「本当に覚えるべきこと」を一目で掴めない

**追加文**:

```markdown
:::message
この運用のコアは次の 4 点です。
1. 振り返りを感想で終わらせない
2. 改善を A / B / C に分類する
3. 最低 1 件は B（自動ガード）に落とす
4. 改善を PR としてチェーンで残す
:::
```

### A-3. 長いコマンド例・補足を `:::details` に畳む

**対象**（重要度順）:
- Step 1 のコマンド例（gh pr list / git log スニペット）
- Step 5 の Skill / ガード実装例
- Step 7 の PR 運用例（PR 本文テンプレート）
- meta-retrospective の詳細な運用メモ

**記法例**:

```markdown
:::details Step 1 のコマンド例
\`\`\`bash
gh pr list --search "merged:>=2026-04-01"
git log --oneline --since="3 days ago"
\`\`\`
:::
```

### A-4. 8 ステップ直後に中間まとめ

**追加位置**: Step 8 の直後、「Anti-patterns / Quality Gates」 章の手前
**理由**: 8 ステップは情報量が多く、読者が「ここまでで何を押さえたか」を再確認できる節目が必要

**追加文**:

```markdown
:::message
ここまでの要点
- 範囲を固定する
- 振り返りを A / B / C に分ける
- 最低 1 件は B（自動ガード）にする
- 改善内容を PR として残す
:::
```

## 推奨対応（優先度 B）

### B-5. 各 Step 直下に 1 〜 2 文の目的要約

**対象**: Step 1 〜 Step 8 の全セクション
**理由**: スクロール読みしやすくする

**例**:

```markdown
## Step 1: 振り返る範囲を固定する

このステップの目的は、振り返り対象を「最近の気分」ではなく、特定の開発セッションに固定することです。
```

### B-6. Quality Gates をチェックリスト/表形式に

**対象**: 既存の Quality Gates 章
**理由**: 連続テキストだと流し読みされる

**修正案（表）**:

```markdown
| Gate | 確認内容 |
| ---- | ---- |
| Scope | 対象セッションが固定されているか |
| Output | 改善案が A/B/C に分類されているか |
| Action | 最低 1 件が B に落ちているか |
| Trace | 改善 PR へのリンクが残っているか |
```

### B-7. 著者固有文脈の一般化

**対象**: 具体的な commit ID / 内部セッション ID など
**理由**: 外部読者にとって処理コストが高い

**例**:

- Before: `12d6e28` のような固有 SHA が本文にある場合
- After: 「あるセッションでは、競合回避のために stash と checkout --theirs を使う迂回が発生した」のように一般化

**注意**: 具体性は失わない。事例は残しつつ一般化する。

### B-8. 最小導入セクションを末尾に追加

**追加位置**: まとめの後、関連記事の手前
**理由**: 実践記事として「まず何から始めるか」を提示すると持ち帰りやすい

**追加文**:

```markdown
## 最小導入で始めるなら

最初から全部やらなくても、次の 3 点だけで十分効果があります。

1. 振り返り対象を直前セッションに固定する
2. 改善を A / B / C に分類する
3. 最低 1 件は B（自動ガード）にする
```

## 禁止事項

- 内容を削りすぎて再現可能性を落とさない
- 具体性を失って抽象論だけにしない
- `:::details` に詰め込みすぎて本文がスカスカにならないようにする
- 著者固有の経験を全部消さない（本文では一般化で残す）

---

## 完成イメージ

編集後の記事は以下の読み心地を目指す:

- 冒頭 30 秒で「誰向けか」「何が得られるか」「何が核か」が分かる
- 本文は要点中心で読める
- 詳細は必要な人だけ開ける
- 長文でもスクロール時に迷子にならない
- 読後に「明日から何をやればいいか」が残る
