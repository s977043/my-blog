### 該当箇所 2
L64-L77 （エージェントBの実装疑似コード）

```javascript
// エージェントAの実装
const isWeekday = (date) => {
  const day = date.getDay();
  return day >= 1 && day <= 5; // 月-金のみ
};

// エージェントBの実装
const isWeekday = (date) => {
  return !isHoliday(date) && isBusinessDay(date); // 祝日除外
};

// 結果：同じ「平日」でも動作が異なる
```

### 問題点
エージェントBの実装は `!isHoliday(date) && isBusinessDay(date)` となっているが、この論理は意味的に冗長（`isBusinessDay` が「営業日」を意味するなら `isHoliday` 判定を内包しているのが通常）。さらに、エージェントAとBの本質的な差分は「曜日ベースか／営業日カレンダーベースか」であるはずが、ロジック上は土日判定がエージェントBから欠落している（`isBusinessDay` が土日を除外するかどうかが示されない）。技術例示として不正確で、読者が「営業日 = 祝日を除く」と誤解する可能性がある。

### 提案
```javascript
// エージェントAの実装：曜日ベース
const isWeekday = (date) => {
  const day = date.getDay();
  return day >= 1 && day <= 5; // 月-金のみ（祝日は考慮しない）
};

// エージェントBの実装：営業日カレンダーベース
const isWeekday = (date) => {
  const day = date.getDay();
  const isMonToFri = day >= 1 && day <= 5;
  return isMonToFri && !isHoliday(date); // 月-金かつ祝日除外
};

// 結果：祝日の扱いで動作が分岐する
```

### 該当箇所 3
L148-L173 （OpenAPI 仕様のサンプル）

```yaml
paths:
  /alarm:
    post:
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              properties:
                weekday_only:
                  type: boolean
                  description: "平日のみ（月-金、祝日除外）"
                alarm_time:
                  type: string
                  format: time
                  example: "07:00:00"
                  description: "アラーム時刻（HH:MM:SS形式）"
              required:
                - weekday_only
                - alarm_time
```

### 問題点
OpenAPI 3.x では `format: time` はOpenAPI仕様そのものには正式には定義されていない（標準的なものは `date`、`date-time`、`password`、`byte`、`binary` など）。`time` は一部ツールで拡張として解釈されるが、仕様書として記述するなら `pattern` による明示か、文字列として `description` に制約を書くのが安全。また `required` が `properties` の外にネストされていない（`schema` 直下ではなく `properties` 直下のように見える位置）で記述されているが、YAMLインデントが紛らわしい。

### 提案
```yaml
paths:
  /alarm:
    post:
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required:
                - weekday_only
                - alarm_time
              properties:
                weekday_only:
                  type: boolean
                  description: "平日のみ（月-金、祝日除外）"
                alarm_time:
                  type: string
                  pattern: "^([01]\\d|2[0-3]):[0-5]\\d:[0-5]\\d$"
                  example: "07:00:00"
                  description: "アラーム時刻（HH:MM:SS形式）"
      responses:
        "200":
          description: "アラーム設定成功"
```

### 該当箇所 4
L221-L235 （原則3「ノンブロッキング」の定義と「承認」の矛盾）

```markdown
## 原則 3: ノンブロッキング（人間は進行を止めない）

...
人間はまず**要件を伝える**。エージェントが仕様にまとめ、必要に応じて調整を行う。確認はするが、**承認待ちのようなブロック要素にはしない**。

最終的に人間がレビューを行うのは、次の条件がすでに満たされた後です。
```

### 問題点
「承認待ちのようなブロック要素にはしない」と言いながら、L228では「最終的に人間がレビューを行う」ことを規定している。最終レビュー自体は承認/非承認の分岐点であり、厳密にはこれもブロック要素になりうる。読者が「結局どこで人間が止めるのか／止めないのか」を混乱しやすい。用語として「ブロッキング」「承認待ち」「最終レビュー」の関係が整理されていない。

### 提案
「途中のブロッキングは作らないが、最終レビューは置く」というニュアンスを明示する。

```markdown
## 原則 3: ノンブロッキング（人間は進行を途中で止めない）

人間はまず**要件を伝える**。エージェントが仕様にまとめ、必要に応じて調整を行う。
途中のマイクロな承認やレビュー待ちは作らず、**実装の進行はエージェント同士に任せる**。

人間が関与するのは、以下のすべてが満たされた**最終レビュー**の1点に集約します。

- エージェント同士のレビュー（設計レビューやPRレビュー）が実施済み
- ドキュメント（READMEや仕様書）が更新済み
- CIが通過し、すべてのテストが合格している
```
