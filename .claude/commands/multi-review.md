# /multi-review

指定対象を **複数視点で検証**し、各視点を突き合わせた統合所見を出す。今セッションで繰り返した「セルフレビュー → 独立検証 → 外部AIレビュー → 統合」の手作業を恒久化したもの。

## 引数

- `$1` = レビュー対象（**冒頭で必ず明示**）。ファイルパス / PR番号 / 「記事の主張」など。曖昧なら着手前に確定する（学び: レビュー依頼は対象を明示する）。
- `$2`（任意）= 観点（`correctness` / `security` / `design` / `e-e-a-t` / `process` 等）。未指定なら対象種別から推定。

## 視点の層（フォールバック設計）

外部レビューツールは**レート制限前提**。可用性で多重化し、CLI を主経路にしない（学び 2026-06-08）。

| 層 | 担当 | 位置づけ | 落ちた時 |
|---|---|---|---|
| **L1（必須）** | 独立 Claude サブエージェント（`general-purpose` 等を `run_in_background`） | 自前・最も可用。白紙で検証しバイアス回避 | — |
| **L2** | GitHub 上の bot（Gemini Code Assist / Copilot） | PR 連動で安定。PR がある場合に活用 | PR が無ければ skip |
| **L3（補助）** | CLI（Codex `codex:codex-rescue` / Gemini CLI） | 補助。**429/trust エラーは即フォールバック**（リトライで粘らない） | skip して L1+L2 で続行 |

**最低 L1 ＋（PRがあれば L2）の 2 視点が揃えば判断可**。L3 が落ちてもブロックしない。

> Gemini CLI 実務: 非対話は `--skip-trust` 必須、`pro` 容量切れ時は `-m gemini-2.5-flash`、プロンプトは stdin に流す。

## 手順

1. **対象とスコープを 1 行で明示**（`$1`）。複数解釈があれば確定してから進む。
2. **セルフレビュー**（自分の視点）: 対象を読み、所見を「分類 / 重大度 / 具体策」で構造化。
3. **L1: 独立 Claude を起動**（`run_in_background: true`）。バイアス回避のため**自分の所見は渡さず白紙で検証**させる。観点・対象パスのみ渡し、**完了を待ってその所見を取得する**（統合前に必ず結果を回収）。
4. **L2**: 対象に関連する PR（または現在のブランチの PR）があるなら、bot レビュー（`gh pr view --json reviews` / inline comments）を取得。無ければ skip。
5. **L3（任意）**: 時間/価値が見合えば Codex か Gemini CLI を 1 つ。429/trust で失敗したら**即 skip**。
6. **統合**: 各視点を突き合わせ、**一致した指摘＝確度が高い**として優先。割れた点は両論を示し判断。重大度順に。
7. 必要なら確定した修正だけ適用（対象が記事なら主張・強調を変えない＝`/review-improve-loop` の不変条件に準ずる）。

## 出力

- 視点ごとの所見サマリ → **突き合わせ表（一致/相違）** → 統合結論（最優先 1 つを明示）。
- 「どの層が走ったか / skip したか」を明記（無音 skip しない＝運用の透明性）。

## 関連

- `/review-improve-loop <slug>` — 記事を review→改善で反復（単一視点ループ、主張不変）
- `/review-article <slug>` — 3ペルソナの単発レビュー
- 本コマンドは「**1 対象を複数の独立した目で検証する**」点が上記と異なる（記事に限らずコード・設計・プロセスにも使う）
